Data Type、Null 與 Empty String
「沒有值」也有學問

預估 30 分鐘 ・ 建議先讀完 4-1 再來。

上一章結尾留了一個問題:加了新欄位之後,舊資料的那個格子裡是什麼?答案比想像中微妙——「什麼都沒有」在資料庫裡,其實有兩種完全不同的狀態。這一章還會先教一個更基本的概念:每個欄位都有自己的「型別」。

  • 認得常見的 Data Type(資料型別)。
  • 分辨 Null 和 Empty String(空字串)的差異。
  • 理解為什麼報表統計會因為這兩者而出現落差。
情境

「兩張報表的數字對不起來」

Northstar Electronics 的行銷團隊做了一份統計:「沒有留 Email 的客戶有幾位?」兩位同事各自拉了報表,結果:

「我拉出來是 3,200 人,你怎麼會是 4,100 人?我們查的是同一張表啊!」

兩個人都沒有算錯。差異的來源,是「沒有留 Email」這句話,在資料庫裡其實對應兩種不同的狀態:一種是 Null,一種是空字串。一份報表只算了其中一種,另一份兩種都算。這一章讀完,您就能一眼看出這類數字落差的原因。

情境例子 1/2

每個欄位,都有自己的「格式規定」

先從 Data Type 說起。想像您在填一張紙本申請表:生日欄要填日期、電話欄要填數字、姓名欄要填文字。如果有人在生日欄寫「藍色」,承辦人員會直接退件——因為格式不對。

Data Type(資料型別)就是資料庫版的「格式規定」。每個欄位在設計時,就決定了它只收哪種資料:

常見型別收什麼例子
文字(Text/String)任意文字姓名、地址
數字(Number)可以計算的數值金額、數量
日期(Date/Datetime)日期或日期加時間訂單日期、建立時間
布林(Boolean)只有真/假兩種值是否訂閱電子報
先記住這句 型別決定了「能填什麼」和「能拿它做什麼」。數字能加總,文字不能;日期能比較先後,文字型的「日期」不能。型別選錯,之後統計和排序都會出問題。
情境例子 2/2

「從來沒填過」和「填了一個空白」

回到「沒有值」的問題。想像申請表上的 Email 欄位,有兩種「看起來都是空的」狀況:

第一種:這格從來沒被碰過。申請人跳過了這一欄,紙上什麼都沒有。這是 Null——代表「未知、從未填寫」,是一種「沒有資料」的狀態。

第二種:申請人拿筆在這格畫了一條橫線,表示「我沒有 Email」。這格其實「被填過了」,填的內容是「空」。這是 Empty String(空字串)——欄位裡確實存了一個值,只是那個值是長度為零的文字。

肉眼看報表,兩種都是空格子。但對資料庫來說,它們是不同的東西:查詢「Email 是 Null 的客戶」不會抓到空字串的那些;查「Email 是空字串的」也不會抓到 Null 的。開頭那兩位同事的數字落差,正是一位只查了 Null,另一位把兩種都算進去了。

正式定義

三個名詞的正式說法

名詞定義
Data Type
資料型別		欄位在設計時定義的資料格式,決定這個欄位能存放哪種資料、能進行哪些運算
Null代表「沒有值、未知」的特殊狀態;這個欄位從未被賦值,或被明確設為未知
Empty String
空字串		長度為零的文字值;欄位「有被填過」,只是填入的內容是空的

一句話區分:Null 是「這格沒有東西」,空字串是「這格有東西,那個東西是『空白』」。聽起來像繞口令,但這個差異在查詢、統計、資料同步時,都會實實在在地影響結果。

串起來看

同一張表,三種 Email 狀態

看 Northstar 的 Customer Table 裡三筆紀錄:

客戶編號  姓名    Email
C001     王小明  wang@example.com   ← 有值
C002     李美麗  (Null)             ← 從未填過
C003     陳大文  ""(空字串)          ← 匯入時被填入了空白

C003 的空字串是怎麼來的?常見的來源是資料匯入:匯入檔案裡那一格是空白,匯入程式把「空白」如實寫了進去,於是欄位裡存了一個空字串,而不是 Null。這也是為什麼老系統轉移資料後,同一個欄位常常混雜著 Null 和空字串兩種狀態——它們來自不同的資料來源和寫入方式。

為什麼要學這個

兩個馬上用得到的場景

場景一:解釋報表數字對不上的原因

兩份「理應相同」的統計數字有落差時,Null 與空字串的處理差異是常見嫌疑犯之一。您可以直接問工程師:「這兩份報表對『沒有值』的定義一樣嗎?有沒有一份只算了 Null?」這一問常常直接命中。

場景二:資料匯入與同步前的規格確認

做系統轉移或整合時,先和雙方確認:「來源欄位是空的時候,要寫成 Null 還是空字串?」這個小約定沒講清楚,轉移後的資料就會出現兩種混雜的「空」,日後每一份統計都要多做一層處理。單元 5 會再深入這個主題(Null 是否可以覆蓋原值)。

核心概念 1/2

型別決定資料的能力

概念一:數字存成文字,就失去了計算能力

如果金額欄位被設成文字型別,裡面存的「1000」只是三個字元,不能直接加總,排序時「9」還會排在「10」後面(因為文字是逐字比較的)。遇到「排序怪怪的」「加總算不出來」,先問一句:「這個欄位的型別是什麼?」

概念二:日期型別讓「先後」有意義

日期存成正確的日期型別,才能做「找出上個月的訂單」這類查詢。存成文字的日期(尤其格式不一,像 2026/7/1 和 07-01-2026 混用)是資料清理的惡夢,這在單元 5 會再遇到。

核心概念 2/2

「空」的兩張面孔

概念三:Null 参與計算時,行為很特別

資料庫處理 Null 有一套特殊規則,例如「Null 不等於任何值,連另一個 Null 都不等於」。您不需要背這些規則,只要記得:查詢條件遇到 Null,結果常常跟直覺不同,這是請工程師確認查詢邏輯時的重要提示。

概念四:混雜的「空」是系統整合的常見地雷

不同系統對「空」的表示方式不同:有的用 Null,有的用空字串,有的甚至用「N/A」這種文字。整合時如果沒有統一處理,下游的統計、去重、比對全都會受影響。發現資料怪怪的,把「空值的表示方式」列入檢查清單。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
Null vs Empty StringNull 是「從未有值」的狀態;空字串是「有值,但值是空白文字」。畫面上看起來一樣,查詢時是兩種不同條件。
「數字 1000」vs「文字 1000」數字型別的 1000 可以加總、正確排序;文字型別的 "1000" 只是四個字元。畫面上顯示一模一樣,能力完全不同。
案例

Northstar Electronics 的報表落差調查

回到開頭的統計落差,追查結果:

問題:「沒有留 Email 的客戶數」兩份報表分別是 3,200 與 4,100
報表 A 的條件:Email 是 Null                    → 3,200 筆
報表 B 的條件:Email 是 Null 或 空字串           → 4,100 筆
空字串的來源:三年前從舊系統匯入的那批資料,
             匯入程式把空白格寫成了空字串
  1. 有哪些角色?兩位拉報表的同事、Customer Table、三年前的匯入程式。
  2. 資料如何移動?舊系統轉移時,空白欄位被寫成空字串;之後系統內新建的客戶若未填 Email 則是 Null。
  3. 各部分負責什麼?報表條件的定義,決定了哪些「空」被算進去。
  4. 哪裡可能出錯?沒有人「算錯」,是兩份報表對「沒有 Email」的定義不同。真正的問題是資料裡混雜著兩種「空」。
  5. 第一個合理的檢查方向?數字對不上時,先比對兩份報表的篩選條件,特別是對 Null 與空字串的處理方式是否一致。

常見誤解

三個很多人踩過的坑

誤解一:「畫面上是空的,資料庫裡就是 Null」

畫面上的空格,背後可能是 Null,也可能是空字串,甚至是只有空格字元的文字。要確認真實狀態,得看資料庫裡實際存了什麼。

誤解二:「型別以後可以隨便改,先隨便選一個沒關係」

欄位有了大量資料之後再改型別,常常需要資料轉換,甚至有轉換失敗的風險。型別在設計時就該選對,這是需求訪談時值得花時間確認的細節。

誤解三:「Null 就是零」

數字欄位的 Null 是「未知」,不是 0。一筆訂單金額是 Null,代表金額沒被記錄;是 0,代表這筆訂單確實是零元。統計平均值時,兩者的影響完全不同。

遇到問題先查什麼

「報表數字對不上」的檢查順序

  1. 確認兩份報表查的是不是同一張表、同一個時間範圍。先排除最基本的差異。
  2. 比對兩份報表的篩選條件。特別是「空值」的定義:只算 Null,還是 Null 加空字串?
  3. 確認欄位裡是否混雜著多種「空」。可以請工程師分別統計 Null 和空字串的筆數。
  4. 追查混雜的來源。通常是某次資料匯入或某條整合寫入方式不同。
  5. 訂出統一規則。之後的報表都用同一種空值定義,必要時安排一次資料清理(單元 5 會教)。
重點整理

帶走這四句話

  1. 每個欄位都有 Data Type,型別決定了能填什麼、能拿資料做什麼。
  2. Null 是「從未有值」,空字串是「有值但內容是空白」,畫面上看起來一樣,查詢時完全不同。
  3. 混雜的「空」通常來自不同的資料來源與寫入方式,是報表數字對不上的常見原因。
  4. 資料匯入或整合前,先約定「空的時候寫什麼」,能省下日後大量的清理工作。
檢核 1/4

欄位「從未被填過任何值」的狀態,叫做什麼?

正確答案是 B。Null 代表「沒有值、未知」;空字串則是有被填過,只是填入的內容是空白文字。
檢核 2/4

是非題:數字欄位裡的 Null,和 0 是同一個意思。

正確答案是「否」。Null 代表金額沒被記錄(未知),0 代表金額確定是零。統計平均值時,兩者的影響完全不同。
檢核 3/4

金額欄位被設成「文字」型別,最可能造成什麼問題?

正確答案是 C。文字型別的「1000」只是字元,不能直接計算;文字排序是逐字比較,「9」會排在「10」後面。型別決定了資料的能力。
檢核 4/4

情境題:兩份查同一張表的報表,「沒有 Email 的客戶數」差了 900 筆。第一個合理的檢查方向是?

正確答案是 A。同一張表的統計出現落差,最常見的原因是篩選條件對 Null 與空字串的處理不同,先比對條件定義就能快速找到答案。
