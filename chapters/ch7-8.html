Error Queue 與 Monitoring — 失敗的事件去了哪裡

預估 30 分鐘 ・ Webhook、事件與自動化

Error Queue 保存處理失敗的事件，Monitoring 追蹤量、延遲與錯誤；兩者讓失敗可被發現與重送。

  • 能說明 Error Queue：保存處理失敗事件，等待檢查與重送的佇列。
  • 能區分 Error Queue 與 Monitoring 在流程中的用途。
  • 遇到問題時，先找 Event ID、Producer 與 Consumer。
案例起點

Error Queue 和 Monitoring 沒有寫清楚時

Northstar 的規格同時提到 Error Queue 與 Monitoring，卻沒有標出兩者的適用條件與處理邊界。

Error Queue：保存處理失敗事件，等待檢查與重送的佇列。；Monitoring：持續觀察成功、失敗、延遲與積壓狀態的機制。

測試時要避免這個結果：只有成功數量，沒有失敗與積壓監控，問題會累積到使用者回報才被看見。

概念拆解 1/2

Error Queue 的處理邊界

Error Queue：保存處理失敗事件，等待檢查與重送的佇列。

套用到本章情境，第一步是找 Event ID、Producer 與 Consumer。確認 Error Queue 的適用範圍後，再檢查 Monitoring，不從畫面直接猜原因。

判斷重點說明 Error Queue 時，要交代誰提供輸入、系統套用哪條規則，以及會產生什麼結果。
概念拆解 2/2

Monitoring 與 Error Queue 的差別

Monitoring：持續觀察成功、失敗、延遲與積壓狀態的機制。

Error Queue 保存處理失敗的事件，Monitoring 追蹤量、延遲與錯誤；兩者讓失敗可被發現與重送。

不要混用Error Queue 與 Monitoring 不能互換；規格要分別寫出兩者的輸入、處理方式與輸出。

問題排查

先找 Event ID、Producer 與 Consumer，再確認事件發生時間與接收時間。這能分辨問題是否位於 Error Queue 或 Monitoring。

核心概念 1/2

輸入、規則與輸出要分開寫

輸入

列出 Error Queue 開始前已知的資料、身分、事件或設定。輸入不明，後面的結果就無法比較。

規則與輸出

Monitoring 執行後，保存實際套用的規則版本、Response、狀態或資料值。不要用「看起來正常」代替可查證結果。

核心概念 2/2

成功條件和失敗條件都要具體

成功條件

Error Queue 保存處理失敗的事件，Monitoring 追蹤量、延遲與錯誤；兩者讓失敗可被發現與重送。 驗收 Error Queue 與 Monitoring 時，寫明預期狀態、資料值與完成時間。

失敗條件

只有成功數量，沒有失敗與積壓監控，問題會累積到使用者回報才被看見。 Error Queue 或 Monitoring 失敗時，保留原始輸入、環境、時間與識別值。

容易混淆

用處理邊界分清楚三組概念

容易混淆實際差別
Error Queue vs MonitoringError Queue：保存處理失敗事件，等待檢查與重送的佇列。 Monitoring：持續觀察成功、失敗、延遲與積壓狀態的機制。
規格存在 vs 實際生效Error Queue 的文件是預期規則；Monitoring 的設定值、Response 或 Log 才能證明實際結果。
單次成功 vs 穩定可重現Error Queue 單次成功不代表 Monitoring 在其他條件下也正確；要用相同輸入重做。

常見誤解

三個會讓排查走錯方向的假設

把 Error Queue 和 Monitoring 當成同一件事

只有成功數量，沒有失敗與積壓監控，問題會累積到使用者回報才被看見。

只看最後畫面

畫面是流程的最後一層。仍要核對 Error Queue 的輸入與 Monitoring 的執行紀錄。

一次改多個條件

同時修改 Error Queue 與 Monitoring 的條件，即使恢復也無法歸因。每次只改一項。

排查順序

Error Queue 發生異常時的排查順序

  1. 1. 找 Event ID、Producer 與 Consumer。
  2. 2. 確認事件發生時間與接收時間。
  3. 3. 查 Queue、Retry 及重複事件。
  4. 4. 檢查是否有循環或先後順序問題。
  5. 5. 把失敗事件送入 Error Queue 並監控。

每一步先記錄原始值再做下一步。若修改 Error Queue 或 Monitoring 的設定，保存修改前後值與時間。

重點整理

  2. Monitoring：持續觀察成功、失敗、延遲與積壓狀態的機制。
  3. 只有成功數量，沒有失敗與積壓監控，問題會累積到使用者回報才被看見。
