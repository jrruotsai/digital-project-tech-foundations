資料表、欄位與紀錄
資料庫的基本單位

預估 30 分鐘 ・ 建議先讀完 1-1(三層架構)再來。

1-1 教過,資料庫是系統的「長期記憶」。從這一章開始,我們要打開這個記憶櫃,看看裡面到底是怎麼收納的。好消息是:如果您用過 Excel,您已經懂一半了。

  • 分辨 Database、Table、Column、Row 這四個基本單位。
  • 看懂一張簡單的資料表。
  • 聽到客戶說「幫我加一個欄位」時,知道這句話在資料庫裡代表什麼。
情境

「就加一個欄位而已,應該很簡單吧?」

Northstar Electronics 的業務主管在需求會議上說:

「我想在客戶資料上多記一個『推薦人』,就加一個欄位而已,應該很簡單吧?」

「加一個欄位」這句話,顧問幾乎天天聽到。要判斷它到底簡不簡單,您得先知道:欄位是什麼?它加在哪裡?加了之後,已經存在的幾萬筆舊資料會發生什麼事?這一章就從這些最基本的單位開始。

情境例子 1/2

一個櫃子、幾本名冊

想像公司有一個大檔案櫃,裡面放著好幾本名冊:一本「客戶名冊」、一本「訂單名冊」、一本「產品目錄」。

Database(資料庫)就是那整個檔案櫃——一個有組織地存放所有資料的地方。

Table(資料表)就是櫃子裡的每一本名冊——每本名冊記錄一種特定主題的資料。客戶的資料放在客戶那本,訂單的資料放在訂單那本,不會混在一起。

翻開其中一本名冊,它長得就像 Excel 的表格:有欄、有列。這就是接下來要拆的更小單位。

先記住這句 Database 是櫃子,Table 是櫃子裡的一本本名冊。一個資料庫裡通常有很多張資料表,各自負責一種主題的資料。
情境例子 2/2

直的是欄位,橫的是一筆資料

翻開「客戶名冊」這張表,長這樣:

客戶編號姓名Email電話
C001王小明wang@example.com0912-345-678
C002李美麗lee@example.com0922-111-222

Column(欄位)是直的那一條——「姓名」「Email」「電話」都是欄位。欄位定義了「這張表要記錄哪些種類的資訊」。

Row(列),又叫 Record(紀錄)——是橫的那一條。一列就是一筆完整的資料:C001 王小明那一整列,就是「王小明這位客戶」在系統裡的一筆紀錄。

回到開頭的需求:「加一個推薦人欄位」,就是在這張表上多加一條直的——之後每一筆客戶紀錄,都會多一個可以填「推薦人」的格子。聽起來簡單,但有個問題值得馬上想到:已經存在的 C001、C002 這些舊紀錄,新欄位裡要填什麼?這正是下一章(Null 與空值)要處理的主題。

正式定義

四個基本單位的正式說法

名詞定義比喻
Database
資料庫		有組織地儲存、管理多張資料表的系統整個檔案櫃
Table
資料表		針對單一主題,以欄與列組織資料的結構櫃子裡的一本名冊
Column
欄位		資料表中定義「要記錄哪種資訊」的直向單位名冊的欄目標題
Row / Record
列/紀錄		資料表中一筆完整資料的橫向單位名冊裡的一行登記

這四個詞是整個單元 4 的地基,後面每一章都會用到。如果只想先記兩個,記「欄位是直的、紀錄是橫的」就夠。

串起來看

從表單送出,到資料庫裡多一列

把 1-1 教過的資料旅程,接到資料庫的內部:

訪客在官網填寫報名表單(姓名、Email)
→ 後端驗證通過
→ 資料庫的「報名名單」Table 裡,新增一筆 Record
→ 這筆 Record 的「姓名」Column 填入王小明
→ 「Email」Column 填入 wang@example.com

所以「表單送出成功」的具體意思,其實就是「某張 Table 裡多了一列,而且每個欄位都填上了對的值」。之後排查「資料不見了」,最終極的檢查就是:去那張表裡看,那一列到底有沒有出現。

為什麼要學這個

兩個馬上用得到的場景

場景一:把需求翻譯成資料的語言

客戶說「我要記錄客戶的推薦人」,您心裡可以立刻翻譯:「客戶 Table 要加一個推薦人 Column」。接著自然會想到該問的問題:推薦人是填文字就好,還是要連到另一筆客戶紀錄?舊資料怎麼辦?這些問題,正是工程師估工時時真正在意的事。

場景二:看懂資料匯入的報錯

做資料匯入時,錯誤訊息常寫著「第 152 列的某欄位格式錯誤」。懂了欄與列,您就知道這是指匯入檔案裡第 152 筆資料的某個欄位有問題,可以直接打開檔案找到那一筆來檢查,不用整批重來。

核心概念 1/2

一張表,一個主題

概念一:每張 Table 負責一種主題,不要什麼都塞同一張

客戶資料放客戶表、訂單資料放訂單表。如果把訂單資訊硬塞進客戶表(例如在客戶表上開「訂單1」「訂單2」「訂單3」欄位),很快就會遇到「第 4 筆訂單放哪裡」的窘境。分表存放,再用關聯把它們接起來(4-4 會教),才是資料庫的正規做法。

概念二:欄位是「事先定義」的,不是想填什麼就填什麼

資料表的欄位結構是事先設計好的,每一筆紀錄都遵循同樣的欄位。這跟自由格式的筆記不同——正因為結構固定,系統才能快速查詢、統計、比對。

核心概念 2/2

改結構和改資料,是兩件不同量級的事

概念三:新增一筆紀錄很平常,新增一個欄位是「改結構」

每天都有新客戶、新訂單寫進資料表,這是日常的資料異動。但「加一個欄位」是改變表的結構本身,會影響到這張表的每一筆既有紀錄,以及所有用到這張表的程式、報表、整合。這就是為什麼「只是加個欄位」在工程師耳裡,從來不是小事。

概念四:紀錄會成千上萬,結構要盡量穩定

一張客戶表可能有幾十萬筆紀錄,但欄位可能只有二十個。資料天天變,結構最好少動——每次動結構,都要評估對既有資料和下游系統的影響。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
Database vs TableDatabase 是整個櫃子,Table 是裡面的一本名冊。說「查資料庫」其實通常是「查資料庫裡的某張表」。
Column vs Row(Record)Column 是直的、定義資訊種類(姓名、Email);Row 是橫的、一筆完整資料(王小明那一整列)。「加欄位」動的是結構,「加一筆資料」動的是內容。

還有一個小提醒:Excel 跟資料表長得像,但 Excel 隨時可以在任何格子亂填,資料表則是結構嚴謹、每筆紀錄都遵循同樣欄位定義。像,但不一樣。

案例

Northstar Electronics 的客戶資料表

Northstar 的 CRM System 背後,客戶資料存放在一張 Customer Table:

Customer Table(客戶資料表)
欄位:客戶編號、姓名、Email、電話、建立日期
目前紀錄數:約 85,000 筆

業務主管的需求:新增「推薦人」欄位
  1. 有哪些角色?業務主管(需求方)、CRM 管理者、Customer Table 以及依賴它的報表與整合。
  2. 資料如何移動?加欄位是改 Table 結構;之後新的客戶紀錄才能開始填推薦人。
  3. 各部分負責什麼?Table 結構定義資料長相;既有的 85,000 筆紀錄,新欄位一開始都是空的。
  4. 哪裡可能出錯?如果有報表或整合程式假設欄位固定不變,新欄位可能影響它們;而 85,000 筆舊資料的推薦人是空的,統計時要小心解讀。
  5. 第一個合理的檢查方向?先問清楚:推薦人要填什麼形式的內容?舊資料需不需要回填?有哪些報表和整合會用到這張表?釐清這三件事,才能正確評估這個「小需求」。
延伸對照(選讀)

這些概念在 Salesforce 裡的樣子

不熟悉這些產品的話,這張卡片可以跳過。

通用概念Salesforce 對照
TableObject(物件),例如 Account、Contact
ColumnField(欄位)
Row / RecordRecord(紀錄)

Salesforce 把資料庫包裝成 Object 與 Field 的形式,但底層邏輯完全相同:加一個 Custom Field,就是在改這個 Object 的結構。

常見誤解

三個很多人踩過的坑

誤解一:「加欄位很簡單,五分鐘的事」

操作本身可能真的只要五分鐘,但影響評估(舊資料、報表、整合)才是主要的工作量。沒做評估就加欄位,問題往往在幾週後才爆出來。

誤解二:「資料庫就是一張大表,所有東西都在裡面」

資料庫裡是很多張各司其職的表。「客戶的訂單資料」通常不在客戶表裡,而是在訂單表中,靠關聯連起來。查不到資料時,可能只是查錯了表。

誤解三:「Excel 就是資料庫」

Excel 適合個人整理資料,但缺乏資料庫的結構約束、多人同時操作與大量資料處理能力。把 Excel 當正式資料庫用,常是資料品質問題(單元 5 的主題)的起點。

遇到問題先查什麼

「客戶要求加欄位」的釐清順序

  1. 確認這個欄位要記錄什麼形式的內容。文字?數字?日期?還是要連到另一筆資料?
  2. 確認既有的舊紀錄怎麼辦。需要回填嗎?還是留空即可?
  3. 確認有哪些報表、畫面、整合會用到這張表。它們需不需要跟著調整?
  4. 確認欄位是否已經存在類似的。避免重複建立意義相近的欄位,造成日後混亂。
  5. 把以上資訊整理好,再和工程師討論工時。這時候的估計才會準。
重點整理

帶走這四句話

  1. Database 是櫃子,Table 是名冊,Column 是直的欄目,Row(Record)是橫的一筆資料。
  2. 一張表負責一種主題;不同主題的資料分表存放,再靠關聯連接。
  3. 「加一筆資料」是日常,「加一個欄位」是改結構,影響範圍完全不同。
  4. 「加個欄位」的真正工作量在影響評估:舊資料、報表、整合,一個都不能漏。
檢核 1/4

資料表中,橫向的一筆完整資料,叫做什麼?

正確答案是 C。橫向的一筆完整資料是 Row(列),也叫 Record(紀錄);直向定義資訊種類的才是 Column(欄位)。
檢核 2/4

是非題:「在客戶表新增一個欄位」和「在客戶表新增一筆客戶資料」,對系統的影響程度差不多。

正確答案是「否」。新增紀錄是日常的資料異動;新增欄位是改變表的結構,會影響既有紀錄、報表與整合,兩者量級完全不同。
檢核 3/4

客戶的訂單明細,通常應該存放在哪裡?

正確答案是 A。一張表負責一種主題:訂單放訂單表,靠關聯連到客戶表。在客戶表上開固定數量的訂單欄位,很快就會不夠用。
檢核 4/4

情境題:客戶說「幫我加一個推薦人欄位,很簡單吧」。您的第一步應該是?

正確答案是 B。加欄位的操作不難,難在影響評估。先釐清這三件事,才能給客戶準確的工時與風險說明。
