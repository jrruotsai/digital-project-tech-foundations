Authentication 與 Authorization 的差異

預估 30 分鐘 ・ Authentication、Authorization 與 SSO

Authentication 驗證提出請求的對象是誰；Authorization 則根據身分、動作、資源與政策，決定是否允許這次操作。兩者是不同的判斷。

  • 說明 Authentication 的輸入、驗證結果與失敗位置。
  • 用「誰要對哪個資源做什麼」描述 Authorization。
  • 從登入結果、Session／Token、權限政策與 HTTP 回應判斷問題層級。
判斷桌

兩次判斷，各自需要哪些資料？

COMPARE / 分開看輸入與輸出

Authentication

輸入
帳號識別、Credential、MFA 或外部 IdP 的驗證結果
輸出
已驗證的 Principal，以及 Session、Assertion 或 Token
典型失敗
Credential 無效、Session 無法建立、Token 驗證失敗

Authorization

輸入
Principal、要求的 Action、Resource、Role／Policy 與執行情境
輸出
Allow 或 Deny；有時還會限制可見欄位或資料範圍
典型失敗
已登入但不能讀取、修改或核准指定資源
先確認系統是否已建立可信身分，再檢查該身分對這個資源是否有權限。
案例起點

「進不去」可能是兩種不同問題

王小姐已通過公司 IdP 登入，也取得應用程式 Session，但開啟「薪資報表」時被拒絕。這不是身分驗證失敗；應用程式已知道她是誰，只是政策不允許她讀取該資源。

先問：Session 或 Token 是否已建立？若答案是肯定的，再查 Role、Permission、Resource 與 Policy。

相反地，如果 IdP 沒有完成驗證、Token 簽章無效或 Session 未建立，才回到 Authentication 層排查。

概念拆解 1/2

Authentication 驗證的是 Credential，不是使用者的自我宣稱

使用者輸入帳號只是在宣稱身分。系統還要驗證密碼、Passkey、憑證或 IdP Assertion，才能把請求綁定到一個 Principal。

驗證成功後，系統通常建立 Session 或簽發 Token。後續請求使用這個結果，不會每次重新要求輸入密碼。

判斷重點密碼是 Credential；Session 與 Token 是驗證成功後的狀態或憑證。三者不要混稱為「帳號」。
概念拆解 2/2

Authorization 評估的是「這次操作」

Authorization 不是登入後只做一次。每次讀取、修改、刪除或核准資源時，系統都可能重新評估 Principal、Action、Resource 與 Policy。

同一個使用者可以讀取自己的報表，卻不能讀取別人的；可以檢視訂單，卻不能核准退款。身分相同，資源或動作不同，結果就可能不同。

規格寫法不要只寫「主管有權限」。要寫成「具有 Manager Role 的使用者，可以 Approve 自己部門、金額低於 10 萬元的訂單」。
精確定義

本章名詞的工作定義

術語完整定義可觀察結果
Authentication系統驗證 Credential 或可信身分聲明，將請求綁定到一個 Principal 的程序。Session 建立、Assertion／Token 簽發或驗證成功。
Authorization系統根據 Principal、Action、Resource、Policy 與情境，決定是否允許操作的程序。Allow／Deny、HTTP 403，或資料列與欄位範圍受到限制。
Principal通過驗證後，系統用來代表使用者、服務帳號或程式的身分主體。User ID、Subject、Service Account ID。

HTTP 401 通常表示缺少或無效的驗證資訊；403 通常表示已理解請求但拒絕授權。實際系統可能採用不同錯誤映射，仍要查看 Response 與 Log。

流程位置

一次受保護請求會經過兩次判斷

1. User → IdP：提交 Credential／完成 MFA
2. IdP → Application：回傳 Assertion 或 Token
3. Application：驗證簽章、Issuer、Audience、時間 → Authentication
4. Application：取得 Principal、Action、Resource、Policy → Authorization
5. Allow：執行操作　／　Deny：回傳錯誤並記錄理由

第三步回答「這個請求代表誰」，第四步回答「這個身分能不能做這次操作」。

規格與設計

身分整合顧問需要把「Authentication 與 Authorization 的差異」改寫成可確認的規則，放進登入流程與身分對照表。

問題排查

先確認使用者、Account 與 Session，再判斷失敗在 Authentication 還是 Authorization。這能分辨問題是否位於 Authentication 或 Authorization。

核心概念 1/2

輸入、規則與輸出要分開寫

輸入

列出 Authentication 開始前已知的資料、身分、事件或設定。輸入不明，後面的結果就無法比較。

規則與輸出

Authorization 執行後，保存實際套用的規則版本、Response、狀態或資料值。不要用「看起來正常」代替可查證結果。

核心概念 2/2

成功條件和失敗條件都要具體

成功條件

Authentication 確認你是誰，Authorization 決定你能做什麼。登入成功不代表有權存取所有功能。 驗收 Authentication 與 Authorization 時，寫明預期狀態、資料值與完成時間。

失敗條件

把權限不足當成登入失敗，會查錯系統層級。 Authentication 或 Authorization 失敗時，保留原始輸入、環境、時間與識別值。

容易混淆

用處理邊界分清楚三組概念

容易混淆實際差別
Authentication vs AuthorizationAuthentication：確認身分。 Authorization：依身分與政策決定可做的事情。
規格存在 vs 實際生效Authentication 的文件是預期規則；Authorization 的設定值、Response 或 Log 才能證明實際結果。
單次成功 vs 穩定可重現Authentication 單次成功不代表 Authorization 在其他條件下也正確；要用相同輸入重做。

常見誤解

三個會讓排查走錯方向的假設

把 Authentication 和 Authorization 當成同一件事

把權限不足當成登入失敗，會查錯系統層級。

只看最後畫面

畫面是流程的最後一層。仍要核對 Authentication 的輸入與 Authorization 的執行紀錄。

一次改多個條件

同時修改 Authentication 與 Authorization 的條件，即使恢復也無法歸因。每次只改一項。

排查順序

Authentication 發生異常時的排查順序

  1. 1. 確認使用者、Account 與 Session。
  2. 2. 判斷失敗在 Authentication 還是 Authorization。
  3. 3. 檢查 IdP／SP 端點與 User Mapping。
  4. 4. 查看 Assertion、Token、Scope 或 Certificate。
  5. 5. 核對 MFA、Provisioning 與停權狀態。

每一步先記錄原始值再做下一步。若修改 Authentication 或 Authorization 的設定，保存修改前後值與時間。

重點整理

讀完後應該能說清楚這四件事

  1. Authentication 驗證 Credential 或可信聲明，輸出可供系統使用的 Principal。
  2. Authorization 以 Principal、Action、Resource 和 Policy 評估每次操作。
  3. 已取得 Session／Token 但不能操作資源，通常先查 Authorization；Session／Token 無法建立或驗證，先查 Authentication。
  4. 排查時保留使用者或 Subject、時間、Endpoint、Response、Session／Token 狀態與政策判斷紀錄。
