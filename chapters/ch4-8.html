Index、View 與 Stored Procedure
資料庫的三個常見名詞

預估 30 分鐘 ・ 單元 4 的最後一章,建議先讀完 4-6 再來。

這一章像一趟收尾的導覽:三個工程師常掛嘴邊、顧問常聽得一頭霧水的資料庫名詞,一次認識。另外還會補上兩個小概念:Transaction 和 Schema。目標不是操作它們,而是聽到時知道對方在說什麼。

  • 用一句話說明 Index、View、Stored Procedure 各自的用途。
  • 理解「查詢很慢」和 Index 的關係。
  • 理解 Transaction「全有或全無」的概念。
情境

「工程師說:加個索引就會快了」

Northstar Electronics 的客服系統,查詢客戶歷史紀錄越來越慢,一查要十幾秒。反映給工程師後,得到一句:

「喔,那張表沒建索引,查詢都在全表掃描。我加個 Index 就會快了。」

隔天,同樣的查詢變成不到一秒。到底發生了什麼魔法?「全表掃描」又是什麼?這一章用一本書的比喻,讓您聽懂這段對話。

情境例子 1/2

沒有目錄的書,只能一頁一頁翻

想像一本一千頁的百科全書,您要找「北極熊」的條目。如果書末有索引頁(按筆畫排好,標著每個詞在第幾頁),翻兩下就找到;如果沒有,您只能從第一頁開始逐頁翻——這就是工程師說的全表掃描(Full Table Scan)。

Index(索引)就是資料表的「索引頁」:針對某個欄位(例如客戶編號)預先建好排序好的查找結構,查詢時直接跳到目標,不用逐筆翻。幾十萬筆的表,有沒有索引,查詢速度可以差上百倍——這就是「加個 Index 就快了」的原理。

先記住這句 Index 是欄位的目錄,讓查詢不用逐筆翻。查詢慢,第一個常見嫌疑就是「查詢條件用到的欄位沒建索引」。

那為什麼不每個欄位都建?因為目錄本身要維護:每新增一筆資料,所有相關的索引頁都要跟著更新。索引越多,寫入越慢、佔空間越多。建哪些索引,是查詢速度與寫入成本之間的取捨,由工程師依實際查詢模式判斷。

情境例子 2/2

存好的視角,和存好的流程

另外兩個名詞,也各有一個貼切的比喻:

View(檢視表)像「存好的查詢視角」。4-7 那種接了三張表、條件複雜的查詢,如果每個人都要重寫一遍,既麻煩又容易寫得不一致。工程師可以把它存成一個 View,取名「VIP訂單總覽」,之後大家直接查這個名字,像查一張普通的表——但它背後其實是即時執行那段查詢。View 本身通常不存資料,它是一面「透鏡」。

Stored Procedure(預存程序)像「存在資料庫裡的標準作業流程」。例如「月底結帳」要依序做十個步驟,工程師把整套流程寫好、存進資料庫,取名 MonthlyClose。之後每個月只要「呼叫」這個名字,十個步驟就照順序執行。它跟 View 的差別:View 是「存好的查詢」,Stored Procedure 是「存好的一連串動作」,裡面可以包含新增、修改等異動。

正式定義

五個名詞的正式說法

名詞定義
Index
索引		針對欄位預先建立的查找結構,加速查詢,但增加寫入與儲存成本
View
檢視表		預先定義、以名字保存的查詢;查它像查一張表,實際是即時執行背後的查詢
Stored Procedure

Schema
結構描述		資料庫的整體結構藍圖:有哪些表、欄位、型別、關聯
串起來看

Transaction:轉帳的「全有或全無」

Transaction 值得多看一眼,因為它保護著資料的一致性。經典例子是轉帳:

轉帳 1,000 元,包含兩個動作:
步驟 1:A 帳戶扣 1,000
步驟 2:B 帳戶加 1,000

沒有 Transaction:步驟 1 成功後系統當機
→ A 的錢扣了,B 沒收到,1,000 元憑空消失

有 Transaction:兩步包成一個交易
→ 步驟 2 失敗,步驟 1 自動還原(Rollback)
→ 要嘛兩步都成功,要嘛都沒發生,錢不會消失

企業系統裡「建訂單+扣庫存」「結帳的十個步驟」都靠 Transaction 保護。它也解釋了一種現象:批次作業中途失敗,資料卻「完全沒進去」而不是「進去一半」——那正是 Transaction 整包還原的結果。

為什麼要學這個

兩個馬上用得到的場景

場景一:回報「系統變慢」時,提供對的線索

「查詢慢」的回報,如果能補上「查哪個功能、用什麼條件查、資料量大概多少」,工程師就能快速判斷是不是索引問題。您甚至可以主動問:「這個查詢條件的欄位,有建索引嗎?」——這一問常讓工程師眼睛一亮。

場景二:聽懂系統文件與交接內容

接手既有系統時,文件裡寫著「報表都查 View,不要直接查底層表」「月結跑 SP_MonthlyClose」,現在您看得懂了:前者是為了統一查詢邏輯,後者是月結流程被封裝成了預存程序。看懂這些,交接會議的吸收效率完全不同。

核心概念 1/2

三個名詞,三種「預先準備」

概念一:它們的共通精神是「先準備好,重複使用」

Index 預先建好目錄、View 預先存好查詢、Stored Procedure 預先存好流程。三者都是把「常用的東西」事先準備好,換取速度、一致性與便利。理解這個精神,三個名詞就不再是孤立的術語。

概念二:View 是透鏡,不是另一份資料

查 View 看到的資料,來自底層的表;底層資料一變,View 看到的就跟著變。所以「View 的數字不對」,根源幾乎都在底層資料或 View 定義的查詢邏輯(回到 4-6、4-7 的排查方法),不是 View「存了舊資料」。

核心概念 2/2

速度與一致性,各有守護者

概念三:查詢慢,先想 Index;資料半套,先想 Transaction

兩個常見症狀,各自對應一個機制:「越查越慢」常是資料量長大後索引缺席;「批次失敗後資料整包沒進去」是 Transaction 還原的正常表現,不用驚慌,修正錯誤後重跑即可。

概念四:Schema 是溝通用的地圖

要理解一個系統的資料,跟工程師要一份 Schema(或資料表清單與關聯圖)是最快的起手式。單元 4 學過的一切——表、欄位、型別、PK、FK、關聯形狀——正是讀懂這份地圖需要的全部語彙。您現在已經具備了。

容易混淆

案例

Northstar Electronics 的月結流程

Northstar 的 ERP 每月一日凌晨自動執行月結,某天財務發現月結資料是空的:

月結設計:Stored Procedure「SP_MonthlyClose」,包含 10 個步驟,
          整個流程包在一個 Transaction 裡
當天狀況:步驟 7(計算部門分攤)因一筆異常資料而失敗
結果:Transaction 整包還原 → 前 6 步的資料也都不存在
      月結資料「完全空白」,而不是「做到一半」
  1. 有哪些角色?ERP、SP_MonthlyClose 預存程序、Transaction 機制、財務團隊。
  2. 資料如何移動?月結流程依序執行 10 步,全部成功才一次生效。
  3. 各機制負責什麼?Stored Procedure 封裝流程;Transaction 保證不會留下「半套」的月結資料。
  4. 哪裡可能出錯?如果誤以為「資料全空 = 流程根本沒跑」,排查方向就偏了;實際上流程跑了、在第 7 步失敗、被整包還原。
  5. 第一個合理的檢查方向?請工程師查執行紀錄,確認流程停在哪一步、錯誤訊息是什麼——修正那筆異常資料後重跑,通常就能完成月結。

常見誤解

三個很多人踩過的坑

誤解一:「索引越多越好,乾脆每個欄位都建」

索引加速查詢,卻拖慢寫入、佔用空間。建哪些索引要看實際查詢模式取捨,不是無腦全建。

誤解二:「View 存著一份舊資料,所以數字不對」

View 通常不存資料,它即時反映底層表。數字不對,查 View 的定義邏輯或底層資料,別怪 View「過期」。

誤解三:「批次失敗後資料全空,代表流程根本沒執行」

更可能是 Transaction 整包還原的結果。查執行紀錄確認停在哪一步,比從頭懷疑排程沒跑更快找到答案。

遇到問題先查什麼

「查詢變慢」的檢查順序

  1. 確認慢的範圍。是特定查詢慢,還是整個系統都慢?整體慢可能是主機資源問題,特定查詢慢才往索引方向想。
  2. 整理這個查詢的條件與資料量。查哪張表?用什麼欄位篩選?表裡大約幾筆資料?
  3. 問工程師:查詢條件的欄位有沒有索引。資料量長大後,缺索引的問題會逐漸浮現。
  4. 確認是不是最近才變慢的。如果是,問問最近資料量是否大增、或查詢邏輯是否被改過。
  5. 把以上資訊一次提供給工程師。範圍、條件、資料量、時間點,四項齊全,排查速度完全不同。
重點整理

帶走這四句話

  1. Index 是欄位的目錄:加速查詢,但增加寫入成本;「查詢慢」先想它。
  2. View 是存好的查詢視角,像透鏡不存資料;Stored Procedure 是存好的流程,可以執行異動。
  3. Transaction 保證「全有或全無」:批次失敗後資料整包消失,是還原機制的正常表現。
  4. Schema 是資料庫的地圖;單元 4 教過的語彙,已足夠您讀懂它、跟工程師對話。

檢核 1/4

查詢很慢,工程師說「加個 Index 就快了」。Index 的作用是?

正確答案是 A。Index 是預先建好的查找結構,查詢時直接跳到目標,免去全表掃描,大資料量下速度可差上百倍。
檢核 2/4

是非題:View 裡面存著一份獨立的資料副本,所以底層表更新後,View 的內容會過期。

正確答案是「否」。View 通常不存資料,查它時即時執行背後的查詢,底層一變它就跟著變。數字不對要查定義邏輯或底層資料。
檢核 3/4

「A 帳戶扣款成功、B 帳戶入帳失敗時,扣款自動還原」——這是哪個機制在保護?

正確答案是 C。Transaction 把多個異動包成「全有或全無」的單位,任一步失敗就整包還原,守住資料的一致性。
檢核 4/4

情境題:月結批次半夜失敗,早上發現月結資料「完全空白」。最合理的解讀是?

正確答案是 B。包在 Transaction 裡的批次,中途失敗會整包還原,所以資料是「全空」而非「半套」。查執行紀錄確認停在哪一步,修正後重跑即可。
