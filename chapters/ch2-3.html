HTTP Method
GET 和 POST 到底差在哪

預估 30 分鐘 ・ 建議先讀完 2-2 再來。

上一章我們拆開了 Request 這封信,看到裡面有 Header、Body。這一章要看信封上另一個重要標記:這封信到底是「想看」還是「想改」。這就是 HTTP Method(方法)的角色。

  • 說出 GET 和 POST 最核心的差異。
  • 認得 PUT、PATCH、DELETE 這幾個方法各自的用途。
  • 看到工程師說「這是一個 POST 請求」時,知道大概在做什麼事。
情境

「查資料跟改資料,系統對待的方式不一樣」

Northstar Electronics 的客服系統偶爾會出現一個怪現象:客戶不小心重新整理頁面,結果同一筆客訴單被重複建立了兩次。工程師解釋:

「這個表單送出用的是會重複執行的方法,重新整理等於再送一次,難怪會建兩筆。如果當初設計成用另一種方法,就不會有這個問題。」

這句話點出了一件事:HTTP 裡「查看資料」和「新增/修改資料」用的是不同的方法,而且這些方法本身的「脾氣」不一樣——有些重複做沒關係,有些重複做會出事。這一章就是要搞懂這些方法的脾氣。

情境例子 1/2

看菜單,還是下訂單

回到餐廳的比喻。您走進餐廳做的事,大致分兩種:

第一種,翻閱菜單看有什麼菜。您看幾次都沒關係,菜單不會因為您多翻幾次就變出新菜色,店裡的庫存也不會因此改變。這是 GET——用來「查看、讀取」資料的方法,理論上重複做,結果應該都一樣,不會改變任何東西。

第二種,跟服務生下訂單。每喊一次「我要一份牛排」,廚房就真的多做一份牛排出來。喊兩次,就變成兩份。這是 POST——用來「新增」資料的方法,每執行一次,通常就會真的新增一筆東西。

先記住這句 GET 像看菜單,查看不會留下痕跡;POST 像下訂單,每喊一次就真的多一份。這正是開頭客訴單被重複建立的原因——表單送出用的是「下訂單」的方式,重新整理等於又喊了一次。
情境例子 2/2

改地址,還是取消訂單

除了看菜單和下訂單,還有另外兩種常見的事:

改一筆已經存在的訂單資料,例如把送貨地址改掉。不管您把地址改成同一個值幾次,結果都是「訂單的地址變成這個值」,不會因為改第二次就變出第二筆訂單。這是 PUT 和 PATCH——用來「修改」已存在資料的方法(兩者細節上有差,PUT 通常是整筆換掉,PATCH 是只改某幾個欄位,但對顧問來說,先記得「都是修改」就夠用)。

取消一筆訂單。取消過一次之後,這筆訂單就不存在了,再取消一次也不會有變化(因為它已經被取消了)。這是 DELETE——用來「刪除」資料的方法。

回到開頭的情境:客訴單表單原本該用「新增一次就該剛好一筆」的邏輯處理,但如果系統設計沒有防範「使用者重新整理頁面」這種情況,POST 被觸發兩次,就會真的產生兩筆客訴單。

正式定義

五個常見方法的正式說法

方法用途重複執行的結果
GET查詢、讀取資料不改變任何東西,重複查詢結果應該一樣
POST新增一筆資料每執行一次,通常就多一筆新資料
PUT整筆更新已存在的資料重複執行結果相同(都變成同一個最終狀態)
PATCH只更新資料中的部分欄位通常重複執行結果相同
DELETE刪除已存在的資料刪除後再刪一次,通常不會有變化

「重複執行結果都一樣」這個特性,有個正式名稱叫 Idempotency(冪等性),單元 6 會再深入教。這裡先記得:GET、PUT、PATCH、DELETE 通常具備這個特性,POST 通常不具備——這正是為什麼「表單重複送出」問題,幾乎都是發生在用 POST 新增資料的情境。

串起來看

一筆客訴單的四種操作方式

用客服系統的客訴單,走一遍四種方法各自對應的操作:

客服人員打開客訴單列表          → GET(查看所有客訴單)
客戶送出一筆新的客訴             → POST(新增一筆客訴單)
客服人員把處理狀態改成「已完成」  → PATCH(更新客訴單的某個欄位)
主管發現這是一筆誤植的重複資料    → DELETE(刪除這筆客訴單)

四個動作各自對應一種方法,背後的資料庫操作(1-1 教過的資料庫層)也不一樣:GET 只是讀取,不改變資料庫內容;POST、PATCH、DELETE 則會真的改變資料庫裡的紀錄。

為什麼要學這個

兩個馬上用得到的場景

場景一:理解「重新整理頁面」為什麼有時候很危險

如果一個操作背後是 POST(新增),使用者在等待回應時重新整理頁面,很可能會再送出一次,造成重複資料。理解這一點,您就能理解為什麼有些系統會在送出後把按鈕鎖住,或跳出「請勿重複點擊」的提示——這是在保護一個天生不具備「重複執行安全」特性的方法。

場景二:跟工程師討論問題時更精準

與其說「送出表單後資料重複了」,不如說「這個表單用的是 POST,使用者重新整理後好像又送了一次」。這種描述能幫助工程師快速鎖定,是不是該加上防止重複送出的機制。

核心概念 1/2

方法的選擇,反映了「這個動作的本質」

概念一:GET 不該拿來做「有副作用」的事

正確的設計原則是:GET 只用來查看資料,不應該用來新增或修改任何東西。如果一個系統把「刪除」動作設計成用 GET 觸發,可能會出現使用者只是點了一個連結,資料就被意外刪除的風險。

概念二:POST 是唯一「預設不具備重複安全」的常見方法

這也是為什麼「防止表單重複送出」幾乎都是在處理 POST 的情境。設計良好的系統,通常會加上額外機制(例如送出後立刻鎖住按鈕),避免使用者操作失誤造成的重複新增。

核心概念 2/2

PUT、PATCH、DELETE 都是對「已存在」資料的動作

概念三:PUT/PATCH/DELETE 的前提是資料已經存在

要修改或刪除一筆資料,前提是這筆資料本來就存在。如果嘗試修改一筆不存在的資料,系統通常會回傳錯誤(單元 2 下一章會教的 Status Code 就是用來表達這類結果)。

概念四:方法只是「意圖的標記」,實際行為還是看系統怎麼實作

HTTP Method 是一種約定俗成的標記慣例,幫助溝通雙方理解這次請求的意圖。但實際上系統背後真正怎麼處理,還是取決於工程師怎麼寫程式——方法本身只是溝通的慣例,不是強制的保證。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
PUT vs PATCHPUT 通常是把整筆資料換成新的內容(即使只改一個欄位,也要把全部欄位都傳過去);PATCH 只需要傳要修改的那幾個欄位。對顧問來說,先記得兩者都是「修改」就足夠應付大多數溝通。
GET vs POSTGET 是查看,不會改變資料,可以放心重複執行;POST 是新增,重複執行通常會產生重複資料,是造成表單重複送出問題的主要原因。
案例

Northstar Electronics 的重複客訴單事件

客服系統的客訴表單發生重複建立的問題,追查後發現:

使用者行為:客戶填完表單、按下送出,畫面等待反應時間較長
使用者反應:以為沒有送出成功,又按了一次送出按鈕
系統設計:送出按鈕沒有在第一次點擊後立即鎖住
  1. 有哪些角色?客戶(Client 端操作者)、客服系統的前端與後端。
  2. 資料如何移動?每次點擊送出按鈕,都各自觸發一次 POST 請求,新增一筆客訴單。
  3. 各部分負責什麼?前端的送出按鈕理應在第一次點擊後立即停用,避免使用者重複觸發 POST。
  4. 哪裡可能出錯?如果送出按鈕沒有做防止重複點擊的設計,使用者的每一次點擊,都是一次獨立、會真的新增資料的 POST 請求。
  5. 第一個合理的檢查方向?先確認送出按鈕在點擊後,有沒有立即被停用或顯示「處理中」的狀態,這是最常見也最容易被忽略的細節。

常見誤解

三個很多人踩過的坑

誤解一:「GET 和 POST 只是技術上的名稱差異,沒有實質不同」

兩者的核心差異在於是否會改變資料。GET 應該是安全、可重複的查詢;POST 通常帶有實質的新增動作,重複執行的後果完全不同。

誤解二:「所有的表單送出,都應該用 POST」

如果表單的目的是查詢或篩選(例如搜尋表單),用 GET 反而更合理,因為查詢不該對資料造成任何改變。是否該用 POST,要看這個動作是不是真的在「新增」東西。

誤解三:「重複點擊送出按鈕沒關係,系統會自動處理」

是否會自動防止重複新增,取決於系統有沒有特別設計這個機制。不能預設所有系統都會妥善處理,這正是本章案例發生的原因。

遇到問題先查什麼

「資料被重複建立」的檢查順序

  1. 確認這個操作背後用的是哪個方法。通常是 POST 才會有重複新增的疑慮。
  2. 確認使用者當下的操作情境。是否重新整理頁面、重複點擊按鈕、或連續送出。
  3. 確認前端有沒有防止重複送出的機制。例如送出後立即停用按鈕。
  4. 確認後端有沒有額外的防重複檢查。例如檢查短時間內是否已有相同內容的紀錄。
  5. 把重複發生的時間與使用者操作情境提供給工程師。這比只說「資料重複了」有效率得多。
重點整理

帶走這四句話

  1. GET 用來查看資料,理論上不會改變任何東西,可以放心重複執行。
  2. POST 用來新增資料,重複執行通常會真的產生重複資料,這是表單重複送出問題的根源。
  3. PUT、PATCH 用來修改已存在的資料,DELETE 用來刪除,三者重複執行通常結果相同。
  4. 方法的選擇反映了這個動作的本質,理解這一點能幫助您跟工程師更精準地討論問題。
檢核 1/4

下列哪一個方法,重複執行通常會造成資料重複?

正確答案是 B。POST 用來新增資料,每執行一次通常就會真的多一筆,這是表單重複送出問題最常見的成因。
檢核 2/4

是非題:GET 方法理論上不應該用來改變資料庫裡的內容。

正確答案是「是」。GET 的設計原則是只用來查看資料,不該帶有新增或刪除等會改變資料的副作用。
檢核 3/4

客服人員要把一筆客訴單的處理狀態改成「已完成」,這個動作對應哪一種方法?

正確答案是 C。修改一筆已存在資料的部分欄位(狀態),對應的是 PATCH(或 PUT)這類更新方法。
檢核 4/4

情境題:客戶回報「填完表單按送出,結果建立了兩筆一模一樣的資料」。第一個合理的懷疑方向是?

正確答案是 B。表單送出通常是 POST,重複執行會產生重複資料,最常見的原因是送出按鈕沒有做好防止重複點擊的設計。
