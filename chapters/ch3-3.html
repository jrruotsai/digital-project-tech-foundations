表單欄位與資料送出
form 與 input

預估 30 分鐘 ・ 建議先讀完 3-1、2-2、2-3 再來。

1-3 案例裡提過的報名表單,這一章要正式拆開來看它的 HTML 結構,並接回 2-2、2-3 教過的 Request 和 Method,把「填表單」這個動作,從頭到尾串成完整的一條線。

  • 認得 form、input、select、textarea、button 這幾個表單元素。
  • 說明 id、class、name 這三個常見屬性的差異。
  • 看到一段表單 HTML,能判斷欄位的角色和送出方式。
情境

「這個欄位的 name 打錯了,難怪後端收不到資料」

Northstar Electronics 的報名表單,顧問發現後端資料庫裡,某個欄位永遠是空的,詢問工程師後得到答案:

「喔,前端那個輸入欄位的 name 屬性打錯字了,跟後端預期接收的名稱對不上,難怪資料傳不過去。」

這句話點出一個關鍵:表單能不能正確把資料送到後端,牽涉到一個叫 name 的屬性有沒有設對。這一章要把表單的完整結構講清楚,包含這個常常被忽略、卻很關鍵的細節。

情境例子 1/2

一張紙本申請表

想像您在填一張紙本申請表:表格最上方有標題「活動報名表」,底下是一格一格要填的欄位:姓名、Email、公司,最後有一個「送出」的按鈕(或印章)。

HTML 裡的 form(表單)就是這整張紙本表格的邊框——把所有相關欄位框在一起,並且知道「填完之後要送到哪裡去」。input(輸入)就是表格上一格一格讓您填寫的空格,可以是文字欄位、也可以是核取方塊(透過 3-1 教過的 type 屬性決定)。

先記住這句 form 是整張表格的框架,input 是表格裡的每一格空白。沒有 form 包起來,瀏覽器不知道這些 input 該一起送到哪裡去。
情境例子 2/2

下拉選單、大段留言框,和送出的印章

除了讓您手打文字的空格,表單還有其他形式的欄位。select(選擇)是一個下拉選單,讓您從固定的幾個選項裡挑一個,像是選擇「國家」時常看到的下拉清單。textarea(文字區域)則是給您打一大段文字的留言框,適合像「備註」這類需要多行文字的欄位。

填完整張表格,最後按下的那個「送出」印章,就是 button(按鈕)。按下它,通常會觸發 2-3 教過的 POST 方法,把整張表格的內容打包送給後端。

回到開頭的問題。每一格欄位除了看得到的文字內容,背後還帶著一個 name 屬性,告訴後端「這一格資料,應該對應到哪個欄位名稱」。如果這個屬性打錯字,就像在紙本表格上,填的欄位標籤和收發室認得的名稱對不上,資料自然收不到正確的地方。

正式定義

幾個表單元素與屬性的正式說法

名稱定義
form把一組相關欄位框在一起,並定義送出目的地與方式的容器元素
input讓使用者輸入資料的欄位,可透過 type 屬性變化成文字、核取方塊等不同形式
select下拉選單,讓使用者從固定選項中選擇
textarea多行文字輸入區域
button可點擊的按鈕,常用來觸發表單送出
id指定給某一個元素的唯一識別名稱,理論上整個頁面裡不該重複
class可以套用在多個元素上的分類標籤,常用於統一設定樣式(3-4 會教)
name表單欄位送出資料時,對應到後端接收的欄位名稱
串起來看

一張報名表單的完整結構

把 1-3 教過的活動報名表單,實際用 HTML 呈現出來:

<form action="/register" method="POST">
  <input type="text" name="firstName" placeholder="First Name">
  <input type="text" name="lastName" placeholder="Last Name">
  <input type="email" name="email" placeholder="Email">
  <select name="country">
    <option>Taiwan</option>
    <option>Japan</option>
  </select>
  <input type="checkbox" name="consent"> 同意接收行銷資訊
  <button type="submit">Submit</button>
</form>

逐段解釋:form 上的 action 屬性(2-1 教過的網址概念),告訴瀏覽器資料要送到哪個網址;method="POST" 呼應 2-3 教過的方法;每個 input 的 name 屬性,決定後端會用什麼名稱收到這筆資料;按下 button,就會把整張表單打包成一次 POST Request 送出去。

為什麼要學這個

兩個馬上用得到的場景

場景一:排查「資料送不出去或送錯地方」的問題

當某個欄位的資料一直收不到,可以先請工程師確認這個欄位的 name 屬性,是否跟後端期待接收的名稱一致。這是表單資料傳遞失敗最常見、也最容易被忽略的原因之一。

場景二:看懂需求訪談中「欄位設計」的討論

客戶討論表單需求時,理解 select(固定選項)、textarea(自由文字)、input checkbox(是非選擇)這幾種形式的差異,能幫助您更準確地把客戶想要的欄位形式,轉達給開發團隊。

核心概念 1/2

form 決定「送去哪裡、怎麼送」

概念一:form 上的 action 和 method,決定了整張表單的行為

action 屬性(網址)和 method 屬性(通常是 GET 或 POST,呼應 2-3),合起來決定了按下送出按鈕後,資料會被送到哪個網址、用哪種方式送。這兩個屬性設定錯誤,會導致整張表單完全送不出去,而不只是某個欄位的問題。

概念二:每個欄位的 name,是後端辨識資料的依據

後端收到表單資料時,是靠每個欄位的 name 屬性來對應「這是哪一項資料」,不是靠畫面上顯示的文字(那只是給使用者看的提示,通常用 placeholder 屬性設定,3-1 提過)。

核心概念 2/2

id、class、name 各自扮演不同角色

概念三:id 是唯一的,class 可以重複套用

id 通常用來精確指定「這一個」元素(例如某個特定的欄位);class 則可以套用在多個元素上,常用來統一設定一批元素的樣式(3-4 會教)。兩者角色不同,不能混為一談。

概念四:name 專門服務於表單資料的傳遞,跟 id、class 目的不同

id 和 class 主要跟畫面呈現、程式操作有關;name 則專門用於表單送出時,標示每個欄位對應的資料名稱。三者看起來都是「幫元素取名字」,但各自服務的目的完全不同。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
id vs classid 是唯一的,一個頁面裡理論上不該有兩個相同的 id;class 可以套用在多個元素上,用來做批次的樣式設定或程式操作。
placeholder vs nameplaceholder 是給使用者看的提示文字(欄位還沒填寫時,淡淡顯示的說明);name 是後端辨識這筆資料對應哪個欄位的依據,使用者通常看不到 name 的內容。
案例

Northstar Electronics 的欄位對應錯誤

報名表單的「公司名稱」欄位,顧問發現後端資料庫裡永遠收不到這筆資料,追查後發現:

前端 HTML:<input type="text" name="compnay">（拼字打錯,少了一個字母排列錯誤)
後端期待接收的欄位名稱:company
  1. 有哪些角色?前端的表單 HTML、後端的資料接收邏輯。
  2. 資料如何移動?使用者填寫的公司名稱,雖然畫面上顯示正常,但送出時的欄位名稱跟後端不一致。
  3. 各部分負責什麼?後端程式是根據事先講好的欄位名稱來接收資料,如果名稱對不上,後端會完全收不到這筆資料,也不會特別報錯。
  4. 哪裡可能出錯?這種拼字錯誤在畫面上完全看不出來(使用者看到的提示文字可能完全正確),只有檢查 HTML 原始碼或後端紀錄才能發現。
  5. 第一個合理的檢查方向?用 2-8 教過的開發者工具,檢查這個欄位的 name 屬性,並跟後端工程師確認期待接收的正確名稱是否一致。

常見誤解

三個很多人踩過的坑

誤解一:「畫面上的提示文字正確,資料就一定送對地方」

畫面上顯示的提示文字(placeholder)和實際傳遞資料用的欄位名稱(name)是兩回事,前者正確不代表後者也正確。

誤解二:「表單填完按送出,一定是即時生效的」

是否即時生效,牽涉到 1-6 教過的即時或批次處理概念,跟表單本身的 HTML 結構是不同層次的問題。

誤解三:「input 只能是文字輸入欄位」

input 透過不同的 type 屬性,可以變成文字欄位、核取方塊、單選按鈕等多種形式,不是只有一種樣貌。

遇到問題先查什麼

「表單資料收不到」的檢查順序

  1. 確認表單本身有沒有成功送出。用 2-8 教過的 Network 工具確認有沒有出現對應的 Request。
  2. 確認 form 的 action 和 method 設定是否正確。是否指向正確的網址與方法。
  3. 確認出問題的那個欄位的 name 屬性。是否跟後端期待接收的名稱一致。
  4. 確認是不是拼字或大小寫上的細微差異。這類錯誤在畫面上完全看不出來。
  5. 把具體的欄位名稱提供給前後端工程師確認。讓雙方能對照名稱是否一致。
重點整理

帶走這四句話

  1. form 是整張表格的框架,決定資料要送去哪裡、用什麼方式送;input、select、textarea 是不同形式的填寫欄位。
  2. 每個欄位的 name 屬性,是後端辨識資料的依據,跟畫面上顯示的提示文字是兩回事。
  3. id 是唯一識別,class 可以套用在多個元素上,name 專門服務表單資料傳遞,三者角色不同。
  4. 表單資料收不到,經常是 name 拼字或大小寫對不上造成的,這種錯誤在畫面上完全看不出來。
檢核 1/4

決定表單欄位資料要對應到後端哪個欄位名稱的屬性是什麼?

正確答案是 B。name 屬性是後端辨識這筆資料對應哪個欄位的依據,跟畫面呈現的提示文字無關。
檢核 2/4

是非題:畫面上的提示文字(placeholder)顯示正確,不代表資料一定會送到後端正確的欄位。

正確答案是「是」。提示文字和欄位的 name 屬性是兩回事,前者正確不代表後者也一定正確。
檢核 3/4

讓使用者從固定選項中挑選一個的表單元素是哪一個?

正確答案是 C。select 是下拉選單,讓使用者從固定的幾個選項中選擇一個。
檢核 4/4

情境題:表單的某個欄位一直收不到資料,但畫面顯示完全正常。第一個合理的檢查方向是?

正確答案是 A。name 屬性的拼字錯誤在畫面上完全看不出來,卻是資料傳遞失敗最常見的原因之一,值得優先檢查。
