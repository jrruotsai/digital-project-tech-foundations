Duplicate 與 Matching
重複資料是怎麼來的

預估 30 分鐘 ・ 建議先讀完 5-2。

客戶名單裡有「王小明」和「小明 王」，Email 也不一樣。這是兩位客戶，還是同一人留下的兩筆紀錄？電腦不會自己知道，顧問得先把判斷規則說清楚。

  • 理解 Duplicate 與 Matching 的差別。
  • 知道 Email 為何不一定適合當唯一識別值。
  • 能說明合併重複資料前必須保留哪些資訊。
情境

「同一家公司，報表卻算成三個客戶」

Northstar 的業務報表出現三筆看似相同的公司：

雲端商貿股份有限公司
雲端商貿(股)公司
Cloud Commerce Co., Ltd.
「名稱不一樣，可是電話和地址相同。可以直接合併嗎？萬一其實是總公司和子公司呢？」

看到相似資料只能說「可能重複」。要不要合併，還需要 Matching 規則和人工判斷。

情境例子 1/2

通訊錄裡的兩個「阿哲」

手機通訊錄裡有兩個阿哲，一筆只有公司電話，另一筆只有手機。光看名字，您不能確定是不是同一人；若公司、職稱與 Email 也相同，把握才會提高。

Duplicate（重複資料）是同一個現實對象被存成兩筆以上紀錄。Matching（比對）則是根據一組欄位與規則，判斷不同紀錄是否代表同一個對象。

先記住這句Duplicate 是問題狀態；Matching 是用來找出疑似重複的方法。
情境例子 2/2

只比 Email，會漏掉也會誤判

同一個人可能用公司信箱註冊產品、用私人信箱訂閱電子報。若只看 Email，系統會把他當成兩個人。反過來，一家小公司可能共用 info@company.com；若把 Email 相同就當同一人，又會把不同聯絡人合併。

Email 很有用，卻不天然等於人的身分。比對通常要組合姓名、電話、公司、地址或內部識別碼，並考慮資料是否可能變動。

別急著合併先定義「完全相同」「高度相似」和「需要人工確認」三種結果，再決定後續動作。
正式定義

四個常用名詞

名詞定義
Duplicate同一個現實對象在資料集中出現兩筆以上紀錄
Matching依欄位、規則或分數判斷兩筆紀錄是否代表同一對象
Exact Match指定欄位的值完全相同，例如會員編號完全一致
Fuzzy Match允許拼字、格式或順序差異的相似比對，例如公司名稱縮寫

Matching 找到的是候選對象，不代表一定能自動 Merge（合併）。規則越寬鬆，抓到的候選越多，誤合併風險也越高。

串起來看

一組重複資料怎麼被處理

以兩筆疑似相同的客戶資料為例：

先用穩定識別碼做 Exact Match
→ 沒比到，再比較姓名、電話、公司與地址
→ 依規則算出相似程度
→ 高信心：送入合併流程
→ 中信心：交由 Data Steward 人工確認
→ 低信心：保留為不同紀錄
→ 合併時記錄來源、保留值與異動紀錄

最後一步常被忽略。若合併後不知道值從哪裡來，日後發現誤合併就很難拆回去。

為什麼要學這個

兩個會直接影響專案的場景

場景一：整併兩套客戶名單

上線前不能只問「總共有幾筆」，還要問重複率、比對欄位、合併原則，以及無法確定的資料由誰確認。這些工作會直接影響時程。

場景二：行銷人數與營收對不上

同一人若有三筆紀錄，可能收到三封信；兩個人被誤合併，消費紀錄又會混在一起。報表異常未必是公式錯，資料身分也可能出了問題。

核心概念 1/2

比對規則要看資料特性

概念一：唯一識別值要穩定、唯一、可取得

內部客戶 ID 通常比姓名穩定，但跨系統時未必共用同一組 ID。Email 可能更換、共用或缺漏，因此不一定能單獨當唯一識別值。

概念二：先標準化，再比對

「02-2345-6789」和「0223456789」其實是同一支電話。先去除格式差異，Matching 才不會把外觀不同誤判成內容不同。下一章會完整說明 Standardization。

核心概念 2/2

漏抓與誤合併之間要取捨

概念三：規則太嚴會漏掉，太鬆會抓錯

要求姓名、電話、Email 全部相同，安全但容易漏掉改過電話的人；只要求姓名相同，又會把同名者混在一起。適合的門檻取決於用途與風險。

概念四：合併要有保留規則

確認為同一人後，還要決定哪一筆是主紀錄、各欄位保留哪個值、關聯的訂單與案件如何搬移。合併不是按一下刪除重複列。

容易混淆

三組常被混在一起的概念

容易混淆怎麼分
Duplicate vs Similar相似只表示長得像；確認代表同一對象，才算重複
Matching vs MergingMatching 找候選；Merging 決定並執行合併
Unique Field vs Reliable Identity欄位設定成不重複，不代表它在現實世界永遠能代表同一個人

看起來相似不等於可以合併。只要記住這點，就能避開最危險的誤判。

案例

Northstar 的客戶比對規則

Northstar 先用一組保守規則找出候選資料：

規則 A：來源客戶 ID + 來源系統完全相同 → 同一筆來源資料
規則 B：姓名標準化後相同 + 手機相同 → 高信心候選
規則 C：公司 Email 相同 + 公司名稱相似 → 中信心候選
規則 D：只有姓名相同 → 不自動合併
  1. 角色：CRM 營運專員檢查候選，客戶資料 Owner 核定規則。
  2. 資料流：候選清單先進人工審查，確認後才合併。
  3. 可能出錯：共用公司信箱讓兩位聯絡人被判成同一人。
  4. 第一個檢查方向：查看這次判定用了哪些欄位，以及原始值是否先被標準化。
顧問可以先問「判斷同一人的最低條件是什麼？不確定時由誰確認？」

常見誤解

三個重複資料的陷阱

誤解一：「Email 一樣，就是同一個人」

家庭或公司可能共用信箱。Email 相同是線索，不是所有情況下都可靠的身分證。

誤解二：「名字不同，就一定不是同一人」

中英文名、空白、順序與縮寫都會造成差異。需要先標準化，再搭配其他欄位判斷。

誤解三：「找到重複就直接刪一筆」

被刪紀錄可能連著訂單、同意狀態或客服案件。正確做法是設計合併與保留規則。

遇到問題先查什麼

疑似重複資料的五步檢查

  1. 確認討論的是人、公司、產品，還是其他資料對象。
  2. 查看每筆資料的來源系統、來源 ID 與建立時間。
  3. 把姓名、電話、地址等欄位先做格式標準化再比較。
  4. 檢查命中的 Matching 規則與信心門檻。
  5. 合併前確認主紀錄、欄位保留與關聯資料處理方式。

前四步是在確認證據。證據還不夠時，保留兩筆通常比誤合併安全。

重點整理

帶走這四句話

  1. Duplicate 是同一對象有多筆紀錄；Matching 是找出疑似重複的方法。
  2. Email 可能更換、共用或缺漏，不一定適合單獨當唯一識別值。
  3. 先標準化再比對，規則太嚴會漏抓，太鬆會誤合併。
  4. 合併前要決定主紀錄、欄位保留、關聯搬移與異動紀錄。
檢核 1/4

Duplicate 與 Matching 的關係，哪個說法正確？

正確答案是 B。重複是資料狀態；比對是依規則找出疑似重複。
檢核 2/4

為什麼 Email 不一定適合單獨當唯一識別值？

正確答案是 C。同一人可能有多個信箱，不同人也可能共用信箱。
檢核 3/4

「02-2345-6789」與「0223456789」在比對前最適合先做什麼？

正確答案是 A。先去除格式差異，才能比較真正的內容。
檢核 4/4

兩筆客戶資料只有姓名相同，其他欄位不足。第一步怎麼做較安全？

正確答案是 B。同名不代表同一人；先補證據，避免誤合併。
