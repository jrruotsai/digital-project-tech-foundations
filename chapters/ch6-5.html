Header、Body 與 Parameter — 資料放在哪裡

預估 30 分鐘 ・ API 與資料交換

Header 放傳輸與驗證資訊，Body 放主要內容，Parameter 用來指定查詢或路徑條件。

  • 能說明 Header／Body：Header 放控制與驗證資訊；Body 放請求或回應的主要內容。
  • 能區分 Header／Body 與 Parameter 在流程中的用途。
  • 遇到問題時，先確認 Endpoint、HTTP 方法與版本。
案例起點

Header／Body 和 Parameter 沒有寫清楚時

Northstar 的規格同時提到 Header／Body 與 Parameter，卻沒有標出兩者的適用條件與處理邊界。

Header／Body：Header 放控制與驗證資訊；Body 放請求或回應的主要內容。；Parameter：用來指定路徑、查詢或控制行為的輸入值。

測試時要避免這個結果：把 Token 放錯位置，服務端會把有效憑證視為缺少驗證。

概念拆解 1/2

Header／Body 的處理邊界

Header／Body：Header 放控制與驗證資訊；Body 放請求或回應的主要內容。

套用到本章情境，第一步是確認 Endpoint、HTTP 方法與版本。確認 Header／Body 的適用範圍後，再檢查 Parameter，不從畫面直接猜原因。

判斷重點說明 Header／Body 時，要交代誰提供輸入、系統套用哪條規則，以及會產生什麼結果。
概念拆解 2/2

Parameter 與 Header／Body 的差別

Parameter：用來指定路徑、查詢或控制行為的輸入值。

Header 放傳輸與驗證資訊，Body 放主要內容，Parameter 用來指定查詢或路徑條件。

不要混用Header／Body 與 Parameter 不能互換；規格要分別寫出兩者的輸入、處理方式與輸出。
精確定義

本章名詞的工作定義

術語定義判斷時要問
Header／BodyHeader 放控制與驗證資訊；Body 放請求或回應的主要內容。它接收什麼輸入，完成後留下什麼結果？
Parameter用來指定路徑、查詢或控制行為的輸入值。它和 Header／Body 的邊界在哪裡？
Evidence能證明 Header／Body 或 Parameter 實際結果的原始資料，例如 Request／Response、Log、時間、版本與識別值。哪一筆原始紀錄能支持目前的判斷？

定義 Header／Body 與 Parameter 時，必須寫出用途、處理邊界和可觀察結果。

流程位置

Header／Body 在整體流程中的位置

Client → Endpoint → Request → Authentication → Response → Log

沿著流程逐段比對輸入與輸出。若 Header／Body 的輸入正確而 Parameter 的輸出不符預期，就從兩者之間的邊界開始查。

規格與設計

整合顧問需要把「Header、Body 與 Parameter — 資料放在哪裡」改寫成可確認的規則，放進 API 規格與錯誤紀錄。

問題排查

先確認 Endpoint、HTTP 方法與版本，再查看 Request Header、Body 與 Parameter。這能分辨問題是否位於 Header／Body 或 Parameter。

核心概念 1/2

輸入、規則與輸出要分開寫

輸入

列出 Header／Body 開始前已知的資料、身分、事件或設定。輸入不明，後面的結果就無法比較。

規則與輸出

Parameter 執行後，保存實際套用的規則版本、Response、狀態或資料值。不要用「看起來正常」代替可查證結果。

核心概念 2/2

成功條件和失敗條件都要具體

成功條件

Header 放傳輸與驗證資訊，Body 放主要內容，Parameter 用來指定查詢或路徑條件。 驗收 Header／Body 與 Parameter 時，寫明預期狀態、資料值與完成時間。

失敗條件

把 Token 放錯位置，服務端會把有效憑證視為缺少驗證。 Header／Body 或 Parameter 失敗時，保留原始輸入、環境、時間與識別值。

容易混淆

用處理邊界分清楚三組概念

容易混淆實際差別
Header／Body vs ParameterHeader／Body：Header 放控制與驗證資訊；Body 放請求或回應的主要內容。 Parameter：用來指定路徑、查詢或控制行為的輸入值。
規格存在 vs 實際生效Header／Body 的文件是預期規則；Parameter 的設定值、Response 或 Log 才能證明實際結果。
單次成功 vs 穩定可重現Header／Body 單次成功不代表 Parameter 在其他條件下也正確；要用相同輸入重做。

常見誤解

三個會讓排查走錯方向的假設

把 Header／Body 和 Parameter 當成同一件事

把 Token 放錯位置，服務端會把有效憑證視為缺少驗證。

只看最後畫面

畫面是流程的最後一層。仍要核對 Header／Body 的輸入與 Parameter 的執行紀錄。

一次改多個條件

同時修改 Header／Body 與 Parameter 的條件，即使恢復也無法歸因。每次只改一項。

排查順序

Header／Body 發生異常時的排查順序

  1. 1. 確認 Endpoint、HTTP 方法與版本。
  2. 2. 查看 Request Header、Body 與 Parameter。
  3. 3. 核對 Authentication、Token 與權限。
  4. 4. 讀 Status Code、Response 與 Request ID。
  5. 5. 依 Rate Limit、Timeout 或 Retry 規則處理。

每一步先記錄原始值再做下一步。若修改 Header／Body 或 Parameter 的設定，保存修改前後值與時間。

重點整理

讀完後應該能說清楚這四件事

  1. Header／Body：Header 放控制與驗證資訊；Body 放請求或回應的主要內容。
  2. Parameter：用來指定路徑、查詢或控制行為的輸入值。
  3. 把 Token 放錯位置，服務端會把有效憑證視為缺少驗證。
  4. 排查時先確認 Endpoint、HTTP 方法與版本，再查看 Request Header、Body 與 Parameter。
