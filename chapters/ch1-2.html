Client、Server 與 Browser
誰在發問，誰在回答

預估 30 分鐘 ・ 建議先讀完 1-1 再來，我們會用到「三層」的觀念。

上一章我們把系統直的切成三層。這一章換個方向，橫著切：每一次系統互動，都有一邊在發問，另一邊在回答。

這個「一問一答」的模型，是之後理解 HTTP、API 的地基。讀完這一章，您應該可以：

  • 用自己的話說出 Client 和 Server 各是什麼角色。
  • 說明 Browser（瀏覽器）為什麼是最常見的 Client。
  • 遇到「打不開」「很慢」的回報時，知道第一個該問什麼。
情境

「只有我打不開，還是大家都打不開？」

某天早上，客戶窗口在群組裡喊：

「行銷平台打不開了！可以幫忙看看嗎？急！」

您回了一句話：

「請問是只有您打不開，還是其他同事也打不開？」

兩分鐘後對方回覆：「咦，其他人都正常，只有我不行。」

就這麼一問一答，問題的範圍從「整個平台掛了」縮小到「這位同事自己的電腦或網路」。原本可能要驚動原廠的大事，變成請對方換個瀏覽器試試的小事。

為什麼這一問這麼有效？因為它在區分：問題出在發問的那一端，還是回答的那一端。這一章就是要把這個直覺，變成您隨時可以拿出來用的工具。

情境例子 1/2

打電話訂位的兩個角色

想像您打電話跟餐廳訂位：

您是打電話的人。您主動撥號、開口提出需求：「我想訂今晚七點、四位。」

餐廳櫃檯是接電話的人。櫃檯不會主動打給您，它只是守在電話旁邊，有電話進來就接、就處理、就回覆：「好的，幫您訂好了。」

系統世界裡的分工一模一樣：

角色做的事
Client（客戶端）主動發出請求的一方。「請給我這個頁面」「請幫我存這筆資料」。
Server（伺服器）守在那裡等請求的一方。收到請求、處理、回覆結果。

發出去的那句話，術語叫 Request（請求）；回來的那句話，叫 Response（回應）。一次互動，就是一個 Request 加一個 Response。先記住這兩個詞，單元 2 會天天見到它們。

先記住這句 沒有請求，就沒有回應。Server 不會主動把畫面塞給您，永遠是 Client 先開口。（企業整合裡有一種例外叫 Webhook，單元 7 會教，現在先不管它。）
情境例子 2/2

那 Browser 是什麼？

Browser（瀏覽器）就是您電腦裡最常用的那個 Client。Chrome、Edge、Safari，都是。

您在網址列輸入公司官網、按下 Enter 的那一刻，Browser 做了三件事：

  1. 替您發出 Request：「請給我這個網頁。」
  2. 接住 Server 回傳的 Response（網頁的內容）。
  3. 把內容「畫」成您看到的畫面。

這裡剛好接上第 1 章：前端程式就是在 Browser 裡執行的。Browser 是舞台，前端是在台上演出的內容。

手機上的 Mobile App 呢？它也是 Client，只是不透過瀏覽器，自己直接向 Server 發 Request。您滑購物 App 看到的商品列表，就是 App 向 Server 要來的。

所以「用網頁」和「用 App」，在 Server 眼中差別沒有想像中大：都是某個 Client 發來的 Request，處理，回覆。變的是 Client 的形式，不變的是一問一答。

正式定義

三個名詞的正式說法

名詞定義
Client
客戶端		在一次互動中，主動發出請求並等待回應的一方。例如瀏覽器、Mobile App，或是去呼叫別的系統的任何程式。
Server
伺服器		持續待命、接收請求、處理後回傳結果的一方。通常是遠端的電腦和跑在上面的服務。
Browser
瀏覽器		裝在使用者裝置上的 Client 軟體，負責發出網頁請求、接收內容、把內容呈現成畫面。

有一件事想特別放大講：Client 和 Server 是「角色」，不是「身分」。

就像一個人在公司是主管、回家是爸爸，同一個系統也會換角色。CRM 面對使用者的瀏覽器時，它是 Server；但當它去呼叫 ERP 要庫存資料時，它就變成了 Client。判斷方法只有一條：這次互動是誰先開口的，誰就是 Client。

串起來看

一問一答，會經過三段路

Client 發出的 Request，不是瞬間傳到 Server 的。它要走過三段路：

Client（您的瀏覽器）
   ↓ 第一段：Client 端
網路（公司網路、電信線路、VPN…）
   ↓ 第二段：中間的路
Server（系統的服務端）
   ↓ 第三段：Server 端
處理完，原路回傳 Response

為什麼要知道這個？因為「打不開」「很慢」這類問題，壞的可能是三段中的任何一段：

  • Client 端出問題：這台電腦的瀏覽器怪怪的、暫存了舊東西。
  • 路上出問題：公司網路塞車、VPN 沒連上。
  • Server 端出問題:系統真的掛了，這時所有人都會受影響。

注意最後一句：Server 掛了，影響的是所有人。反過來說，如果只有一個人有問題，Server 多半是無辜的。這就是開頭那一問的原理。

為什麼要學這個

兩個馬上用得到的場景

場景一：快速縮小問題範圍

「只有一個人有問題」通常指向 Client 端：那個人的瀏覽器、電腦或網路。「所有人都有問題」通常指向 Server 端或中間的網路。

一句「其他人正常嗎？」，就決定了接下來往哪邊查、找誰處理。查 Client 端找客戶自己的 IT，查 Server 端找系統管理員或原廠。方向對了，速度就快了。

場景二：聽懂需求的真正規模

客戶說：「我們希望手機也能用。」這句話有兩種完全不同的做法：

  • 把網頁做成手機上也好看好用（同一個 Client：Browser）。
  • 開發一個獨立的 Mobile App（多養一種全新的 Client）。

第二種的成本高出許多：要開發、要上架、要持續維護。聽懂差別的顧問，才能在客戶輕輕一句「手機也能用」時，幫雙方把期待對齊。

核心概念 1/2

一問一答，角色會換

概念一：一次互動 = Request + Response

您在系統裡做的每個動作，開頁面、按儲存、查報表，背後都是一次或多次的一問一答。這個模型簡單到不像重點，但單元 2 的 HTTP、單元 6 的 API，全部蓋在它上面。地基打好，後面都輕鬆。

概念二：誰先開口，誰就是 Client

再看一次這個例子，這次用 Northstar Electronics 的系統：

互動一：業務的瀏覽器 → 向 CRM 要客戶清單
        瀏覽器是 Client，CRM 是 Server

互動二：CRM → 向 ERP 要這位客戶的訂單金額
        CRM 是 Client，ERP 是 Server

同一個 CRM，在兩次互動裡扮演不同角色。之後看整合架構圖時，別問「這個系統是 Client 還是 Server」，要問「在這條線上，是誰先開口」。

核心概念 2/2

Browser 很方便，但它會「記舊帳」

概念三：Browser 是最普及的 Client

不用安裝、輸入網址就能用，這是 Browser 最大的優點。企業系統大多做成網頁形式，原因就在這裡。

但 Browser 有個習性要認識：為了讓網頁開得快，它會把看過的內容暫存起來，下次直接拿舊的出來用。這個暫存叫 Cache（快取）。

Cache 帶來一種經典的假象：系統明明更新了，某位使用者的畫面卻還是舊的。這不是 Server 的錯，是那台電腦的 Browser 在拿舊庫存。所以「請重新整理頁面試試」不是敷衍，是有道理的第一步。單元 2 會教更完整的處理方式。

概念四：排查問題，先想三段路

Client 端、中間的網路、Server 端。遇到「連不上」先別急著深入任何一段，用影響範圍判斷方向：一個人有事查 Client，全部人有事查 Server 和網路。這個習慣和第 1 章的分層思考是同一家人，只是切的方向不同。

容易混淆

三組容易搞混的概念

容易混淆的一組怎麼分
Client vs 前端很接近，但角度不同。「前端」講的是系統的分層（負責畫面的那層）；「Client」講的是互動中的角色（先開口的那方）。前端程式跑在 Client 裝置上，但一個完全沒有畫面的系統去呼叫別人時,它也是 Client。
Server vs 後端「Server」是提供服務的那台電腦和角色；「後端」是跑在 Server 上的程式。日常對話混著用通常沒事，寫正式的問題描述時分清楚會更專業。
Browser vs Mobile App兩個都是 Client。Browser 是通用軟體，開網址就能用；App 要安裝，直接和 Server 溝通，還能用到手機的推播、相機等功能。

第一組最容易卡住。卡住的話記這句就好：前端是「層」，Client 是「角色」。

案例

Northstar Electronics 的一次登入

Northstar 的業務同仁早上打開瀏覽器，登入 CRM 查客戶資料。整個過程拆開來看：

1. 瀏覽器發出 Request：「請給我登入頁」
2. CRM 回傳登入頁（Response）
3. 業務輸入帳密、按登入（又一個 Request）
4. CRM 驗證帳密，回傳客戶清單頁
5. 瀏覽器把內容畫成畫面

短短一次登入，就有兩組一問一答。瀏覽器全程是 Client，CRM 全程是 Server。

如果這位業務今天登入失敗，可能出錯的位置有：

  • Client 端：這台電腦的瀏覽器有問題，或存了過期的登入狀態。
  • 路上：公司網路或 VPN 斷了，Request 根本沒送到。
  • Server 端：CRM 服務異常，這時其他同事也會遇到。
第一個好問題 還是那句：「其他人正常嗎？」只有一人異常，先查 Client 端；全部異常，先查 Server 端與網路。然後請對方截圖錯誤畫面、記下時間，這些是之後請工程師查紀錄的重要線索。

不用 Salesforce 的話，這張卡片可以直接翻過去。

通用概念Salesforce 裡的樣子
Client使用者的瀏覽器、Salesforce Mobile App、來呼叫平台 API 的外部系統

外部系統透過 API 呼叫 Salesforce 時，那個外部系統就是 Client。反過來，Salesforce 的 Outbound Message 或 Callout 去呼叫別人時，Salesforce 就是 Client。角色看方向，不看名字。

常見誤解

三個很多人踩過的坑

誤解一：「系統打不開，就是系統掛了」

不一定。可能只是那位使用者的瀏覽器、電腦或網路有狀況。先確認影響範圍，別急著通報「系統故障」。喊過幾次狼來了，之後真的出事就沒人信了。

誤解二：「Client 就是客戶的電腦」

Client 是「先開口的角色」，跟機器是誰的、大台小台都無關。一台強大的伺服器去呼叫別的系統時，它就是那次互動的 Client。

誤解三：「畫面顯示的一定是最新資料」

Browser 可能給您看 Cache 裡的舊內容。看到「資料怎麼沒更新」時，先重新整理頁面，再判斷是不是真的有問題。

遇到問題先查什麼

「打不開、很慢」的檢查順序

  1. 確認影響範圍。只有一個人，還是很多人？這一步決定後面所有方向。
  2. 確認環境與時間。正式還是測試環境？從什麼時候開始的？昨天正常嗎？
  3. 換一個 Client 試試。換瀏覽器、換裝置，或改用手機網路。換了就正常，代表問題在原本的 Client 端或它的網路。
  4. 確認 Server 端狀態。請客戶 IT 或原廠確認服務是否正常、有沒有維護公告。
  5. 保留證據。請使用者截圖錯誤畫面、記下發生時間。之後工程師查紀錄，全靠這些線索對時間。

前三步都不需要技術工具，是顧問自己就能完成的。很多問題走到第三步就水落石出了。

重點整理

帶走這四句話

  1. Client 主動發出 Request，Server 等著接、處理、回 Response。一問一答是所有網路互動的基本形。
  2. Client 和 Server 是角色不是身分。誰先開口，誰就是這次互動的 Client。
  3. Browser 和 Mobile App 都是 Client，只是形式不同。Browser 會暫存舊內容，畫面舊不一定是資料舊。
  4. 遇到「連不上、很慢」，第一步永遠是確認影響範圍：一個人有事查 Client，大家都有事查 Server 和網路。

接下來是四題檢核，慢慢作答就好。

檢核 1/4

在一次系統互動中，判斷「誰是 Client」的依據是什麼？

正確答案是 B。Client 與 Server 是互動中的角色：先開口發出 Request 的一方就是 Client，跟機器大小、有沒有畫面都無關。
檢核 2/4

是非題：CRM 系統永遠是 Server，不可能是 Client。

正確答案是「否」。CRM 面對使用者的瀏覽器時是 Server，但它去呼叫 ERP 的 API 時，就是那次互動的 Client。角色依互動方向而定。
檢核 3/4

下列哪一個「不是」Client？

正確答案是 C。守在那裡等請求、處理後回覆的，是 Server 的角色。其他三個都是主動發出請求的 Client，包括沒有畫面的排程程式。
檢核 4/4

客戶回報「行銷平台打不開」。下列哪一個是資訊量最大的第一個問題？

正確答案是 B。影響範圍能立刻區分 Client 端問題（一個人）和 Server 端或網路問題（很多人），是成本最低、收斂最快的一問。

留一個小小的思考題，散步時想想就好：

您手上的專案裡，找一條「系統 A 呼叫系統 B」的整合。在這條線上，誰是 Client、誰是 Server？如果它今天失敗了，您會先請哪一邊查什麼？
