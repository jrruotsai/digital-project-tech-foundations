JavaScript 與網頁事件
網頁為什麼會「動」

預估 30 分鐘 ・ 建議先讀完 3-1、3-2 再來。

HTML 負責結構、CSS 負責外觀,那網頁的「互動」是誰負責的?點一個按鈕、畫面跳出提示、欄位即時檢查格式,這些「會動」的行為,都是這一章要教的 JavaScript 在幕後運作。

  • 說明 JavaScript 在網頁裡扮演的角色。
  • 認得 DOM 和 Event 這兩個核心概念。
  • 看到「按了按鈕沒反應」的問題時,知道可能要往哪個方向排查。
情境

「按了按鈕,畫面完全沒反應」

Northstar Electronics 的報名表單,客戶反映按下「送出」按鈕後畫面完全沒動靜,顧問請教工程師,對方查看瀏覽器的 Console(2-8 提過的另一個開發者工具頁籤)後回覆:

「原始碼裡有個 JavaScript 錯誤,導致綁定在按鈕上的事件沒有正確執行,難怪按了沒反應。」

「事件沒有正確執行」是什麼意思?這一章要把 JavaScript 怎麼讓網頁「動起來」的原理講清楚,幫助您理解這類問題背後在發生什麼事。

情境例子 1/2

舞台佈景,和演員的即時反應

如果說 HTML 是舞台上的佈景(有什麼道具、擺在哪裡),CSS 是佈景的美術設計(顏色、質感),那麼 JavaScript 就是舞台上的演員——負責對觀眾的反應做出即時回應。觀眾拍手(使用者點擊),演員立刻鞠躬(畫面產生變化);觀眾喊安可(使用者操作某個功能),演員立刻加演一段(觸發某個功能)。

沒有演員的舞台,佈景再漂亮也是靜態的,不會對觀眾的行為有任何反應。這正是 JavaScript 存在的意義:讓網頁具備「回應使用者操作」的能力。

先記住這句 HTML 是結構、CSS 是外觀、JavaScript 是行為。三者分工合作,才構成一個完整、會動的網頁。
情境例子 2/2

觀眾的動作,和演員的劇本

要讓演員知道「觀眾做了什麼動作」,舞台需要有一份「觀眾席地圖」,標明每個座位在哪裡——這就是 DOM(Document Object Model,文件物件模型),它是瀏覽器把 HTML 結構,轉換成程式可以讀取、操作的一份地圖,讓 JavaScript 知道畫面上有哪些元素、它們在哪裡。

而「觀眾做了什麼動作」這件事本身,叫做 Event(事件)——像是 Click(點擊)、Submit(送出)、Load(載入完成),都是常見的事件類型。JavaScript 的工作,就是「監聽」某個元素上會不會發生特定事件,一旦發生,就執行對應的劇本(程式碼)。

回到開頭按鈕沒反應的問題。工程師說「JavaScript 錯誤」導致「事件沒有正確執行」,意思是:原本應該綁定在按鈕上「監聽 Click 事件、送出表單」的這段劇本,因為程式碼裡有錯誤,整段劇本沒有被正確設置,所以演員(JavaScript)壓根不知道觀眾(使用者)做了點擊這個動作。

串起來看

按下按鈕之後,發生了什麼事

接續 3-3 教過的報名表單,加上 JavaScript 讓按鈕產生反應:

<button id="submitBtn" type="button">送出</button>

<script>
  // 監聽 submitBtn 這個元素的 Click 事件
  document.getElementById("submitBtn").addEventListener("click", function() {
    console.log("按鈕被點擊了");   // Console Log,2-8 提過的偵錯工具
    // 這裡通常還會有更多程式碼,例如檢查欄位、送出表單
  });
</script>

逐行解釋:getElementById 這段程式,是透過 DOM 找到那個 id 為 submitBtn 的按鈕(呼應 3-3 教過的 id 屬性);addEventListener("click", ...) 是在說「監聽這個按鈕的 Click 事件」;一旦真的被點擊,大括號裡的程式碼就會被執行。如果這段程式碼本身有錯誤(例如拼字錯誤、邏輯錯誤),整段監聽可能無法正確設置,導致按鈕看起來「沒反應」。

為什麼要學這個

兩個馬上用得到的場景

場景一:理解「畫面沒反應」不一定是後端的問題

1-1 教過三層架構,現在您知道前端裡還可以再細分:HTML 結構、CSS 外觀、JavaScript 行為。按鈕沒反應,可能連 Request 都還沒送出去(呼應 2-8 教過的判斷方式),問題卡在最前端的 JavaScript,根本還沒輪到後端出場。

場景二:知道 Console 是排查 JavaScript 問題的重要工具

開發者工具裡的 Console 頁籤,會顯示 JavaScript 執行時的錯誤訊息。下次遇到「畫面沒反應」的問題,可以請工程師先看一眼 Console 有沒有紅色的錯誤訊息,這往往是最快找到線索的地方。

核心概念 1/2

JavaScript 讓網頁從「靜態」變「互動」

概念一:沒有 JavaScript,網頁只能呈現固定內容,無法回應操作

單純的 HTML 加 CSS,可以構成一個好看的頁面,但按鈕按下去不會有任何反應、欄位不會即時檢查格式。JavaScript 補上了「回應使用者行為」這個關鍵能力。

概念二:DOM 是 JavaScript 操作畫面的地圖

JavaScript 要找到某個元素、改變它的內容或樣式,都要透過 DOM 這份地圖來定位。理解這一點,能幫助您理解為什麼元素的 id、class(3-3、3-4 教過)對 JavaScript 來說也很重要——這些名稱正是用來在地圖上找到正確位置的依據。

核心概念 2/2

Event 是連結「使用者行為」和「程式反應」的橋樑

概念三:每種操作行為,對應著特定的 Event 類型

點擊對應 Click、送出表單對應 Submit、頁面載入完成對應 Load,不同的操作行為,JavaScript 需要監聽不同的 Event 類型,才能正確做出反應。

概念四:JavaScript 出錯,通常會讓「後續的劇本」整段失效

如果程式碼裡某一段有錯誤,可能導致原本該執行的後續動作(例如綁定按鈕事件)完全沒有被設置。這也是為什麼一個小小的程式錯誤,有時會讓整個功能「看起來完全沒反應」,而不只是部分失常。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
HTML/CSS vs JavaScriptHTML 和 CSS 決定畫面「長什麼樣子」(結構與外觀),是相對靜態的描述;JavaScript 決定畫面「怎麼回應操作」,是動態的行為邏輯。三者分工不同,但常被籠統地混稱為「前端」。
Console(2-8 提過)vs Network(2-8 教過)Console 顯示程式執行時的訊息或錯誤(通常跟 JavaScript 有關);Network 顯示 Request 和 Response 的往來紀錄(跟後端溝通有關)。排查「沒反應」問題時,兩個頁籤都值得看一眼,但關注的層面不同。
案例

Northstar Electronics 的按鈕失效事件

追查報名表單送出按鈕沒反應的問題,工程師打開開發者工具的 Console,發現:

Console 顯示的錯誤訊息:
Uncaught TypeError: Cannot read property 'value' of null

追查原因:
JavaScript 程式碼裡,嘗試透過某個 id 抓取一個欄位,
但那個 id 在 HTML 裡打錯字了(呼應 3-1、3-3 教過的 id 屬性)
  1. 有哪些角色?HTML 裡的表單欄位、JavaScript 裡嘗試操作這個欄位的程式碼。
  2. 資料如何移動?這裡問題發生在使用者按下按鈕的那一刻,JavaScript 嘗試透過 DOM 找到某個欄位,卻因為 id 對不上而失敗。
  3. 各部分負責什麼?JavaScript 依賴 HTML 裡設定的 id 屬性,來準確定位它要操作的元素,如果名稱對不上,就會找不到目標而出錯。
  4. 哪裡可能出錯?HTML 裡的 id 拼字錯誤,或是 JavaScript 程式碼裡引用的 id 名稱寫錯,兩邊只要有一邊出錯,整個監聽和後續動作都會失敗。
  5. 第一個合理的檢查方向?打開開發者工具的 Console 頁籤,查看有沒有紅色的錯誤訊息,這通常能直接指出問題發生在哪一行程式碼、涉及哪個元素。

常見誤解

三個很多人踩過的坑

誤解一:「按鈕沒反應,一定是後端沒收到請求」

問題也可能發生在更早的階段——JavaScript 根本沒有正確執行,連 Request 都還沒被送出去,這時候後端完全沒有機會出場。

誤解二:「JavaScript 只是讓網頁有一些花俏的動畫效果」

JavaScript 負責的範圍遠不只是視覺特效,包括表單驗證(3-7 會教)、追蹤使用者行為(3-8 會教)、呼叫其他系統的 API(單元 6 會教),都仰賴 JavaScript 運作。

誤解三:「一個小小的程式錯誤,只會影響那一小段功能」

有時候一個錯誤會導致整段程式碼中斷執行,連帶讓後面原本該設置的其他功能(例如其他按鈕的事件監聽)也一併失效,影響範圍可能比想像中大。

遇到問題先查什麼

「按鈕或功能沒反應」的檢查順序

  1. 打開開發者工具的 Console 頁籤。看有沒有紅色的錯誤訊息。
  2. 打開 Network 頁籤,確認有沒有發出對應的 Request。如果完全沒有,代表問題卡在更早的 JavaScript 執行階段(呼應 2-8 教過的判斷方式)。
  3. 確認相關的 HTML 元素 id 或 class,是否跟 JavaScript 程式碼裡引用的名稱一致。
  4. 確認這個功能是最近才出問題,還是一直都有問題。幫助判斷是不是最近的改動造成的。
  5. 把 Console 裡的錯誤訊息完整提供給工程師。這通常是最直接、最有價值的線索。
重點整理

帶走這四句話

  1. JavaScript 讓網頁具備回應使用者操作的能力,是前端裡負責「行為」的部分。
  2. DOM 是 JavaScript 用來定位和操作畫面元素的地圖,Event 是連結使用者行為與程式反應的橋樑。
  3. 「按鈕沒反應」不一定是後端問題,可能是 JavaScript 執行時發生錯誤,連請求都還沒送出。
  4. Console 頁籤是排查 JavaScript 問題最重要的工具,紅色錯誤訊息通常能直接指出問題所在。

在網頁的分工中,負責讓畫面回應使用者操作(例如點擊按鈕產生反應)的是誰?

正確答案是 C。JavaScript 負責讓網頁具備互動與動態行為能力,回應使用者的操作。
檢核 2/4

是非題:按鈕按下去沒有任何反應,一定代表後端沒有收到請求。

正確答案是「否」。問題可能發生在更早的 JavaScript 執行階段,導致連 Request 都還沒被送出去,後端完全沒有機會參與。

開發者工具裡,哪個頁籤最適合用來查看 JavaScript 執行時的錯誤訊息?

正確答案是 A。Console 頁籤會顯示 JavaScript 執行時的訊息或錯誤,是排查這類問題最重要的工具。
檢核 4/4

情境題:客戶反映按下送出按鈕後畫面完全沒反應。第一個合理的檢查方向是?

正確答案是 A。先查看 Console 是否有 JavaScript 錯誤訊息,是排查「沒反應」問題成本最低、也最直接的第一步。
