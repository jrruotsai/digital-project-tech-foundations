Root Cause 與 Workaround — 治本與治標

預估 30 分鐘 ・ 系統化除錯與 Root Cause Analysis

Root Cause 是造成問題的根本原因，Workaround 是暫時避開問題的方法。

  • 能說明 Root Cause：造成問題的根本原因。
  • 能區分 Root Cause 與 Workaround 在流程中的用途。
  • 遇到問題時，先寫 Expected Result 與 Actual Result。
判斷桌

先恢復服務，還是直接修根因？

COMPARE / 先看條件再選

Workaround

目標

判斷口訣：Workaround 爭取時間；Root Cause 才能避免再次發生。
案例起點

Root Cause 和 Workaround 沒有寫清楚時

Northstar 的規格同時提到 Root Cause 與 Workaround，卻沒有標出兩者的適用條件與處理邊界。

Root Cause：造成問題的根本原因。；Workaround：暫時避開問題、維持工作的方法。

測試時要避免這個結果：有替代流程不代表問題已修好，也不能把暫時措施當永久方案。

概念拆解 1/2

Root Cause 的處理邊界

Root Cause：造成問題的根本原因。

套用到本章情境，第一步是寫 Expected Result 與 Actual Result。確認 Root Cause 的適用範圍後，再檢查 Workaround，不從畫面直接猜原因。

判斷重點說明 Root Cause 時，要交代誰提供輸入、系統套用哪條規則，以及會產生什麼結果。
概念拆解 2/2

Workaround 與 Root Cause 的差別

Workaround：暫時避開問題、維持工作的方法。

Root Cause 是造成問題的根本原因，Workaround 是暫時避開問題的方法。

不要混用Root Cause 與 Workaround 不能互換；規格要分別寫出兩者的輸入、處理方式與輸出。
精確定義

本章名詞的工作定義

術語定義判斷時要問
Root Cause造成問題的根本原因。它接收什麼輸入，完成後留下什麼結果？
Workaround暫時避開問題、維持工作的方法。它和 Root Cause 的邊界在哪裡？
Evidence能證明 Root Cause 或 Workaround 實際結果的原始資料，例如 Request／Response、Log、時間、版本與識別值。哪一筆原始紀錄能支持目前的判斷？

定義 Root Cause 與 Workaround 時，必須寫出用途、處理邊界和可觀察結果。

流程位置

Root Cause 在整體流程中的位置

Issue → Reproduce → Evidence → Layered Checks → Root Cause → Fix／Regression Test
→ 觀察結果與例外
→ 記錄負責人與處理決定

沿著流程逐段比對輸入與輸出。若 Root Cause 的輸入正確而 Workaround 的輸出不符預期，就從兩者之間的邊界開始查。

專案應用

這個概念會出現在哪些工作？

規格與設計

問題處理顧問需要把「Root Cause 與 Workaround — 治本與治標」改寫成可確認的規則，放進Issue Description 與 Log。

問題排查

先寫 Expected Result 與 Actual Result，再記錄 Reproduce 步驟、Scope 與 Frequency。這能分辨問題是否位於 Root Cause 或 Workaround。

核心概念 1/2

輸入、規則與輸出要分開寫

輸入

列出 Root Cause 開始前已知的資料、身分、事件或設定。輸入不明，後面的結果就無法比較。

規則與輸出

Workaround 執行後，保存實際套用的規則版本、Response、狀態或資料值。不要用「看起來正常」代替可查證結果。

核心概念 2/2

成功條件和失敗條件都要具體

成功條件

Root Cause 是造成問題的根本原因，Workaround 是暫時避開問題的方法。 驗收 Root Cause 與 Workaround 時，寫明預期狀態、資料值與完成時間。

失敗條件

有替代流程不代表問題已修好，也不能把暫時措施當永久方案。 Root Cause 或 Workaround 失敗時，保留原始輸入、環境、時間與識別值。

容易混淆

用處理邊界分清楚三組概念

容易混淆實際差別
Root Cause vs WorkaroundRoot Cause：造成問題的根本原因。 Workaround：暫時避開問題、維持工作的方法。
規格存在 vs 實際生效Root Cause 的文件是預期規則；Workaround 的設定值、Response 或 Log 才能證明實際結果。
單次成功 vs 穩定可重現Root Cause 單次成功不代表 Workaround 在其他條件下也正確；要用相同輸入重做。

規格落點

名詞應該寫進哪些文件？

內容文件中的寫法
Root Cause造成問題的根本原因。 補上負責系統、輸入與成功結果。

常見誤解

三個會讓排查走錯方向的假設

把 Root Cause 和 Workaround 當成同一件事

有替代流程不代表問題已修好，也不能把暫時措施當永久方案。

只看最後畫面

畫面是流程的最後一層。仍要核對 Root Cause 的輸入與 Workaround 的執行紀錄。

一次改多個條件

同時修改 Root Cause 與 Workaround 的條件，即使恢復也無法歸因。每次只改一項。

排查順序

Root Cause 發生異常時的排查順序

  1. 1. 寫 Expected Result 與 Actual Result。
  2. 2. 記錄 Reproduce 步驟、Scope 與 Frequency。
  3. 3. 用 Timestamp、Request ID 串起 Log。
  4. 4. 沿 User、Browser、API、Application 到 Database 分層查。
  5. 5. 分開 Root Cause、Workaround、Permanent Fix 與 Regression Test。

每一步先記錄原始值再做下一步。若修改 Root Cause 或 Workaround 的設定，保存修改前後值與時間。

重點整理

讀完後應該能說清楚這四件事

  1. Root Cause：造成問題的根本原因。
  2. Workaround：暫時避開問題、維持工作的方法。
  3. 有替代流程不代表問題已修好，也不能把暫時措施當永久方案。
  4. 排查時先寫 Expected Result 與 Actual Result，再記錄 Reproduce 步驟、Scope 與 Frequency。
