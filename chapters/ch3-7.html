表單驗證
擋在送出之前的檢查

預估 30 分鐘 ・ 建議先讀完 3-3、3-6、1-1 再來。

1-1 教過「重要規則的把關在後端,前端只是提醒」。這一章要把這個觀念,在表單驗證這個具體場景裡講得更透徹,並接回 3-6 教過的 JavaScript 事件。

  • 分辨前端驗證與後端驗證的角色差異。
  • 理解 Required 屬性的作用與限制。
  • 看到「明明有檢查,資料還是有問題」的情況時,知道可能發生了什麼事。
情境

「Email 欄位有做檢查,怎麼資料庫還是有格式錯誤的資料?」

Northstar Electronics 的顧問查看報名資料時,發現有幾筆 Email 欄位格式明顯不對(缺少 @ 符號),但表單明明有做即時檢查,詢問工程師:

「奇怪,表單有檢查 Email 格式啊,怎麼還會有錯的資料進到資料庫?」工程師解釋:「前端的檢查可以被繞過,如果後端沒有再做一次驗證,漏網的資料就會直接存進去。」

這正好呼應 1-1 教過的核心概念。這一章要把「前端驗證」這件事的原理和限制講清楚,讓您理解為什麼「有檢查」不等於「絕對安全」。

情境例子 1/2

櫃檯的初步提醒,和最終的正式審核

想像您去銀行辦事,櫃檯人員在您填單子時,瞄一眼說:「這欄好像忘了填喔。」這是一個貼心的提醒,讓您當場補上,省去之後被退件的麻煩。但真正決定這張單子能不能生效的,是後續銀行內部的正式審核流程,不是櫃檯那一眼的提醒。

前端驗證(Client-side Validation)就像櫃檯的提醒——在使用者填寫的當下,即時檢查格式對不對、有沒有漏填,讓使用者能立刻發現並修正,體驗更順暢。

後端驗證(Server-side Validation)則像銀行內部的正式審核——不管前端有沒有檢查過,資料送到後端時,還是要重新檢查一次,才是真正決定資料能不能被接受的最後防線。

先記住這句 前端驗證是為了「體驗」,讓使用者少走冤枉路;後端驗證是為了「安全」,是真正把關的那一道關卡。兩者都需要,缺一不可。
情境例子 2/2

櫃檯的提醒,可以被跳過

如果有人不透過正常的窗口填單,而是直接把文件塞進銀行內部的收發系統,那麼「櫃檯的提醒」這一關,自然就被繞過去了。表單也是一樣的道理:如果有人不透過瀏覽器上的表單畫面填寫,而是直接用工具對後端發送資料(呼應 2-2 教過的 Request 概念),那麼前端寫的所有檢查邏輯,根本沒有機會被執行。

回到開頭 Email 格式錯誤的案例。前端的即時檢查,只在「使用者透過畫面正常填寫、正常點擊送出」的情境下才會生效。如果後端沒有在收到資料時,再次確認格式是否正確,那些繞過前端檢查(不管是刻意還是意外)送來的資料,就會直接被存進資料庫,不會被攔下來。

正式定義

幾個關鍵名詞的正式說法

名詞定義
前端驗證
Client-side Validation		在使用者的裝置上,由 JavaScript 即時檢查輸入內容是否符合格式或規則
後端驗證
Server-side Validation		資料送達後端伺服器後,再次進行的格式與規則檢查,是真正的把關防線
Required
必填		HTML 屬性,標記某個欄位為必填,若未填寫,前端會即時提示,但同樣可以被繞過
串起來看

一次表單送出,兩層檢查各自的時機

把一次表單送出的過程拆開來看,前後端驗證分別在哪個環節發生:

1. 使用者填寫 Email 欄位,輸入「abc」(格式明顯不對)
2. 使用者按下送出,前端的 JavaScript(3-6 教過的事件監聽)立刻檢查格式
   → 發現不符合 Email 格式,擋下送出動作,畫面顯示提示訊息
3. 假設使用者修正成正確格式,或前端檢查被繞過,Request 被送到後端(2-2 教過)
4. 後端再次檢查這筆資料的 Email 格式是否正確
   → 符合,才寫入資料庫;不符合,回傳錯誤(呼應 2-4 教過的 Status Code,例如 400)

第 2 步和第 4 步,都在做「同一件事」(檢查 Email 格式),但發生的位置、能不能被繞過,完全不同。健全的系統,這兩步都會做,不會只依賴其中一步。

為什麼要學這個

兩個馬上用得到的場景

場景一:訪談需求時,主動確認「後端也有驗證嗎?」

當客戶提出「這個欄位一定要檢查格式」的需求時,可以主動詢問工程師:「前端和後端都會做這個檢查嗎?」確保重要的規則不會因為只依賴前端,而留下被繞過的風險。

場景二:排查「資料庫裡出現不該存在的資料」時,懂得往哪裡查

發現資料庫裡有格式明顯錯誤的資料,懂得表單驗證原理的您,會直接懷疑「後端是不是沒有做驗證」,而不是誤以為前端的檢查邏輯「壞了」——因為前端檢查原本就無法保證百分之百攔住所有資料。

核心概念 1/2

前端驗證讓體驗更好,但不是安全的保證

概念一:前端驗證的即時性,是它最大的價值

使用者填錯格式,不用等送出、等後端回應,當場就能看到提示並修正,大幅提升填寫體驗。這是前端驗證存在的主要理由。

概念二:前端驗證的邏輯,理論上可以被任何懂技術的人繞過

不管是刻意繞過,還是某些特殊情境下前端程式碼沒有正常執行(例如 3-6 教過的 JavaScript 錯誤),都可能導致前端驗證沒有真正發揮作用。這也是為什麼它不能作為唯一的防線。

核心概念 2/2

後端驗證才是真正的最後防線

概念三:重要的規則,一定要在後端也檢查一次

不管前端做了多完善的檢查,涉及資料正確性、安全性的規則,後端都必須獨立再檢查一次,不能假設「前端檢查過了,後端就不用再檢查」。

概念四:Required 屬性只是前端的提醒,不是保證

HTML 的 required 屬性能讓瀏覽器在使用者忘記填寫時跳出提示,但這仍然屬於前端驗證的範疇,一樣可能被繞過,重要欄位是否真的有值,後端仍需要再次確認。

容易混淆

一組最容易搞混的概念

容易混淆的一組怎麼分
前端驗證 vs 後端驗證前端驗證重視「即時體驗」,在使用者填寫當下就能提示;後端驗證重視「安全把關」,是資料真正被接受前的最後一道防線。兩者目的不同,健全的系統應該同時具備。

這組概念其實就是 1-1「前端負責呈現、後端負責把關」這條大原則,套用在「表單驗證」這個具體情境上的延伸應用。

案例

Northstar Electronics 的 Email 格式異常事件

追查資料庫裡出現格式錯誤 Email 的問題,發現:

前端:有設定即時檢查 Email 格式的 JavaScript 邏輯
後端:當初開發時,只在前端做了驗證,後端接收資料時沒有重複檢查
結果:少數透過非正常管道(或前端邏輯異常時)送出的資料,直接被存進資料庫
  1. 有哪些角色?前端的表單驗證邏輯、後端的資料接收邏輯、資料庫。
  2. 資料如何移動?不符合格式的資料,繞過了前端檢查,直接被後端接受並寫入資料庫。
  3. 各部分負責什麼?後端本應該是資料正確性的最後把關者,但這個案例中後端沒有承擔起這個責任。
  4. 哪裡可能出錯?開發時可能誤以為「前端已經檢查過了,後端不用再查一次」,這正是本章要破解的常見誤解。
  5. 第一個合理的檢查方向?確認後端的資料接收邏輯,是否有獨立的格式驗證機制,而不是只依賴前端的檢查。

通用概念常見情境
後端驗證Salesforce 的 Validation Rule(驗證規則),就是在平台後端(1-1 教過的對照表)強制執行的資料檢查機制,不受前端表單設計影響
常見誤解

三個很多人踩過的坑

誤解一:「前端有做檢查,資料就一定沒問題」

前端驗證可以被繞過,重要的規則一定要後端也獨立檢查一次,才能真正保證資料正確性。

誤解二:「表單標了 required,使用者就一定會填」

required 只是前端的提醒機制,一樣可能因為各種原因被繞過,不能當作絕對的保證。

誤解三:「後端驗證只是重複做一次前端已經做過的事,浪費資源」

後端驗證的目的是安全把關,不是單純的重複勞動。兩者服務的目的不同(體驗 vs 安全),都有存在的必要。

遇到問題先查什麼

「資料庫出現不合理資料」的檢查順序

  1. 確認前端是否有做相關的驗證邏輯。如果有,先確認邏輯本身是否正確。
  2. 確認後端是否也有獨立的驗證機制。這往往是問題的關鍵所在。
  3. 確認這些異常資料是透過正常表單管道,還是其他方式送入的。
  4. 如果後端確實缺少驗證,建議補上,而不只是加強前端檢查。
  5. 把發現的具體異常資料範例,提供給工程師參考。
重點整理

帶走這四句話

  1. 前端驗證重視即時體驗,後端驗證重視安全把關,兩者目的不同,缺一不可。
  2. 前端的檢查邏輯理論上可以被繞過,不能單獨依賴它保證資料正確性。
  3. Required 屬性只是前端的提醒機制,不是絕對的保證。
  4. 發現資料庫裡有不合理的資料,經常是後端缺少獨立驗證造成的,值得優先檢查這個方向。
檢核 1/4

真正決定資料能不能被系統接受的最後防線是什麼?

正確答案是 B。後端驗證是資料真正被接受前的最後一道把關,前端驗證可能被繞過,不能作為唯一防線。
檢核 2/4

是非題:前端的表單驗證邏輯,理論上可以被繞過。

正確答案是「是」。前端驗證是在使用者的裝置上執行的,理論上可以透過各種方式繞過,這正是後端驗證不可或缺的原因。
檢核 3/4

HTML 的 required 屬性,主要提供的是什麼?

正確答案是 C。required 只是前端提醒機制的一種,能提升填寫體驗,但無法絕對保證資料一定會被正確填寫。
檢核 4/4

情境題:資料庫裡出現了格式明顯錯誤的 Email,但前端表單確實有做即時檢查。最可能的原因是什麼?

正確答案是 A。前端檢查可能因為各種原因被繞過,如果後端沒有獨立驗證,這些資料就會直接被接受並存入資料庫。
