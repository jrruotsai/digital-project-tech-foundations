Master Data 與資料責任
Data Owner 與 Data Steward

預估 30 分鐘 ・ 建議先讀完 5-1。

上一章決定「哪個系統是正本」。接著還有一個更現實的問題：誰負責讓正本維持可用？如果沒有人做決定、也沒有人處理日常品質，再正式的規則都會慢慢失效。

  • 理解 Master Data 為何會被多個流程重複使用。
  • 分清 Data Owner 與 Data Steward 的責任。
  • 遇到資料爭議時，知道該找誰決策、找誰追查。
情境

「客戶產業別，到底誰能改？」

Northstar Electronics 的業務想把一批客戶的產業別改成新的分類；行銷部門擔心改完後，既有名單會全部跑掉。資訊部門則說欄位技術上可以改，但不願替業務規則背書。

「系統權限我可以開，可是誰有權決定分類規則？改錯之後，又是誰要把三千筆資料整理回來？」

這時缺的不是一個更大的權限，而是明確的資料責任。

情境例子 1/2

連鎖餐廳共用的食材名冊

一家連鎖餐廳有幾十間分店。各店每天都會用到「食材名稱、供應商、規格、過敏原」這份名冊。若每家店各自把同一項食材取不同名字，採購、庫存和菜單標示很快就會對不起來。

Master Data（主資料）很像這份共用名冊：它不是某一張訂單，而是許多交易與流程都會反覆引用的基礎資料。

先記住這句主資料是「大家反覆共用的基本名冊」，例如客戶、產品、供應商與組織資料。
情境例子 2/2

一個人定規則，一個人顧日常

餐廳總部決定「食材分類要分到哪一層、誰可以新增供應商」，這是管理上的決策。每天檢查重複品名、補齊過敏原、追供應商確認規格，則是持續的整理工作。

前者像 Data Owner，有權決定規則並承擔結果；後者像 Data Steward，負責把規則落到日常資料上。兩個角色可能由不同人擔任，也可能由同一個小組兼任。

判斷口訣要拍板，找 Owner；要追資料、執行規則，找 Steward。
正式定義

三個名詞的正式說法

名詞定義
Master Data
主資料		跨流程、跨系統反覆使用，用來描述核心業務對象的基礎資料，如客戶、產品、供應商
Data Owner
資料擁有者		對某一資料領域有決策權並承擔責任的業務角色，負責核定規則、用途與品質標準
Data Steward
資料管理者		執行資料規則、監看品質、協調修正與處理日常例外的人員

Owner 不等於「資料在他的電腦裡」，Steward 也不只是幫忙輸入資料。這兩個詞談的是治理責任。

串起來看

一項主資料從新增到修正

以 Northstar 新增客戶產業分類為例：

業務提出新增分類需求
→ Data Steward 檢查是否已有相近分類、整理影響範圍
→ Data Owner 核定分類名稱、適用規則與生效日
→ 系統管理者依核定內容調整選項
→ Data Steward 檢查既有資料、安排清理並追蹤品質

資訊部門負責讓設定正確上線，但不替業務部門決定「產業分類應該怎麼分」。角色切開，責任才不會混在一起。

為什麼要學這個

顧問最常遇到的兩個場景

場景一：需求卡在「沒人敢決定」

欄位規則談了三次仍沒結論，先別急著排開發。請客戶確認這個資料領域的 Owner，讓有決策權的人進會議。

場景二：上線後品質一路變差

規則文件寫得很好，卻沒有人定期檢查空值與重複。這通常是 Steward 的日常責任沒有被指派，不一定是系統功能不足。

核心概念 1/2

先分資料，再分責任

概念一：主資料和交易資料用途不同

客戶、產品是可被許多流程引用的主資料；訂單、付款、客服案件是一次次發生的交易或活動紀錄。兩者都重要，但管理方式不同。

概念二：Owner 通常來自業務，不是資訊部門

「客戶等級怎麼定」是業務政策。資訊部門可以說明系統限制，卻不該代替業務主管決定分級規則。

核心概念 2/2

責任要落到可執行的動作

概念三：一個人不必包辦所有事

Owner 可以授權 Steward 處理日常例外，但重大規則、品質門檻與跨部門爭議仍要由 Owner 拍板。

概念四：角色名稱不重要，責任寫清楚才重要

有些公司不使用 Data Steward 這個職稱。只要文件明確寫出誰核定、誰檢查、誰修正、多久回報一次，治理就能運作。

容易混淆

三組看起來相近的概念

容易混淆怎麼分
Master Data vs Transaction Data前者是反覆引用的基本對象；後者是一次交易或事件留下的紀錄
Data Owner vs System OwnerData Owner 決定資料規則；System Owner 負責系統生命週期與運作
Data Steward vs Data EntrySteward 監看品質並協調例外，不只是把資料輸入系統

若還覺得模糊，先問一句：「這個人是在決定資料規則，還是在執行與維護規則？」通常就能分出來。

案例

Northstar 的客戶主資料責任表

Northstar 把客戶主資料拆成幾個責任領域：

資料領域       Data Owner       Data Steward
客戶基本資料    銷售營運主管      CRM 營運專員
帳單與稅務資料  財務主管          ERP 主檔管理人員
行銷訂閱狀態    行銷營運主管      行銷平台管理人員
  1. 角色：三位 Owner、三類 Steward，以及使用資料的業務人員。
  2. 資料流：各領域依 5-1 指定的正本系統維護，再同步到副本。
  3. 可能出錯：把「客戶資料」全部交給同一位 Owner，導致稅務與訂閱規則沒人敢決定。
  4. 第一個檢查方向：先查爭議欄位屬於哪個資料領域，再找對應的 Owner。
顧問可以先問「這個欄位的規則由哪個業務角色核定？日常異常由誰處理？」

常見誤解

三個責任分工的陷阱

誤解一：「資料放在 CRM，CRM 管理員就是 Owner」

管理員負責系統設定，不代表他有權決定業務規則。Owner 通常是該資料領域的業務主管。

誤解二：「找了 Owner，他就要親自修每一筆資料」

Owner 負責決策與承擔結果；日常檢查、協調修正多由 Steward 執行。

誤解三：「設一位全公司 Data Owner 最省事」

客戶、財務、產品各有不同專業。責任切得太粗，最後仍會回到沒人能拍板。

遇到問題先查什麼

資料規則沒人決定時的五步

  1. 確認爭議的是哪一個欄位或資料領域。
  2. 查責任表是否已指定 Data Owner 與 Data Steward。
  3. 找 Owner 確認規則、品質門檻與例外處理原則。
  4. 找 Steward 盤點受影響資料並安排修正。
  5. 把決定寫回責任表、資料字典或作業流程。

前兩步只需要把責任文件和欄位清單攤開，不用先查程式。

重點整理

帶走這四句話

  1. Master Data 是跨流程反覆使用的基本名冊，常見例子是客戶、產品與供應商。
  2. Data Owner 有決策權，負責核定規則並承擔結果。
  3. Data Steward 執行規則、監看品質並協調日常修正。
  4. 先按資料領域分責任，再把核定、檢查與修正動作寫清楚。
檢核 1/4

哪一項最符合 Master Data？

正確答案是 B。產品主檔會被許多訂單與流程反覆引用，屬於主資料。
檢核 2/4

誰應該核定「客戶產業分類」的業務規則？

正確答案是 A。Data Owner 對資料領域有決策權；系統管理員可以協助實作，但不代替業務拍板。
檢核 3/4

哪一項最像 Data Steward 的日常工作？

正確答案是 C。監看品質、追查問題與協調修正，正是 Steward 的日常責任。
檢核 4/4

欄位規則討論三次仍沒人敢決定，顧問第一步該做什麼？

正確答案是 A。問題卡在決策責任，不是技術；先找有權拍板的 Owner。
