Status Code
系統回覆的暗號

預估 30 分鐘 ・ 建議先讀完 2-2、2-3 再來。

您一定看過網頁跳出「404」或聽過工程師說「回傳 500」。這些三位數的數字,是 Server 在 Response 裡附上的「暗號」,告訴 Client 這次請求到底處理得怎麼樣。這一章要教您看懂最常見的幾個暗號。

  • 看懂 200、201、400、401、403、404、429、500 這幾個常見狀態碼的意思。
  • 分辨「這是誰的問題」——是 Client 端的問題,還是 Server 端的問題。
  • 拿到一個陌生的狀態碼時,知道大概該往哪個方向猜。
情境

「畫面說 403,這是什麼意思?」

Northstar Electronics 的客服人員想開啟某個內部系統頁面,畫面卻顯示了一段陌生的訊息,她拍了截圖傳給 IT:

「畫面上寫著 403 Forbidden,這是系統壞了嗎?」

其實不是系統壞了。403 是 Server 明確告訴 Client:「我收到你的請求了,但我不允許你做這件事。」這跟「系統掛掉」是完全不同的狀況。這一章會教您把這些數字讀成一句話,而不是一串亂碼。

情境例子 1/2

信件回條上的簡短代號

想像您寄了一封信,對方寄回一張回條,上面只印了一個代號,不同代號代表不同意思:「已收到並辦妥」「地址錯誤,查無此人」「這件事我不受理,禁止辦理」。您一看代號,不用看完整封信,就能知道大概發生了什麼事。

HTTP 的 Status Code(狀態碼)正是這樣的機制。Server 每次回傳 Response,都會附上一個三位數的代號,像 200、404、500,用最短的方式告訴 Client 這次請求的結果。

先記住這句 狀態碼的第一位數字,是分類的關鍵:2 開頭表示成功,4 開頭表示「問題出在 Client 這邊」,5 開頭表示「問題出在 Server 這邊」。
情境例子 2/2

「你寫錯地址」和「我們家出事了」

延續回條的比喻,4 開頭的代號,像是「您寫的地址有誤」「這裡禁止辦理這件事」——問題出在寄件人(Client)這邊,可能是地址寫錯、沒有權限辦這件事。5 開頭的代號,則像是「收件單位內部出了狀況,沒辦法處理您的信」——問題出在收件方(Server)自己身上,不是您的錯。

回到開頭客服人員遇到的 403。它屬於 4 開頭,代表問題出在「請求方這邊」——具體來說,是這位客服人員的帳號沒有被授權存取這個頁面,不是系統故障。正確的處理方式,是去確認她的帳號權限設定,而不是通報系統當機。

正式定義

常見狀態碼的正式說法

狀態碼意思誰的問題
200成功,請求已正常處理—
201成功,而且新建立了一筆資料—
400請求的格式或內容有誤,Server 看不懂Client 端
401尚未通過身分驗證,需要先登入或提供正確憑證Client 端
403身分已確認,但沒有權限做這件事Client 端
404找不到這個網址對應的內容Client 端
429短時間內請求次數太多,被限制了Client 端
500Server 內部發生錯誤,無法正常處理Server 端

200 系列代表成功,不需要特別擔心;4 開頭需要先檢查 Client 端(是不是網址打錯、有沒有登入、有沒有權限);5 開頭則要請工程師去查 Server 端到底出了什麼狀況。

串起來看

同一個登入頁面,不同的狀態碼結果

用一個登入場景,走一遍不同狀態碼可能出現的情境:

情境一:帳號密碼都對 → Server 回傳 200,登入成功,進入系統
情境二:帳號密碼打錯   → Server 回傳 401,提示請重新輸入
情境三:帳號已停用     → Server 回傳 403,即使密碼正確也拒絕登入
情境四:網址打錯了     → Server 回傳 404,找不到這個頁面
情境五:系統當機了     → Server 回傳 500,連處理請求的能力都出問題了

五種情境,使用者看到的可能都是「登入失敗」,但背後的狀態碼完全不同,對應的解決方式也完全不同。這也是為什麼請使用者提供狀態碼或截圖,對排查問題非常有幫助。

為什麼要學這個

兩個馬上用得到的場景

場景一:第一時間判斷問題方向

使用者回報問題時,先問一句「畫面上有沒有出現數字或錯誤代碼?」如果是 4 開頭,可以先自行檢查帳號權限、網址是否正確;如果是 5 開頭,就該立刻通知工程師,因為問題出在系統本身。

場景二:向客戶精準說明問題性質

與其說「系統出錯了」,不如說「系統回傳 403,代表您的帳號目前沒有存取這個功能的權限」。這種說法能讓客戶清楚知道問題不是系統故障,而是權限設定需要調整,避免不必要的緊張。

核心概念 1/2

第一位數字,決定了大方向

概念一:2 開頭是成功,不用緊張

200 和 201 都代表請求成功處理,差別只在 201 特別強調「這次還新建立了一筆資料」。看到 2 開頭,代表這段溝通本身沒有問題。

概念二:4 開頭,先檢查請求方自己

400、401、403、404、429 這些 4 開頭的狀態碼,共通點是「問題出在這次請求本身」——可能是格式錯誤、沒登入、沒權限、網址錯誤,或是請求太頻繁。看到 4 開頭,第一直覺應該是檢查請求端的設定,而不是懷疑系統壞了。

核心概念 2/2

5 開頭,是系統自己說「我這邊有狀況」

概念三:5 開頭代表 Server 端出了問題

500 這類狀態碼,代表 Server 在處理請求的過程中,自己發生了無法處理的錯誤。這不是使用者操作有誤造成的,而是系統本身需要工程師介入排查。

概念四:狀態碼是溝通問題的共同語言

不管是跟工程師、客戶 IT,還是原廠 Support 溝通,提供準確的狀態碼,遠比描述「畫面怪怪的」更有效率。這是顧問可以立刻開始使用、幾乎零成本的溝通升級。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
401 vs 403401 是「還不知道你是誰」,通常代表需要先登入或提供正確憑證;403 是「已經知道你是誰,但不允許你做這件事」,通常代表權限不足。前者先解決身分驗證,後者要調整權限設定。
4 開頭 vs 5 開頭4 開頭是請求本身有問題(格式、身分、權限、網址、頻率);5 開頭是 Server 自己處理時發生錯誤。前者先檢查請求端,後者要交給工程師查 Server。
案例

Northstar Electronics 的 API 串接異常

Northstar 的行銷平台每天定時呼叫 CRM System 的 API,同步最新的客戶名單。某天開始,同步作業持續失敗:

錯誤紀錄:連續多次呼叫都回傳 429
時間點:剛好是行銷平台換了新的排程頻率,從每小時一次改成每五分鐘一次
CRM System 的說明文件:針對這個 API,限制每小時最多呼叫 10 次
  1. 有哪些角色?行銷平台(呼叫方,Client)、CRM System(被呼叫方,Server)。
  2. 資料如何移動?行銷平台定期發送請求給 CRM System,要求提供最新客戶資料。
  3. 各系統負責什麼?CRM System 為了保護自己的服務品質,對每個呼叫方設定了頻率限制。
  4. 哪裡可能出錯?行銷平台把排程頻率改得太高,超過了 CRM System 允許的呼叫次數上限。
  5. 第一個合理的檢查方向?看到 429,第一步就該去查這個 API 的呼叫頻率限制文件,確認最近有沒有調整過排程設定,而不是懷疑 CRM System 故障。

常見誤解

三個很多人踩過的坑

誤解一:「畫面出現數字錯誤代碼,一定是系統壞了」

4 開頭的狀態碼通常代表請求本身有問題,不是系統故障。先看清楚是哪個數字,再判斷該往哪個方向排查。

誤解二:「401 和 403 是一樣的意思」

401 是身分還沒確認,403 是身分已確認但權限不足。兩者的解決方式不同:前者要處理登入或憑證,後者要調整權限設定。

誤解三:「狀態碼是 200 就代表整個流程都成功了」

200 只代表這一次 Request/Response 的溝通本身成功,不代表後續所有處理都完美無誤。就像 1-1 教過的:畫面顯示成功,不保證資料真的完整落地,狀態碼也是類似的道理——它只回答「這次溝通有沒有順利完成」。

遇到問題先查什麼

「系統出現錯誤代碼」的檢查順序

  1. 先看清楚具體是哪個狀態碼。請使用者提供截圖或完整訊息。
  2. 判斷是 4 開頭還是 5 開頭。決定接下來該檢查請求端還是系統端。
  3. 如果是 4 開頭,先確認網址、身分驗證、權限是否正確。這些通常顧問自己就能初步排查。
  4. 如果是 5 開頭,立刻通知工程師查看 Server 端的錯誤紀錄。這類問題通常需要技術人員介入。
  5. 把狀態碼和發生時間一併提供給處理的人。這能大幅縮短排查時間。
重點整理

帶走這四句話

  1. Status Code 是 Server 附在 Response 裡的三位數暗號,快速說明這次請求的處理結果。
  2. 2 開頭代表成功;4 開頭代表問題出在請求方(格式、身分、權限、頻率);5 開頭代表問題出在 Server 自己身上。
  3. 401 是身分未確認,403 是身分確認但權限不足,兩者處理方式不同。
  4. 提供準確的狀態碼給工程師或客戶,遠比描述「畫面怪怪的」更有效率。
檢核 1/4

使用者已經登入成功,但嘗試存取一個沒有權限的頁面,最可能收到哪個狀態碼?

正確答案是 B。403 代表身分已經確認,但沒有權限做這件事,正是這個情境的典型狀態碼。
檢核 2/4

是非題:狀態碼 500 代表問題出在 Server 端,不是使用者操作有誤。

正確答案是「是」。500 代表 Server 內部處理請求時發生錯誤,問題出在系統本身,需要工程師介入排查。
檢核 3/4

某個排程作業在短時間內對同一個 API 呼叫太多次,最可能收到哪個狀態碼?

正確答案是 C。429 代表短時間內請求次數太多,被系統限制了,這是排程頻率設定過高時常見的狀態碼。
檢核 4/4

情境題:客戶說畫面顯示「404」。您的第一個合理懷疑方向是?

正確答案是 A。404 代表找不到這個網址對應的內容,通常是網址打錯、頁面被移除或連結失效,問題出在請求方這一端要先確認的細節。
