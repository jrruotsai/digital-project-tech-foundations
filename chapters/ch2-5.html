HTTPS、Certificate
與安全傳輸

預估 30 分鐘 ・ 建議先讀完 2-1、2-2 再來。

網址前面的那個小鎖頭圖示,還有 2-1 提過的 https://,到底在保護什麼?這一章要把「安全傳輸」這件事講清楚,也會提到一個之後常會遇到的麻煩:憑證過期。

  • 說明 HTTPS 比 HTTP 多做了什麼事。
  • 理解 Certificate(憑證)在這個過程中扮演的角色。
  • 知道憑證過期時,系統會出現什麼樣的警訊。
情境

「網站突然跳出『不安全』的警告」

Northstar Electronics 的官網某天早上,訪客打開網站時,瀏覽器跳出一則警告畫面,寫著「您的連線不是私人連線」。行銷同仁緊張地問:

「網站被駭客入侵了嗎?怎麼瀏覽器說不安全?」

先別緊張,這種情況最常見的原因,其實跟入侵完全無關,而是這一章要教的「憑證過期」。理解 HTTPS 和 Certificate 的運作方式,您就能一眼看出這是個需要盡快處理、但性質完全不同的問題。

情境例子 1/2

寄明信片,還是寄加密信件

想像您要寄一份重要文件。第一種方式,是寫在明信片上直接投遞——路上經手的每個郵差,都能輕易看到內容。這就像 HTTP:資料在傳輸過程中是「明碼」的,理論上路徑中間的人可以偷看到內容。

第二種方式,是把文件放進一個加了密碼鎖的信封,只有收件人才有鑰匙打得開。這就是 HTTPS(HTTP Secure)——在 HTTP 的基礎上,多加了一層加密,讓資料在傳輸過程中即使被攔截,也無法直接被看懂內容。

先記住這句 HTTPS 不是另一套全新的協定,是 HTTP 加上一層加密的保護。這也是為什麼網址前面看到 https://,而不是完全不同的字。
情境例子 2/2

那把鎖上鑰匙的信封,鑰匙從哪裡來

問題來了:加密信封需要一把鑰匙,這把鑰匙要怎麼證明是「真正屬於這個網站」的,而不是別人假冒的?這裡需要一個公正的第三方來擔保。

Certificate(憑證)就像是網站的「身分證」,由公正的發證機構核發,證明「這個網址,確實屬於這家公司,可以放心使用加密通訊」。瀏覽器會檢查這張身分證,如果核對無誤,就顯示那個安全的鎖頭圖示。

回到開頭的警告訊息。身分證跟真實信件一樣,是有使用期限的。如果 Northstar 官網的憑證過期了,瀏覽器會認為「我沒辦法確認這個網站的身分證是不是還有效」,因此跳出警告,提醒訪客要小心。這不代表網站被入侵,而是「身分證該換發了,卻沒人記得去換」。

正式定義

三個名詞的正式說法

名詞定義
HTTPS在 HTTP 的基礎上,加入加密機制的安全傳輸協定
SSL/TLS實現 HTTPS 加密功能的技術標準,TLS 是 SSL 的後繼版本,一般口語仍常合稱 SSL
Certificate
憑證		由公正機構核發,證明某個網域確實屬於特定單位,並支援加密通訊的電子身分證明

簡單記憶:HTTPS 是「安全傳輸這件事」,SSL/TLS 是「怎麼做到加密的技術」,Certificate 是「證明身分的那張文件」。三者合作,才構成瀏覽器上那個安心的鎖頭圖示。

串起來看

打開一個 HTTPS 網站,背後多做了什麼

接續 2-1 教過的旅程,補上 HTTPS 特有的步驟:

1. Browser 透過 DNS 找到 Server 的 IP(2-1 教過)
2. Browser 向 Server 要求查看它的 Certificate(身分證)
3. Browser 核對這張 Certificate 是否有效、是否過期、是否真的屬於這個網域
4. 核對通過,雙方建立加密連線
5. 接下來的 Request 和 Response,都透過這條加密連線傳輸

第 2、3 步是 HTTPS 特有的,HTTP 沒有這道關卡。多這幾步雖然增加了一點點時間,但換來的是資料傳輸過程中,不容易被中途攔截偷看。

為什麼要學這個

兩個馬上用得到的場景

場景一:判斷「網站不安全」警告的真正原因

看到瀏覽器跳出安全警告,先確認是不是憑證過期或設定錯誤,而不是立刻假設遭到入侵。這兩種情況處理的急迫性和方式都不同:憑證問題通常是忘記更新,入侵則需要完全不同層級的資安應變。

場景二:提醒客戶注意憑證到期日

憑證通常有使用期限,需要定期更新。作為顧問,提早提醒客戶留意即將到期的憑證,能避免臨時出現「網站被判定不安全」的尷尬狀況,尤其是在正式上線前的檢查清單裡,這是很值得列入的一項。

核心概念 1/2

HTTPS 保護的是「傳輸過程」

概念一:HTTPS 解決的是「中途被偷看或竄改」的風險

資料從 Client 傳到 Server 的路上,可能經過很多中間節點(2-1 提過的網路路徑)。HTTPS 確保即使資料經過這些節點,也不會輕易被讀懂或竄改內容。

概念二:HTTPS 不能保證「網站內容本身」值得信任

HTTPS 只保護傳輸過程的安全,不代表這個網站的內容一定正確或可靠。一個詐騙網站一樣可以申請憑證、使用 HTTPS。看到鎖頭圖示,只代表傳輸過程有加密,不代表這個網站本身沒有問題。

核心概念 2/2

憑證是有期限的,需要人去維護

概念三:憑證過期是最常見的「假警報」

比起真正的資安入侵,憑證忘記更新導致的警告訊息,在實務上發生的頻率高出許多。遇到安全警告時,先檢查憑證到期日,往往能快速找到答案。

概念四:憑證的維護責任,通常在 IT 或負責維運的團隊

憑證需要定期更新,如果沒有排入例行檢查或自動化更新機制,很容易因為疏忽而過期。這也是單元 9 之後會再深入教的內容之一——憑證管理跟身分驗證機制息息相關。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
HTTPS vs 「網站內容安全」HTTPS 保護的是傳輸過程不被竊聽或竄改;網站內容本身是否可信,是另一回事。看到鎖頭圖示,不代表網站內容一定沒問題。
憑證過期 vs 網站遭入侵憑證過期通常只是忘記更新這份「身分證」,瀏覽器因此無法確認網站身分而發出警告;遭入侵則是網站本身的內容或系統被惡意操控,兩者性質完全不同,處理方式也不一樣。
案例

Northstar Electronics 的憑證過期事件

追查行銷同仁反映的警告訊息,發現原因如下:

官網憑證的有效期限:一年
上次更新時間:剛好滿一年前
負責維護的同仁:已經離職,沒有交接這項例行工作
結果:憑證到期沒人注意到,直到訪客反映警告訊息才被發現
  1. 有哪些角色?官網 Server、憑證發證機構、IT 維運團隊、訪客。
  2. 資料如何移動?訪客的 Browser 嘗試核對憑證有效性,發現已過期,因此顯示警告。
  3. 各角色負責什麼?IT 團隊原本該負責定期更新憑證,確保不會過期。
  4. 哪裡可能出錯?憑證更新是一項容易被忽略的例行工作,尤其在人員異動時,如果沒有明確交接或自動化提醒機制,很容易被遺漏。
  5. 第一個合理的檢查方向?看到安全警告,先查憑證的到期日,確認是不是這個常見原因,再考慮是否有更嚴重的資安疑慮。

常見誤解

三個很多人踩過的坑

誤解一:「看到瀏覽器安全警告,就代表網站被駭客入侵」

最常見的原因其實是憑證過期或設定錯誤,不一定跟入侵有關。先查憑證狀態,再考慮更嚴重的可能性。

誤解二:「有 HTTPS 鎖頭圖示,就代表這個網站絕對安全可信」

HTTPS 只保護傳輸過程,不代表網站內容本身值得信任。任何人都可以為自己的網站申請憑證,包括詐騙網站。

誤解三:「憑證申請一次就永久有效」

憑證通常有固定的有效期限,需要定期更新。這是一項需要排入例行維護排程的工作,不是一勞永逸的設定。

遇到問題先查什麼

「網站出現安全警告」的檢查順序

  1. 確認警告訊息的具體內容。是提到憑證問題,還是其他類型的警告?
  2. 查詢這個網域的憑證到期日。許多線上工具可以快速查詢,或請 IT 團隊確認。
  3. 確認最近是否有更換網域或伺服器設定。設定錯誤也可能導致憑證核對失敗。
  4. 如果憑證確實過期,盡快聯繫負責的 IT 團隊更新。這通常是可以快速解決的問題。
  5. 如果排除憑證問題後警告仍然出現,才進一步懷疑資安風險。這時候需要更專業的資安人員介入排查。
重點整理

帶走這四句話

  1. HTTPS 是在 HTTP 的基礎上加入加密,保護資料在傳輸過程中不被輕易偷看或竄改。
  2. Certificate(憑證)像網站的身分證,證明這個網域確實屬於特定單位,並支援加密通訊。
  3. 憑證有使用期限,過期是造成瀏覽器安全警告最常見的原因,不代表一定遭到入侵。
  4. 看到安全警告,先查憑證狀態,這往往是成本最低、也最常見的答案。
檢核 1/4

HTTPS 相較於 HTTP,主要多做了什麼事?

正確答案是 B。HTTPS 是在 HTTP 的基礎上,加入加密機制,保護資料在傳輸過程中不被輕易偷看或竄改。
檢核 2/4

是非題:瀏覽器跳出「連線不安全」的警告,最常見的原因是憑證過期,不一定代表網站遭到入侵。

正確答案是「是」。實務上,憑證過期或設定錯誤是造成這類警告最常見的原因,遠比真正的入侵事件常見得多。
檢核 3/4

Certificate(憑證)在 HTTPS 的運作中,主要扮演什麼角色?

正確答案是 C。Certificate 就像網站的身分證,由公正機構核發,證明這個網域確實屬於特定單位,並支援加密通訊。
檢核 4/4

情境題:官網突然出現安全警告,行銷同仁很緊張。您的第一個合理檢查方向是?

正確答案是 B。憑證過期是造成安全警告最常見的原因,先確認這一點,能快速釐清狀況,避免不必要的恐慌。
