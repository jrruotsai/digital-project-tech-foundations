Cache 與 CORS
為什麼畫面是舊的／請求被擋掉

預估 30 分鐘 ・ 建議先讀完 1-2、2-2 再來。

1-2 提過 Browser 會暫存舊內容,這一章要正式講清楚這個機制,還會教一個常讓工程師頭痛的詞:CORS。這兩個概念常常一起出現在「畫面怪怪的」這類回報裡。

  • 說明 Cache 造成的常見假象。
  • 用白話理解 CORS 在防範什麼。
  • 看到「這是 CORS 問題」時,知道大概是怎麼回事。
情境

「我們明明改了網站內容,客戶看到的還是舊的」

Northstar Electronics 更新了官網的產品價格,行銷同仁隔天收到反映:

「客戶說看到的價格還是舊的,但我們自己這邊看起來是新的,是不是資料庫沒更新成功?」

先別急著懷疑資料庫。這個現象十之八九跟 1-2 提過的 Cache(快取)有關——客戶的瀏覽器,拿著舊版本的頁面內容重複使用,沒有跟 Server 要最新的版本。這一章要把這個機制講透。

情境例子 1/2

抄一份筆記,下次直接看筆記

想像您每次要用某份資料,都得跑一趟很遠的圖書館去借。為了省時間,您第一次借到之後,乾脆抄一份筆記留在身邊,下次要用就直接看筆記,不用再跑一趟。

Cache(快取)就是這份「隨身筆記」。Browser 第一次向 Server 要某個網頁內容時,會把它暫存一份在本機;下次再造訪同一個網址,如果判斷這份筆記還「夠新」,就直接顯示筆記內容,不用重新跑一趟去問 Server。這樣做的好處是網頁開得更快,壞處就是筆記可能已經過時,而您沒發現。

先記住這句 Cache 是為了「更快」而暫存的舊筆記。它的存在,天生就帶著「資料可能不是最新的」這個風險。
情境例子 2/2

不同店家借東西,得先問過對方願不願意借

現在換個場景。假設 Northstar 的官網,想直接向另一個完全不同網域的系統要資料來顯示在自己頁面上。這時候瀏覽器會有一層安全機制介入:它會先確認「這個資料的提供方,有沒有同意讓我這個網域的網站來拿?」

這就像您想跟隔壁店家借工具,店家會先確認「我認識你這家店嗎?我允許嗎?」如果對方沒有明確同意,即使東西就在那裡,瀏覽器也會擋下這次請求,不讓資料被拿走。這道安全機制,叫做 CORS(Cross-Origin Resource Sharing,跨來源資源共享)。

CORS 不是在防普通使用者,而是瀏覽器在保護「未經同意,一個網站不能任意向另一個網站要資料」這條防線。如果 Server 端沒有正確設定「我同意讓這些網域來要資料」,即使資料本身沒有問題,瀏覽器也會主動擋下,這在工程師之間常被簡稱為「CORS 錯誤」。

正式定義

兩個名詞的正式說法

名詞定義
Cache
快取		Browser(或其他中間節點)暫存的資料副本,用來加快下次存取同樣內容的速度
CORS瀏覽器的安全機制,限制網頁只能向明確授權的其他網域請求資料,防止未經同意的跨網域資料存取

兩者看似不相關,但都跟「瀏覽器如何處理跨頁面、跨網域的請求」有關,也都是排查「畫面怪怪的」或「資料要不到」時,常被提到的關鍵字。

串起來看

一次網頁載入,Cache 和 CORS 各自登場的時機

情境一(Cache):
使用者造訪 northstar-electronics.com
→ Browser 檢查本機是否已有這個網址的快取
→ 如果有,且判斷還夠新鮮,直接顯示快取內容(不問 Server)
→ 如果沒有,或已過期,才向 Server 要最新內容

情境二(CORS):
northstar-electronics.com 的頁面,想跟另一個網域 api.partner.com 要資料
→ Browser 先確認 api.partner.com 是否同意讓這個網域請求資料
→ 同意,才把資料交給頁面顯示;不同意,直接擋下,並回報 CORS 錯誤

兩種情境都發生在 Browser 這一層,跟 2-2 教過的 Request/Response 息息相關,只是分別在處理「要不要重新問一次」和「能不能跨網域問」這兩個不同的問題。

為什麼要學這個

兩個馬上用得到的場景

場景一:判斷「畫面沒更新」是不是 Cache 造成的

客戶反映「內容還是舊的」,先請對方試著強制重新整理頁面(通常是按住特定按鍵再重新整理),如果這樣就恢復正常,代表原因就是 Cache,而不是後端資料真的沒更新。這能省下大量往資料庫方向排查的時間。

場景二:理解跨系統整合時,CORS 是常被卡住的一關

如果專案需要讓某個網頁直接向另一個系統要資料(而不是透過後端中轉),CORS 設定是否正確,是工程師常常需要事先確認的細節。提早知道這個概念,能在需求討論階段就提醒工程師留意。

核心概念 1/2

Cache 讓「快」和「新」之間有取捨

概念一:Cache 存在多個層級,不是只有 Browser

除了使用者的 Browser,網路路徑中間的一些節點,也可能暫存內容(這超出本章要深入的範圍,先知道 Cache 不是只發生在使用者自己的電腦上即可)。這也是為什麼有時候即使清了自己的瀏覽器快取,問題依然存在。

概念二:「強制重新整理」是排查 Cache 問題最簡單的第一步

大多數瀏覽器都有「不使用快取、直接跟 Server 要最新內容」的重新整理方式。這是判斷問題是否為 Cache 造成的,成本最低的檢查手段。

核心概念 2/2

CORS 是保護機制,不是系統故障

概念三:CORS 錯誤代表安全機制正常運作,不是系統壞掉

看到 CORS 錯誤,代表瀏覽器正確地執行了它該做的保護動作——擋下一次未經授權的跨網域請求。要解決它,通常是請提供資料的那一方,在自己的設定裡明確同意讓對方網域存取,而不是想辦法繞過這道機制。

概念四:CORS 只在「跨網域」時才會介入

如果 Request 是發給自己所在的同一個網域,不會觸發 CORS 的檢查。只有當網頁想跟不同網域的系統要資料時,才會遇到這道關卡,這也是為什麼許多內部串接不會遇到 CORS 問題,只有對外部系統的整合才容易碰上。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
Cache 問題 vs 資料庫沒更新Cache 問題通常只影響「看到的畫面」,強制重新整理就能驗證;資料庫沒更新則是資料本身真的還沒改變,重新整理也不會有幫助。用強制重新整理測試,能快速分辨兩者。
CORS 錯誤 vs 一般的網路連線失敗CORS 錯誤是瀏覽器主動擋下的安全機制,通常訊息裡會明確提到 CORS 或跨來源相關字眼;一般連線失敗則是根本連不上,兩者的錯誤訊息和排查方向不同。
案例

Northstar Electronics 的價格更新事件

追查客戶反映「價格還是舊的」問題,發現:

Northstar 端:資料庫的價格已經正確更新,後端回傳的內容也是正確的
客戶端:瀏覽器暫存了舊版頁面,還沒有向 Server 要最新內容
驗證方式:請該客戶強制重新整理頁面,結果立刻顯示正確的新價格
  1. 有哪些角色?官網後端與資料庫(Northstar 端)、客戶的 Browser(Client 端)。
  2. 資料如何移動?資料庫和後端內容其實已經正確,只是客戶的 Browser 還在使用本機暫存的舊版本。
  3. 各部分負責什麼?Cache 機制原本的目的是加快網頁載入速度,但也帶來了「顯示內容可能落後」的副作用。
  4. 哪裡可能出錯?如果第一時間就往資料庫方向排查,反而會浪費時間,因為問題根本不在那裡。
  5. 第一個合理的檢查方向?先請客戶強制重新整理頁面,這個動作幾乎零成本,卻能立刻排除或確認 Cache 是否為原因。
延伸對照(選讀)

這些概念在系統整合裡的樣子

不熟悉這些產品的話,這張卡片可以跳過。

通用概念常見情境
Cache行銷平台的活動頁面更新後,CDN(內容傳遞網路)快取需要一段時間才會刷新到全球節點
CORS企業官網的前端要直接呼叫第三方 API(例如地圖服務)時,需要對方明確開放 CORS 設定
常見誤解

三個很多人踩過的坑

誤解一:「畫面沒更新,一定是後端或資料庫的問題」

Cache 是造成這類現象非常常見的原因,而且驗證成本很低——先請使用者強制重新整理,再往其他方向排查。

誤解二:「CORS 錯誤代表網路壞了」

CORS 錯誤代表瀏覽器主動擋下了一次跨網域請求,通常是設定沒有正確開放授權,不是網路本身有問題。

誤解三:「CORS 問題可以靠使用者端解決」

CORS 的授權設定,必須由提供資料的那一方(Server)去開放,使用者或前端這一側通常無法自行繞過或修正這個限制。

遇到問題先查什麼

「畫面沒更新」或「資料要不到」的檢查順序

  1. 先請使用者強制重新整理頁面。如果內容恢復正常,原因就是 Cache。
  2. 確認是否為跨網域的請求。如果錯誤訊息提到跨來源或 CORS 字眼,方向就對了。
  3. 確認是否只有部分使用者受影響。Cache 問題通常因人而異,取決於各自瀏覽器暫存的內容。
  4. 如果排除 Cache,再確認後端資料本身是否正確。這時才需要往資料庫或後端邏輯排查。
  5. 如果是 CORS 問題,請工程師確認提供資料的一方是否已正確開放授權設定。
重點整理

帶走這四句話

  1. Cache 是 Browser 暫存的舊內容,目的是加快載入速度,代價是可能顯示過時的資料。
  2. CORS 是瀏覽器的安全機制,擋下未經授權的跨網域資料請求。
  3. 「畫面沒更新」先懷疑 Cache,強制重新整理是成本最低的驗證方式。
  4. CORS 錯誤代表安全機制正常運作,解法是請資料提供方正確開放授權,不是想辦法繞過去。
檢核 1/4

Browser 暫存資料以加快下次載入速度的機制,叫做什麼?

正確答案是 B。Cache(快取)是暫存的資料副本,用來加快下次存取同樣內容的速度。
檢核 2/4

是非題:CORS 錯誤代表瀏覽器的安全機制正常運作,不代表系統故障。

正確答案是「是」。CORS 錯誤代表瀏覽器主動擋下了一次未經授權的跨網域請求,是安全機制在正常運作,不是系統故障。
檢核 3/4

客戶反映網頁內容還是舊版本,最低成本的第一個驗證方式是什麼?

正確答案是 C。強制重新整理能讓瀏覽器不使用快取、直接向 Server 要最新內容,是驗證 Cache 是否為原因,成本最低的第一步。
檢核 4/4

情境題:工程師說「這是 CORS 問題」。正確的解決方向是什麼?

正確答案是 B。CORS 的授權設定必須由提供資料的一方調整,使用者端無法自行繞過,也不是單純重開機能解決的問題。
