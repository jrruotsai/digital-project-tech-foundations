HTTP Request 與 Response
一問一答的規則

預估 30 分鐘 ・ 建議先讀完 1-2 和 2-1 再來。

1-2 教過「Client 發問、Server 回答」,2-1 教過「網址怎麼找到 Server」。這一章要打開那個「問」和「答」的信封,看看裡面實際裝了什麼。這是之後理解 API(單元 6)的必要基礎。

  • 說明 HTTP Request 和 Response 各自包含哪些部分。
  • 認得 Header、Body、Query Parameter 這三個常見詞。
  • 看到工程師提到「這個 Request 帶了什麼」時,知道他在問什麼。
情境

「工程師說要看 Request 內容,我完全不知道那是什麼」

Northstar Electronics 的顧問在協助排查一個表單問題,工程師回了一句:

「我需要看一下那個 Request 的 Header 和 Body,才能確認問題出在哪。」

如果您完全不知道 Header、Body 是什麼,這句話聽起來就像外星語言。但只要理解「一次 HTTP 溝通,就像寄一封結構化的信」,這些詞馬上會變得具體。這一章就是要把這封信拆開來看。

情境例子 1/2

一封信的信封和信紙

想像您寄一封正式信件,信封上會寫收件地址、寄件地址;信紙裡才是您真正想說的內容。HTTP(HyperText Transfer Protocol,超文本傳輸協定)就是規定「這封信該怎麼寫、怎麼寄」的一套規則,是 2-1 提過那串網址前面 https:// 的正式名稱由來。

Client 發出的那封信,叫 Request(請求)——「我要做什麼、要什麼資料」。Server 回的那封信,叫 Response(回應)——「這是處理結果、這是您要的內容」。這正是 1-2 教過的一問一答,只是這裡要看得更細。

先記住這句 Request 是「我要什麼」,Response 是「這是結果」。兩者結構很像,都分成「信封上的資訊」和「信紙裡的內容」兩部分。
情境例子 2/2

信封上的備註,和信紙裡的正文

一封 Request(或 Response)信件,拆開來看有兩個部分:

Header(標頭)就像信封上額外寫的備註:這封信是什麼格式(純文字還是有附件)、寄件人是誰、需不需要對方先確認身分。它是溝通這次往來的「附加資訊」,不是主要內容本身。

Body(本文)就是信紙裡真正的內容:如果是 Request,可能是您在表單裡填的資料;如果是 Response,可能是 Server 回傳給您的實際資訊。

還有一個常見的詞:Query Parameter(查詢參數),它是寫在網址後面、用問號開頭的一小段資訊,像是 ?keyword=手機,常用來告訴 Server「我要篩選哪些條件」,可以想成是寫在信封正面的「特別指定條件」,比 Header 更顯眼,也更常被使用者直接看到(因為它就出現在網址列上)。

視覺圖解

Request 去，Response 回

一次 HTTP 溝通不是單向傳資料，而是 Client 問一次、Server 回一次。

正式定義

幾個關鍵名詞的正式說法

名詞定義
HTTP規範 Client 與 Server 之間如何請求與回應的通訊協定
Request
請求		Client 發送給 Server 的訊息,說明想做什麼、想要什麼資料
Response
回應		Server 處理完 Request 後,回傳給 Client 的結果
Header
標頭		Request 或 Response 中,描述這次溝通額外資訊的部分(如格式、身分驗證資訊)
Body
本文		Request 或 Response 中,真正承載的資料內容
Query Parameter
查詢參數		附加在網址後面、用來傳遞篩選或設定條件的資訊
串起來看

填一次表單,一次 Request/Response 的完整內容

接續 1-1 的報名表單案例,拆開這次 Request 和 Response 裡實際裝了什麼:

使用者送出的 Request(簡化示意):
  方法:POST(2-3 會教這是什麼意思)
  網址:northstar-electronics.com/register
  Header:內容格式是「表單資料」
  Body:{ "name": "王小明", "email": "wang@example.com" }

Server 回傳的 Response(簡化示意):
  狀態:200(2-4 會教,代表成功)
  Header:內容格式是「JSON 資料」
  Body:{ "status": "success", "message": "報名成功" }

使用者填的姓名、Email,放在 Request 的 Body 裡;Server 說「有沒有成功」,放在 Response 的 Body 裡。Header 則像是兩邊溝通時,附加說明「這包資料是什麼格式」的小紙條。

為什麼要學這個

兩個馬上用得到的場景

場景一:聽懂工程師在查什麼

工程師說「Request 有沒有帶正確的 Header」,通常在確認身分驗證資訊或資料格式有沒有設定對;說「Body 裡的資料對不對」,則是在確認實際送出的內容是否正確。聽懂這些詞,能讓您跟得上排查問題的討論,甚至幫忙補充資訊。

場景二:描述問題時更精準

與其說「表單怪怪的」,不如說「送出表單時,Request 的 Body 裡好像缺了 Email 欄位」。這種描述能讓工程師直接鎖定方向,不用再花時間問您細節。

核心概念 1/2

Request 和 Response 結構是對稱的

概念一:兩邊都有 Header 和 Body

不管是 Client 發出的 Request,還是 Server 回傳的 Response,都遵循同一套結構:Header 放附加資訊,Body 放實際內容。理解了一邊,另一邊就很容易類推。

概念二:Header 是「關於這次溝通」的資訊,不是「內容本身」

常見的 Header 資訊包括:這次傳的是什麼格式的資料、有沒有附上身分驗證的憑證。它們幫助 Server(或 Client)正確解讀 Body 裡的內容,但本身不是使用者真正關心的那份資料。

核心概念 2/2

Query Parameter 是最容易被使用者看見的部分

概念三:網址列裡的問號後面,就是 Query Parameter

下次搜尋網站時,留意一下網址列——後面那段 ?keyword=xxx 就是 Query Parameter,在向 Server 傳達「我要用這個關鍵字搜尋」的條件。它跟 Body 一樣是傳遞資料,只是位置和用途略有不同,通常用在篩選、排序這類「附加條件」。

概念四:三個概念合在一起,才是一次完整的溝通

Header 說明格式與身分、Query Parameter 傳遞篩選條件、Body 承載主要資料,三者各自負責不同角色,合在一起才構成一次完整的 Request 或 Response。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
Header vs BodyHeader 是「關於這次溝通的說明」,例如格式、身分驗證;Body 是「真正想傳遞的內容」,例如表單填的資料。一個是信封上的備註,一個是信紙裡的正文。
Body vs Query Parameter兩者都能傳遞資料,但 Body 通常放在信件內容裡,不會直接顯示在網址上;Query Parameter 附加在網址後面,使用者在網址列就能看到,較不適合放敏感資料。
案例

Northstar Electronics 的產品搜尋功能

Northstar 官網有個產品搜尋框,使用者輸入「手機」後按下搜尋:

Request:
  網址:northstar-electronics.com/search?keyword=手機
  Header:說明這次溝通的格式
  (這裡沒有 Body,因為搜尋條件已經寫在網址的 Query Parameter 裡)

Response:
  狀態:200
  Body:{ "results": [ {"name":"手機A"}, {"name":"手機B"} ] }
  1. 有哪些角色?使用者的 Browser(Client)、Northstar 的官網後端(Server)。
  2. 資料如何移動?搜尋關鍵字透過 Query Parameter 傳給 Server,結果透過 Response 的 Body 傳回。
  3. 各部分負責什麼?Query Parameter 傳遞搜尋條件,Body 傳回搜尋結果清單。
  4. 哪裡可能出錯?如果 Query Parameter 裡的關鍵字沒有正確帶入網址(例如打錯字或編碼問題),Server 可能收到錯誤或空白的搜尋條件,導致搜不到預期結果。
  5. 第一個合理的檢查方向?先看網址列的 Query Parameter 是否正確帶上了使用者輸入的關鍵字,再往 Server 端的搜尋邏輯排查。

常見誤解

三個很多人踩過的坑

誤解一:「Request 只有 Body,沒有其他東西」

很多 Request 其實不帶 Body(例如剛才的搜尋案例),條件全靠 Query Parameter 傳遞。是否需要 Body,要看這次溝通的性質。

誤解二:「Header 不重要,可以忽略」

Header 裡常常帶著身分驗證資訊,如果 Header 設定錯誤,即使 Body 的內容完全正確,Server 也可能因為認不出身分而拒絕這次請求。

誤解三:「網址列上看不到資料,就代表資料很安全」

Body 雖然不會顯示在網址列,但透過瀏覽器工具還是能被看到,不能單靠「藏在 Body 裡」當作安全機制,敏感資料的保護要靠單元 9 會教的身分驗證與加密機制。

遇到問題先查什麼

「表單送出後資料怪怪的」的檢查順序

  1. 確認這次操作是否需要傳遞資料。是透過網址的 Query Parameter,還是透過 Body?
  2. 確認 Header 是否正確。特別是身分驗證相關的資訊有沒有正確帶上。
  3. 確認 Body 裡的資料內容是否完整。可以請工程師用瀏覽器工具(2-8 會教)實際查看送出的內容。
  4. 確認 Response 回傳了什麼。Server 說成功,還是回傳了錯誤訊息?
  5. 把 Request 和 Response 的內容都提供給工程師。這比只描述「表單怪怪的」有效率得多。
重點整理

帶走這四句話

  1. HTTP 是規範 Client 與 Server 之間如何請求與回應的協定,Request 是問句,Response 是答句。
  2. Request 和 Response 都有 Header(附加資訊)和 Body(實際內容)兩個部分。
  3. Query Parameter 附加在網址後面,常用來傳遞篩選或設定條件,使用者在網址列就能看到。
  4. 描述問題時,分清楚是 Header、Body 還是 Query Parameter 的問題,能讓工程師更快定位方向。
檢核 1/4

使用者在表單裡填寫的實際資料,通常會放在 Request 的哪個部分?

正確答案是 C。Body 是 Request 或 Response 真正承載的資料內容,使用者填寫的表單資料通常放在這裡。
檢核 2/4

是非題:所有的 Request 一定都要帶 Body。

正確答案是「否」。像搜尋這類操作,常常只用網址上的 Query Parameter 傳遞條件,不需要額外帶 Body。
檢核 3/4

下列哪一個最可能出現在 Header 裡?

正確答案是 A。Header 存放的是這次溝通的附加資訊,身分驗證憑證是典型例子;使用者填寫的資料和結果訊息通常放在 Body 裡。
檢核 4/4

情境題:客戶回報「搜尋功能找不到正確結果」。第一個合理的檢查方向是?

正確答案是 B。搜尋條件通常透過 Query Parameter 傳遞,先確認這段資訊有沒有正確帶入,是成本最低的第一步。
