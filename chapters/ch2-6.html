Cookie、Session
與 Browser Storage

預估 30 分鐘 ・ 建議先讀完 1-2、2-2 再來。

您登入一個網站後,通常不用每翻一頁就重新輸入帳密一次。這件「系統記得我」的事,是怎麼做到的?這一章要教您瀏覽器記住東西的幾種方式。

  • 分辨 Cookie、Session、Local Storage 各自存的是什麼。
  • 理解為什麼「登入狀態」需要被記住。
  • 看到「請清除瀏覽器 Cookie 再試試」這個建議時,知道它在解決什麼問題。
情境

「我明明登入了,怎麼點下一頁又要我重新登入?」

Northstar Electronics 的客戶反映使用會員中心時的怪現象:

「我登入之後,點了訂單頁面沒事,但點到另一個頁面,系統又要我重新登入,好像忘記我是誰一樣。」

1-2 教過,HTTP 是一問一答:每次 Request 都是獨立的,Server 收到請求後回應完就結束了,不會自動記得「這是同一個人」。既然如此,系統要怎麼在很多次獨立的請求之間,記得「這是剛剛登入的那位使用者」?答案就在這一章。

情境例子 1/2

寄物櫃的號碼牌

想像您去一個遊樂園,入場時領了一張手環,上面印著一組號碼。之後不管您去哪個設施排隊,工作人員看一眼手環號碼,就知道「這位是已經買票入場的客人」,不用每個設施都重新驗票。

Cookie就像這條手環——是 Server 在您第一次造訪時,發給您 Browser 的一小段資料。之後您 Browser 每次向同一個網站發出 Request,都會自動附上這個 Cookie,讓 Server 認出「這是同一位訪客」。

而 Server 那邊,通常會有一份對照表,記錄「這個號碼對應的是哪位客人、目前登入了沒有」。這份對照表和它所代表的「這次造訪的狀態」,就是 Session(工作階段)。Cookie 像是您手上的號碼牌,Session 則是遊樂園工作人員手中,那份對應號碼牌的紀錄簿。

先記住這句 Cookie 是「Browser 這邊拿著的號碼牌」,Session 是「Server 那邊記錄的對應狀態」。兩者合作,才能讓系統記得您是誰。
情境例子 2/2

手環會到期,置物櫃裡的東西不會自動清空

手環通常有使用期限,遊樂園打烊或您離開後,這條手環可能就失效了。Cookie 也分成兩種:Session Cookie(工作階段 Cookie)——關掉瀏覽器就消失,像是當天限定的手環;Persistent Cookie(持久性 Cookie)——會保存一段設定好的時間,即使關掉瀏覽器、下次再打開,還記得您上次的狀態,像是可以重複使用一段時間的會員卡。

除了 Cookie,Browser 本身還有另一種儲存空間,叫 Local Storage(本機儲存)——比較像是您自己帶來的置物櫃,由網站的程式主動存放資料在您的裝置上,不會像 Cookie 那樣每次 Request 都自動夾帶傳送,通常用來存放一些不需要每次都告訴 Server 的資料,例如網頁介面上的個人化設定。

回到開頭的情境。如果客戶在某幾頁被要求重新登入,可能是負責記住登入狀態的 Cookie 或 Session 出了問題——例如設定的有效範圍不小心只涵蓋了部分頁面,導致換頁後系統「認不出手環了」。

串起來看

登入之後,每一頁背後發生的事

接續 2-2 教過的 Request/Response,補上 Cookie 和 Session 的角色:

1. 使用者輸入帳密登入,Server 驗證成功
2. Server 建立一筆 Session,記錄「這位使用者已登入」
3. Server 在 Response 裡,附上一個 Cookie(裡面是這個 Session 的識別碼)
4. Browser 收到這個 Cookie,自動保存起來
5. 使用者點下一頁,Browser 自動把 Cookie 一起附在新的 Request 裡
6. Server 看到這個 Cookie,對照 Session 紀錄,認出「這是剛剛登入的那位」

第 5、6 步會在使用者瀏覽網站的過程中不斷重複。這也解釋了 2-2 提過的 Header——Cookie 通常就是放在 Request 的 Header 裡傳遞的。

為什麼要學這個

兩個馬上用得到的場景

場景一:理解「清除 Cookie 再試試」為什麼常常有效

當使用者的登入狀態卡在異常情況(例如舊的 Session 資料錯亂),清除 Cookie 等於「丟掉手上舊的號碼牌」,重新登入會拿到一張全新的,常常就能解決問題。理解原理後,您能更有信心地向使用者說明這個建議背後的邏輯,而不只是照本宣科。

場景二:判斷「登入狀態異常」該往哪裡查

如果只有某些頁面要求重新登入,先懷疑是不是 Cookie 的有效範圍設定有誤;如果是所有使用者都頻繁被登出,則可能是 Server 端的 Session 管理機制出了問題。分清楚這兩種情況,能幫助工程師更快定位。

核心概念 1/2

Cookie 讓「一問一答」有了記憶

概念一:HTTP 本身不記得任何事,是 Cookie/Session 補上這個功能

1-2 教過,每次 Request 都是獨立的一問一答。Cookie 和 Session 存在的目的,正是為了在這些各自獨立的請求之間,建立起「這是同一個人在連續操作」的連結。

概念二:Cookie 是自動附加的,Local Storage 不是

Cookie 會被 Browser 自動夾帶在每次符合條件的 Request 裡送出;Local Storage 裡的資料不會自動傳送,需要網站的程式主動去讀取、使用。這個差異也影響了兩者適合存放的資料類型。

核心概念 2/2

「記得您」也代表要留意保存多久、給誰看

概念三:Session Cookie 和 Persistent Cookie,決定了「記多久」

單純的登入狀態,通常適合用 Session Cookie(關瀏覽器就消失),避免使用者在公用電腦上忘記登出時留下風險。如果是「記住我」這類選項,才會用到 Persistent Cookie,刻意延長保存時間。

概念四:Cookie 涉及識別使用者,因此和單元 10 的追蹤與同意議題有關

Cookie 除了記錄登入狀態,也常被用來追蹤使用者行為(單元 10 會深入教)。這也是為什麼許多網站需要顯示 Cookie 同意提示——因為 Cookie 牽涉到辨識與追蹤使用者的能力,不只是單純的技術機制。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
Cookie vs SessionCookie 是存在 Browser 端的號碼牌;Session 是存在 Server 端、記錄實際狀態的紀錄。Cookie 裡通常只放一個識別碼,不是完整的使用者資料。
Cookie vs Local StorageCookie 會自動隨每次 Request 傳送給 Server,適合放需要讓 Server 知道的小型識別資訊;Local Storage 只存在 Browser 端,不會自動傳送,適合放單純給網頁介面自己使用的設定。
案例

Northstar Electronics 的登出異常

會員中心的客戶反映,瀏覽某些頁面時會被要求重新登入,追查後發現:

問題頁面:僅限「訂單查詢」這個子路徑會出現異常
Cookie 設定:登入時發出的 Cookie,有效範圍被誤設為只涵蓋首頁路徑
結果:換到訂單查詢頁面時,Browser 判斷這個 Cookie 不適用於這個路徑,沒有附上
  1. 有哪些角色?使用者的 Browser、會員中心的 Server(含 Session 管理)。
  2. 資料如何移動?登入時取得的 Cookie,理應在所有頁面的 Request 中都被附上。
  3. 各部分負責什麼?Cookie 的有效範圍設定,決定了它會在哪些路徑下被自動夾帶傳送。
  4. 哪裡可能出錯?如果 Cookie 的有效範圍設定得太窄,只涵蓋了部分頁面路徑,換到範圍外的頁面時,Server 就收不到這個 Cookie,自然認不出使用者已登入。
  5. 第一個合理的檢查方向?先確認是所有頁面都會被要求重新登入,還是只有特定頁面——只有特定頁面,通常就要往 Cookie 的有效範圍設定去查。

通用概念常見情境

常見誤解

三個很多人踩過的坑

誤解一:「Cookie 裡存了使用者的完整個人資料」

Cookie 通常只存放一個識別碼,真正的使用者資料是存在 Server 端的 Session 或資料庫裡,Cookie 只是用來對照的那把鑰匙。

誤解二:「清除 Cookie 會刪除我在網站上的所有資料」

清除 Cookie 通常只會讓您需要重新登入,不會刪除您帳號裡儲存的實際資料(那些資料保存在 Server 端)。

誤解三:「Session 和 Cookie 是同一個東西」

兩者是合作關係,不是同一個東西。Cookie 是 Browser 端拿著的號碼牌,Session 是 Server 端對應的紀錄,少了任何一邊,這套機制都無法運作。

遇到問題先查什麼

「登入狀態異常」的檢查順序

  1. 確認異常的範圍。是所有頁面,還是特定頁面才會發生?
  2. 確認發生的頻率。是每次都這樣,還是偶爾發生?
  3. 請使用者嘗試清除 Cookie 後重新登入。這能快速排除本機端殘留舊資料的可能性。
  4. 確認是否有多分頁或多裝置同時登入的情況。這也可能影響 Session 的判斷。
  5. 如果清除 Cookie 無法解決,請工程師檢查 Cookie 的有效範圍與 Session 逾時設定。
重點整理

帶走這四句話

  1. HTTP 本身不記得任何事,Cookie 和 Session 是補上「記得您是誰」這個功能的機制。
  2. Cookie 是 Browser 端拿著的號碼牌,Session 是 Server 端對應的紀錄狀態。
  3. Cookie 會自動隨請求傳送,Local Storage 不會,兩者適合存放的資料類型不同。
  4. 登入狀態異常時,先確認影響範圍,再判斷是 Cookie 設定問題還是 Server 端的 Session 問題。
檢核 1/4

Server 端記錄「這次造訪狀態」的機制,叫做什麼?

正確答案是 B。Session 是 Server 端記錄的工作階段狀態,通常透過 Cookie 裡的識別碼來對照。
檢核 2/4

是非題:Cookie 通常只存放一個識別碼,真正的使用者資料保存在 Server 端。

正確答案是「是」。Cookie 通常只是一把對照用的鑰匙,真正的資料保存在 Server 端的 Session 或資料庫裡。
檢核 3/4

下列哪一種儲存方式,不會自動隨每次 Request 傳送給 Server?

正確答案是 C。Local Storage 只存在 Browser 端,需要網站程式主動讀取,不會像 Cookie 那樣自動附加在每次請求裡傳送。
檢核 4/4

情境題:客戶反映「只有訂單查詢頁面會被要求重新登入,其他頁面都正常」。第一個合理的懷疑方向是?

正確答案是 A。只有特定頁面出現異常,最可能的原因是該頁面路徑的 Cookie 有效範圍設定有誤,導致 Cookie 沒有被正確傳送。
