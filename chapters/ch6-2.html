REST 與 SOAP — 兩種常見的 API 設計方式

預估 30 分鐘 ・ API 與資料交換

REST 是一組架構限制，通常用 URI 表示資源，並沿用 HTTP Method、Status Code 與快取語意。SOAP 是訊息協定，以 XML Envelope 定義訊息格式，企業服務常搭配 WSDL、XML Schema 與 WS-* 標準。

  • 分辨「架構風格」REST 與「訊息協定」SOAP。
  • 讀懂兩者的服務契約、請求、回應與錯誤表達方式。
  • 根據既有服務、治理、安全與交易需求選擇整合方式。
判斷桌

用契約與訊息結構比較，不用「新或舊」比較

COMPARE / 看實際介面

REST

介面模型
資源 URI，加上 GET、POST、PUT、PATCH、DELETE 等 HTTP Method
訊息格式
常見 JSON，也可以使用 XML 或其他 Media Type；JSON 不是 REST 的必要條件
契約與錯誤
常用 OpenAPI 描述；以 HTTP Status Code 加 Response Body 表達錯誤

SOAP

介面模型
以 Operation 與 XML Message 交換資料，可經由 HTTP 等 Transport 傳送
訊息格式
固定的 XML Envelope、Header 與 Body 結構
契約與錯誤
常用 WSDL／XSD 描述；錯誤放在 SOAP Fault，可搭配 WS-Security 等標準
如果現有 ERP 已提供 WSDL 與 WS-Security，整合端通常要遵守該 SOAP 契約；不能只因團隊較熟 JSON 就改稱 REST。
案例起點

先讀供應端已提供的契約

Northstar 的新 CRM 要把訂單送進既有 ERP。ERP 只提供 WSDL，要求特定 XML Namespace、Client Certificate 與 SOAPAction；CRM 團隊原本準備傳送 JSON。

先確認 ERP 暴露的是 REST Resource 還是 SOAP Operation。這會決定 Request 結構、驗證方式與錯誤格式。

此案例不能由呼叫端自行「選 REST」。供應端已存在的服務契約，就是整合必須遵守的邊界。

概念拆解 1/2

REST 把業務資料建模成 Resource

/orders/123 表示訂單資源。GET 讀取、PATCH 部分修改、DELETE 刪除；成功或失敗由 Status Code 與 Response Body 說明。

REST 的重點不是 URL 長得簡單，也不是一定回傳 JSON，而是 Client 與 Server 透過一致的介面與無狀態請求互動。

判斷重點看到單一 Endpoint 全部使用 POST，不代表一定不是 REST；但要進一步確認它是否仍以資源、HTTP 語意與可預期的狀態轉移設計。
概念拆解 2/2

SOAP 傳遞有固定結構的 XML Message

SOAP Envelope 包含可選的 Header 和必要的 Body。WSDL 可描述有哪些 Operation、每個 Message 的 Schema，以及服務 Endpoint。

SOAP 經常使用 HTTP，但不把業務動作直接等同於 GET 或 POST；HTTP 可能只是承載 SOAP Message 的 Transport。

判斷重點HTTP 200 也可能包著 SOAP Fault；排查時要同時讀 HTTP Status、Envelope 與 Fault 內容。
精確定義

本章名詞的工作定義

術語完整定義常見規格證據
REST一種分散式系統架構風格；常見 API 會以 Resource URI、HTTP Method、Representation 與 Status Code 實作其一致介面。OpenAPI、Endpoint、Method、Media Type、Status Code。
SOAP定義 XML Message 結構與處理規則的協定；企業服務常以 WSDL／XSD 定義 Operation 與資料型別。WSDL、XSD、Namespace、SOAPAction、Envelope、Fault。
Service ContractClient 與 Provider 對 Operation、資料格式、驗證、錯誤與版本的共同約定。OpenAPI 或 WSDL，以及供應端版本文件。

REST API 可以有嚴格契約；SOAP 也不等於「比較安全」。安全性取決於實際驗證、加密、簽章、權限與部署方式。

流程位置

同一個「建立訂單」需求，訊息不同

REST
POST /orders
Content-Type: application/json
{ "customerId": "C01", "amount": 800 }
→ 201 Created + Location: /orders/123

SOAP
POST /OrderService
SOAPAction: CreateOrder
<Envelope><Body><CreateOrder>...</CreateOrder></Body></Envelope>
→ <CreateOrderResponse>... 或 <Fault>...

兩者都能建立訂單，但契約、資料驗證與錯誤解析方式不同。

問題排查

先確認 Endpoint、HTTP 方法與版本，再查看 Request Header、Body 與 Parameter。這能分辨問題是否位於 REST 或 SOAP。

核心概念 1/2

輸入、規則與輸出要分開寫

輸入

列出 REST 開始前已知的資料、身分、事件或設定。輸入不明，後面的結果就無法比較。

規則與輸出

SOAP 執行後，保存實際套用的規則版本、Response、狀態或資料值。不要用「看起來正常」代替可查證結果。

核心概念 2/2

成功條件和失敗條件都要具體

成功條件

REST 常以資源與 HTTP 方法設計；SOAP 使用較嚴格的 XML 訊息與服務契約。選擇要看既有環境與需求。 驗收 REST 與 SOAP 時，寫明預期狀態、資料值與完成時間。

失敗條件

只因 REST 較常見就否定 SOAP，可能忽略企業既有服務與交易規範。 REST 或 SOAP 失敗時，保留原始輸入、環境、時間與識別值。

容易混淆

用處理邊界分清楚三組概念

容易混淆實際差別
REST vs SOAPREST：以資源與 HTTP 方法設計服務介面的風格。 SOAP：以 XML 訊息與嚴格服務契約交換資料的協定。
規格存在 vs 實際生效REST 的文件是預期規則；SOAP 的設定值、Response 或 Log 才能證明實際結果。
單次成功 vs 穩定可重現REST 單次成功不代表 SOAP 在其他條件下也正確；要用相同輸入重做。

常見誤解

三個會讓排查走錯方向的假設

把 REST 和 SOAP 當成同一件事

只因 REST 較常見就否定 SOAP，可能忽略企業既有服務與交易規範。

只看最後畫面

畫面是流程的最後一層。仍要核對 REST 的輸入與 SOAP 的執行紀錄。

一次改多個條件

同時修改 REST 與 SOAP 的條件，即使恢復也無法歸因。每次只改一項。

排查順序

REST 發生異常時的排查順序

  1. 1. 確認 Endpoint、HTTP 方法與版本。
  2. 2. 查看 Request Header、Body 與 Parameter。
  3. 3. 核對 Authentication、Token 與權限。
  4. 4. 讀 Status Code、Response 與 Request ID。
  5. 5. 依 Rate Limit、Timeout 或 Retry 規則處理。

每一步先記錄原始值再做下一步。若修改 REST 或 SOAP 的設定，保存修改前後值與時間。

重點整理

讀完後應該能說清楚這四件事

  1. REST 是架構風格，不等於 JSON；常見實作以 Resource、HTTP Method 與 Status Code 組成一致介面。
  2. SOAP 是 XML 訊息協定；WSDL／XSD 常用來描述 Operation、資料型別與 Endpoint。
  3. 選擇通常受供應端既有契約、治理、安全標準與工具支援限制，不是偏好問題。
  4. 排查 REST 要讀 Method、URI、Header、Body 與 Status；排查 SOAP 還要讀 Namespace、SOAPAction、Envelope 與 Fault。
