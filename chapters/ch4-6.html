SQL 查詢的基本概念
SELECT 與 WHERE

預估 30 分鐘 ・ 建議先讀完 4-1、4-3 再來。

前五章認識了資料的收納方式,這一章要學「開口跟資料庫要資料」的語言:SQL。放心,目標不是讓您寫出查詢,而是讓您看得懂工程師寫的簡單查詢——這在確認報表邏輯、排查資料問題時非常好用。

  • 說明 SELECT、WHERE、ORDER BY、GROUP BY 分別控制欄位、篩選、排序與分組。
  • 分辨「查資料」和「改資料」的指令,理解風險等級的差異。
  • 拿到一段簡單查詢時,能用中文說出它在問什麼。
動手試

把業務問題翻成 SELECT 與 WHERE

PRACTICE / 先作答再展開
SQL QUERYTRY IT
需求：從 Orders 表找出 status 為 Paid，且 amount 大於 10000 的 order_id 與 amount。

SELECT ______
FROM ______
WHERE ______;
完成後，展開參考答案
SELECT order_id, amount FROM Orders WHERE status = 'Paid' AND amount > 10000;　先確認欄位、表名與條件，再執行查詢。
情境

「報表數字怪怪的,工程師貼了一段咒語給我」

Northstar Electronics 的顧問質疑某張報表的數字,工程師回訊息:

「報表背後就是這段查詢,你看一下條件對不對:
SELECT * FROM Orders WHERE Amount > 10000 AND Status = 'Paid'」

看不懂的話,這是一串咒語;看得懂的話,這就是一句白話:「從訂單表裡,找出金額大於一萬、而且狀態是已付款的訂單」。看懂它,您就能直接回答:「啊,問題在這——我們要的是『含』一萬,條件應該是大於等於。」這一章就是要讓您具備這個能力。

情境例子 1/2

SQL 就是「點餐的句型」

SQL(Structured Query Language,結構化查詢語言)是對資料庫下指令的標準語言。它的查詢句型,跟您跟圖書館員說話的方式很像:

「請幫我找書名和作者(要哪些欄位),從歷史區(哪張表),條件是2020 年以後出版的(篩選條件),照出版年份排序(排序方式)。」

對應到 SQL,四個部分各有關鍵字:

SELECT 書名, 作者          ← 要哪些欄位
FROM   歷史區書籍表         ← 從哪張表
WHERE  出版年份 >= 2020    ← 篩選條件
ORDER BY 出版年份          ← 怎麼排序
先記住這句 讀 SQL 查詢的口訣:「SELECT 要什麼、FROM 從哪拿、WHERE 篩什麼、ORDER BY 怎麼排」。九成的簡單查詢,用這四句就能翻譯成中文。
情境例子 2/2

再加一個「分組統計」,報表就是這樣算的

報表常見的「各分類加總」,靠的是另一個關鍵字 GROUP BY(分組)。像是問圖書館員:「每個書區各有幾本書?」——先把書按書區分堆,再各自數數量。

SELECT 客戶編號, SUM(金額)   ← SUM 是加總;每位客戶各自加總金額
FROM   Orders
WHERE  Status = 'Paid'       ← 只算已付款的
GROUP BY 客戶編號            ← 按客戶分組

翻成中文:「從訂單表,只看已付款的,按客戶分組,算出每位客戶的消費總額。」您看報表上的「客戶消費排行」,背後多半就是這樣一段查詢。

發現了嗎?報表的每個數字,背後都是一段查詢;報表怪怪的,通常就是查詢的條件跟您想的不一樣。WHERE 少了一個條件、GROUP BY 分錯了組,數字就會對不上——這也呼應了 4-2 報表落差的故事。

正式定義

查詢與異動的關鍵字總表

關鍵字作用性質
SELECT指定要取出哪些欄位查詢:只讀取,不改變資料
WHERE在回傳結果前，以布林條件篩選符合條件的資料列
ORDER BY依一個或多個欄位排列查詢結果，可指定 ASC 升冪或 DESC 降冪
GROUP BY分組統計(搭配 SUM、COUNT 等)
INSERT新增一筆紀錄異動:會真的改變資料,風險高
UPDATE修改既有紀錄的欄位值
DELETE刪除符合 WHERE 條件的資料列；省略 WHERE 會刪除表中所有資料列

上下兩區的風險等級完全不同:SELECT 查一百次也不會弄壞資料;UPDATE 和 DELETE 一句下錯,幾萬筆資料就變了。這是這一章最重要的一條分界線。

串起來看

最危險的一句話:忘了 WHERE 的 UPDATE

看兩句只差一行的指令:

UPDATE Customers SET Phone = '0912-000-000'
WHERE  CustomerId = 'C001'
→ 只有 C001 的電話被改掉(正確)

UPDATE Customers SET Phone = '0912-000-000'
→ 沒有 WHERE!全部 85,000 筆客戶的電話,通通變成同一支(災難)

WHERE 在查詢裡只是「篩選」,在 UPDATE 和 DELETE 裡卻是「保命符」——它界定了「改誰」。少了它,就是「改全部」。這也是為什麼 1-5 教過的環境觀念這麼重要:這類指令,永遠先在測試環境驗證,確認影響筆數符合預期,才碰正式環境。

為什麼要學這個

兩個馬上用得到的場景

場景一:驗證報表邏輯,不用等來回猜

質疑報表數字時,直接請工程師把背後的查詢貼給您。用「SELECT 要什麼、WHERE 篩什麼」的口訣讀一遍,常常一眼就發現:「條件只算了 Null,沒算空字串」「日期用的是建立日,不是成交日」。原本要來回猜三天的問題,十分鐘收工。

場景二:提出資料修正需求時,說清楚範圍

請工程師批次修正資料時,用 WHERE 的思維描述範圍:「請把『建立日在 2026 年 1 月之前、且狀態是暫存』的訂單改為作廢」——這句話幾乎可以直接翻成一句 UPDATE。範圍說得精準,工程師做得放心,您也不會收到「改超過」的意外。

核心概念 1/2

查詢是安全的,異動是危險的

概念一:SELECT 永遠不會弄壞資料

查詢只是「看」,看幾次都不會改變資料庫的內容。所以排查問題時,請工程師「先查給我看」永遠是安全的第一步;真正需要謹慎審批的,是後面的異動動作。

概念二:UPDATE/DELETE 前,先用 SELECT 數一次

工程師的標準習慣:要改資料前,先用同樣的 WHERE 條件跑一次 SELECT,確認「即將被改到的是哪幾筆、共幾筆」。作為顧問,您可以把這個習慣變成流程要求:「執行前,請先給我看影響筆數。」這一句能擋掉大多數批次修改事故。

核心概念 2/2

條件的細節,決定數字的對錯

概念三:大於和大於等於,是兩份不同的報表

「金額超過一萬」到底含不含一萬?「上個月」是日曆月還是最近三十天?這些邊界條件,業務方和工程師的預設常常不同。看得懂 WHERE,您就能在需求和實作之間,把這些模糊地帶一一釘牢。

概念四:Null 在條件裡的行為,要特別留意

4-2 提過:Null 不等於任何值。「WHERE Phone <> '0912'」這種「不等於」條件,不會抓到電話是 Null 的紀錄——它們既不等於、也不不等於,直接被略過。報表少了一群「欄位空白」的資料,常常就是這個原因。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
SELECT vs UPDATE/DELETESELECT 只讀不改,安全;UPDATE/DELETE 會真的改變資料,危險。請工程師「查」不用緊張,請工程師「改」要先確認範圍與筆數。
WHERE 在查詢 vs 在異動中的角色查詢裡的 WHERE 是篩選,寫錯只是查錯;異動裡的 WHERE 是界定「改誰」,寫漏就是改全部。同一個關鍵字,風險天差地遠。
案例

Northstar Electronics 的報表對帳

行銷主管質疑「上月 VIP 消費報表」的總額偏低,工程師提供背後的查詢:

SELECT 客戶編號, SUM(金額)
FROM   Orders
WHERE  訂單日期 >= '2026-06-01' AND 訂單日期 <= '2026-06-30'
       AND 客戶等級 = 'VIP'
       AND Status = 'Paid'
GROUP BY 客戶編號
  1. 有哪些角色?行銷主管(質疑數字)、工程師(提供查詢)、顧問(居中翻譯驗證)。
  2. 資料如何移動?報表數字直接來自這段查詢的結果。
  3. 查詢在問什麼?六月份、VIP 等級、已付款的訂單,按客戶加總金額。
  4. 哪裡可能出錯?顧問逐條核對後發現:業務認知的「上月消費」包含「已出貨未付款」的訂單,但查詢只算了 Paid。認知差在 Status 那一行。
  5. 第一個合理的檢查方向?報表數字有爭議時,先拿到背後的查詢,逐條翻成中文,和業務方的定義一條一條對——差異幾乎都藏在 WHERE 裡。

常見誤解

三個很多人踩過的坑

誤解一:「SQL 是工程師的事,顧問看不懂也沒關係」

不需要會寫,但看懂簡單查詢,能讓您直接驗證報表邏輯、精準描述修改範圍,是投資報酬率極高的技能。

誤解二:「請工程師查資料跟改資料,都一樣要走一週的申請流程」

兩者風險等級不同:查詢是唯讀且安全的,流程可以輕;異動才需要嚴格的審批與備份。分清楚,能讓排查速度快很多。

誤解三:「條件寫『不等於某值』,就會抓到所有其他資料」

Null 的紀錄不會被「不等於」抓到。報表莫名少一群資料時,把「欄位是 Null 的那群人」列入嫌疑清單。

遇到問題先查什麼

「報表數字有爭議」的檢查順序

  1. 拿到報表背後的查詢。這是事實的源頭,比互相描述快得多。
  2. 用口訣翻譯:SELECT 要什麼、FROM 哪張表、WHERE 篩什麼、GROUP BY 怎麼分。
  3. 把 WHERE 的每一條,和業務方的定義逐條對。邊界(含不含等於)、狀態、日期欄位,是三大高發區。
  4. 檢查 Null 的處理。條件會不會意外排除了空值的那群資料?
  5. 確認後,把「正確定義」白紙黑字記下來。讓下一次拉報表的人,不用重走一遍這個過程。
重點整理

帶走這四句話

  1. 讀查詢的口訣:SELECT 要什麼、FROM 從哪拿、WHERE 篩什麼、ORDER BY 怎麼排、GROUP BY 怎麼分組。
  2. SELECT 只讀不改,安全;INSERT/UPDATE/DELETE 會改變資料,風險完全不同。
  3. UPDATE/DELETE 少了 WHERE 就是「改全部」;執行前先用 SELECT 確認影響筆數。
  4. 報表數字的爭議,九成藏在 WHERE 的條件細節裡——邊界、狀態、日期欄位、Null。
檢核 1/4

SQL 查詢中,負責「設定篩選條件」的關鍵字是?

正確答案是 A。WHERE 負責篩選;SELECT 指定要哪些欄位,ORDER BY 負責排序。
檢核 2/4

是非題:SELECT 查詢執行再多次,也不會改變資料庫裡的資料。

正確答案是「是」。SELECT 是唯讀操作,安全無虞;真正需要謹慎的是 INSERT、UPDATE、DELETE 這些會改變資料的指令。
檢核 3/4

一句 UPDATE 指令忘了寫 WHERE,會發生什麼事?

正確答案是 C。異動指令裡的 WHERE 界定「改誰」,少了它就是改全部——這是資料庫操作最經典的災難之一,所以執行前要先用 SELECT 確認影響範圍。
檢核 4/4

情境題:業務主管質疑報表總額偏低。您的第一步應該是?

正確答案是 B。報表爭議的根源幾乎都在查詢條件與業務定義的落差,拿到查詢逐條核對,是最快找到差異的方法。
