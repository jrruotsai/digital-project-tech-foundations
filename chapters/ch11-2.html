Configuration、Code 與 Metadata — 改的是哪一種東西

預估 30 分鐘 ・ 環境、版本與部署

Configuration 是可調設定，Code 是程式邏輯，Metadata 描述結構與行為；部署方式可能不同。

  • 能說明 Configuration：可由環境或部署設定的參數，而非程式邏輯本身。
  • 能區分 Configuration 與 Code／Metadata 在流程中的用途。
  • 遇到問題時，先確認環境、版本與變更範圍。

案例起點

Configuration 和 Code／Metadata 沒有寫清楚時

Northstar 的規格同時提到 Configuration 與 Code／Metadata，卻沒有標出兩者的適用條件與處理邊界。

Configuration：可由環境或部署設定的參數，而非程式邏輯本身。；Code／Metadata：Code 是程式邏輯；Metadata 描述結構、設定或部署物件。

測試時要避免這個結果：漏部署一項 Metadata，程式碼正確也可能無法運作。

概念拆解 1/2

Configuration 的處理邊界

Configuration：可由環境或部署設定的參數，而非程式邏輯本身。

套用到本章情境，第一步是確認環境、版本與變更範圍。確認 Configuration 的適用範圍後，再檢查 Code／Metadata，不從畫面直接猜原因。

判斷重點說明 Configuration 時，要交代誰提供輸入、系統套用哪條規則，以及會產生什麼結果。
概念拆解 2/2

Code／Metadata 與 Configuration 的差別

Code／Metadata：Code 是程式邏輯；Metadata 描述結構、設定或部署物件。

Configuration 是可調設定，Code 是程式邏輯，Metadata 描述結構與行為；部署方式可能不同。

不要混用Configuration 與 Code／Metadata 不能互換；規格要分別寫出兩者的輸入、處理方式與輸出。
精確定義

本章名詞的工作定義

術語定義判斷時要問
Configuration可由環境或部署設定的參數，而非程式邏輯本身。它接收什麼輸入，完成後留下什麼結果？
Code／MetadataCode 是程式邏輯；Metadata 描述結構、設定或部署物件。它和 Configuration 的邊界在哪裡？
Evidence能證明 Configuration 或 Code／Metadata 實際結果的原始資料，例如 Request／Response、Log、時間、版本與識別值。哪一筆原始紀錄能支持目前的判斷？

定義 Configuration 與 Code／Metadata 時，必須寫出用途、處理邊界和可觀察結果。

流程位置

Configuration 在整體流程中的位置

Change → Version Control → Build → Deploy → Smoke Test → Release／Rollback
→ 觀察結果與例外
→ 記錄負責人與處理決定

沿著流程逐段比對輸入與輸出。若 Configuration 的輸入正確而 Code／Metadata 的輸出不符預期，就從兩者之間的邊界開始查。

專案應用

這個概念會出現在哪些工作？

規格與設計

部署協調人員需要把「Configuration、Code 與 Metadata — 改的是哪一種東西」改寫成可確認的規則，放進版本紀錄與上線檢核表。

問題排查

先確認環境、版本與變更範圍，再分辨 Code、Configuration 與 Metadata。這能分辨問題是否位於 Configuration 或 Code／Metadata。

核心概念 1/2

輸入、規則與輸出要分開寫

輸入

列出 Configuration 開始前已知的資料、身分、事件或設定。輸入不明，後面的結果就無法比較。

規則與輸出

Code／Metadata 執行後，保存實際套用的規則版本、Response、狀態或資料值。不要用「看起來正常」代替可查證結果。

核心概念 2/2

成功條件和失敗條件都要具體

成功條件

Configuration 是可調設定，Code 是程式邏輯，Metadata 描述結構與行為；部署方式可能不同。 驗收 Configuration 與 Code／Metadata 時，寫明預期狀態、資料值與完成時間。

失敗條件

漏部署一項 Metadata，程式碼正確也可能無法運作。 Configuration 或 Code／Metadata 失敗時，保留原始輸入、環境、時間與識別值。

容易混淆

用處理邊界分清楚三組概念

容易混淆實際差別
Configuration vs Code／MetadataConfiguration：可由環境或部署設定的參數，而非程式邏輯本身。 Code／Metadata：Code 是程式邏輯；Metadata 描述結構、設定或部署物件。
規格存在 vs 實際生效Configuration 的文件是預期規則；Code／Metadata 的設定值、Response 或 Log 才能證明實際結果。
單次成功 vs 穩定可重現Configuration 單次成功不代表 Code／Metadata 在其他條件下也正確；要用相同輸入重做。

常見誤解

三個會讓排查走錯方向的假設

把 Configuration 和 Code／Metadata 當成同一件事

漏部署一項 Metadata，程式碼正確也可能無法運作。

只看最後畫面

畫面是流程的最後一層。仍要核對 Configuration 的輸入與 Code／Metadata 的執行紀錄。

一次改多個條件

同時修改 Configuration 與 Code／Metadata 的條件，即使恢復也無法歸因。每次只改一項。

排查順序

Configuration 發生異常時的排查順序

  1. 1. 確認環境、版本與變更範圍。
  2. 2. 分辨 Code、Configuration 與 Metadata。
  3. 3. 檢查 Branch、Commit 與審查紀錄。
  4. 4. 核對 Secret、備份與 Rollback Plan。
  5. 5. 部署後執行 Smoke Test、監控與業務驗證。

每一步先記錄原始值再做下一步。若修改 Configuration 或 Code／Metadata 的設定，保存修改前後值與時間。

重點整理

  3. 漏部署一項 Metadata，程式碼正確也可能無法運作。
