使用 Browser Network
觀察 Request

預估 30 分鐘 ・ 單元 2 的最後一章,建議先讀完 2-1 至 2-7。

前面七章教了很多名詞:Request、Response、Header、Status Code、Cookie、Cache……這一章要帶您親手打開瀏覽器內建的工具,實際「看見」這些東西長什麼樣子。看得到,才會真的有感覺。

  • 打開瀏覽器的開發者工具(Developer Tools),找到 Network(網路)頁籤。
  • 找到一筆實際的 Request,看懂它的 Method 和 Status Code。
  • 知道下次排查問題時,可以自己先看一眼 Network,再找工程師。

情境

「工程師隨手打開一個面板,我完全跟不上」

Northstar Electronics 的顧問曾經陪同工程師一起排查問題,對方隨手按了幾個鍵,螢幕上跳出一堆英文和數字:

「你看,這裡回傳 404,網址打錯了啦。」工程師輕描淡寫地說,螢幕上那個面板讓顧問完全摸不著頭緒。

那個面板,就是這一章要教的 Network 工具。它其實不難,難的是第一次沒人帶著看。今天我們就一起打開它,看一次真實的 Request 長什麼樣子。

情境例子 1/2

裝了透明玻璃的信件分揀室

還記得 2-2 教過,HTTP 溝通就像寄信一樣,有信封(Header)、信紙(Body)嗎?平常這些信件在 Browser 和 Server 之間往返,我們是看不到的,就像郵局的分揀室,一般人進不去。

Developer Tools(開發者工具)裡的 Network(網路)頁籤,就像在這間分揀室裝了一面透明玻璃,讓您可以親眼看到每一封信:什麼時候寄出的、寄去哪裡、用哪一種方法(2-3 教過的 GET/POST)、對方回了什麼狀態碼(2-4 教過)。

先記住這句 Network 工具不會「改變」任何東西,它只是把原本就在發生的溝通,用看得到的方式呈現出來。打開它很安全,不用擔心會弄壞什麼。
情境例子 2/2

實際打開的步驟

現在跟著做(用桌面版 Chrome 或 Edge 瀏覽器示範,其他瀏覽器操作方式類似):

  1. 打開任何一個網站,例如公司官網。
  2. 按下鍵盤的 F12(或按右鍵選擇「檢查」),畫面右側或下方會跳出開發者工具面板。
  3. 在面板上方找到並點選 Network 頁籤。
  4. 按下瀏覽器的重新整理按鈕,面板裡會開始出現一筆一筆的紀錄,那就是這個網頁載入過程中,發生的每一次 Request。

不用擔心操作出錯,這個面板只是「觀察」,不會影響網站的實際運作。您可以放心打開、關閉,多練習幾次就會熟悉。

正式定義

Network 面板裡常見的欄位

欄位對應到您學過的哪個概念
Name這次 Request 的網址或檔案名稱(呼應 2-1 的 URL)
Method這次 Request 使用的方法,例如 GET、POST(呼應 2-3)
StatusServer 回傳的狀態碼,例如 200、404(呼應 2-4)
Headers(點進單筆紀錄後可見)這次 Request 和 Response 的 Header 內容(呼應 2-2)
Response(點進單筆紀錄後可見)Server 實際回傳的 Body 內容(呼應 2-2)

您不需要記住每個欄位的細節,只要知道「這幾個名詞我在前面章節都學過」,面板上的資訊就不再是天書,而是課本內容的具體呈現。

串起來看

把前七章的內容,對應到畫面上

用一次實際的登入操作,示範整個對照過程:

1. 打開登入頁,Network 面板出現一筆 Name 是登入頁網址的紀錄
   → Method 顯示 GET,Status 顯示 200(呼應 2-3、2-4)

2. 輸入帳密按下登入,Network 面板新增一筆紀錄
   → Method 顯示 POST(這次是送出資料,不是查看,呼應 2-3)
   → 點進去看 Headers,能找到 Cookie 相關資訊(呼應 2-6)

3. 登入成功,畫面跳轉,又出現新的 GET 紀錄
   → 這次的 Request Headers 裡,能看到剛才拿到的 Cookie 被自動帶上(呼應 2-6)

這樣走一遍,您會發現前面七章教的名詞,全部都能在這個面板上被實際指認出來,不再只是抽象的定義。

為什麼要學這個

兩個馬上用得到的場景

場景一:回報問題前,先自己看一眼

客戶說「表單送不出去」,您可以先打開 Network 面板,自己看一眼送出表單那筆 Request 的 Status Code,如果是 400 或 500,就能直接把這個資訊附在回報裡給工程師,省去一來一回確認的時間。

場景二:陪同工程師排查時,聽得懂在看什麼

下次工程師打開這個面板討論問題時,您不會再一頭霧水,而是能跟上對話,甚至主動指出「這筆 Status 是 404 耶」,展現您具備基礎的技術判斷能力。

核心概念 1/2

Network 面板是「觀察窗」,不是「操作介面」

概念一:面板呈現的是真實正在發生的溝通

面板裡看到的每一筆紀錄,都是這個網頁實際發出去、收回來的真實 Request 和 Response,不是模擬或範例,這也是為什麼它對排查問題這麼有價值。

概念二:面板只是觀察,不會影響網站的運作

打開或關閉開發者工具,不會對網站本身的功能造成任何影響。這是一個安全、隨時可以練習的工具,不用擔心不小心弄壞什麼。

核心概念 2/2

看懂 Status 欄位,就能做初步判斷

概念三:紅色或標示異常的紀錄,通常代表出了問題

多數瀏覽器會用顏色或特殊標示,凸顯狀態碼異常(如 4 開頭或 5 開頭)的紀錄,方便您一眼掃過去就能發現異常之處,不需要逐筆檢查。

概念四:找不到某筆該有的 Request,本身也是重要線索

有時候問題不是 Request 失敗,而是「這個 Request 根本沒有被送出去」。這種情況下,面板裡完全看不到相關紀錄,這件事本身就是很有用的線索,代表問題可能卡在更早的前端邏輯,連請求都還沒發出。

容易混淆

兩組容易搞混的概念

容易混淆的一組怎麼分
Console vs Network開發者工具裡還有一個叫 Console 的頁籤,顯示的是程式執行時的訊息或錯誤;Network 顯示的是 Request 和 Response 的往來紀錄。兩者都有用,但關注的層面不同。
「看得到」vs「能改」Network 面板讓您看得到溝通內容,但一般顧問不需要、也不建議在這裡直接修改任何設定或資料,它是觀察與診斷用的工具。
案例

Northstar Electronics 的表單送出異常

客戶回報報名表單按下送出後沒有反應,顧問決定自己先打開 Network 面板看看:

操作:重新填寫表單,打開 Network 面板,再按一次送出
觀察結果:面板裡完全沒有出現新的 POST 紀錄
初步判斷:表單根本沒有把 Request 送出去,問題可能卡在前端(呼應 1-1 的三層架構)
  1. 有哪些角色?顧問(操作 Network 面板的人)、表單的前端程式。
  2. 資料如何移動?正常情況下,按下送出應該會觸發一筆新的 POST 紀錄出現。
  3. 這次觀察負責什麼?確認問題是「請求根本沒發出」,還是「請求發出了,但 Server 回應異常」。
  4. 哪裡可能出錯?如果連 Request 都沒有出現,通常代表前端的邏輯(可能是 3-7 會教的表單驗證)在送出前就攔住了,還沒機會呼叫後端。
  5. 第一個合理的檢查方向?把「Network 面板裡完全沒有新紀錄」這個具體觀察告訴工程師,比只說「按了沒反應」精準得多,能幫助工程師直接往前端邏輯排查。

常見誤解

三個很多人踩過的坑

誤解一:「打開開發者工具很危險,一般人不該碰」

Network 頁籤純粹是觀察用途,不會對網站造成任何影響。放心打開來看,是每個顧問都該練習的基本技能。

誤解二:「看不懂裡面所有欄位,就代表這個工具對我沒用」

不需要看懂每一個細節,只要能找到 Method 和 Status 這兩個最關鍵的欄位,就已經能提供很有價值的資訊給工程師。

誤解三:「面板裡沒有紅色標記,就代表一切正常」

沒有錯誤標記,不代表流程完全正確——例如該出現的 Request 根本沒有出現,也是一種異常,只是它不會被標成醒目的紅色。

遇到問題先查什麼

「操作沒反應」的檢查順序（實際動手）

  1. 打開開發者工具的 Network 頁籤。按 F12 或右鍵選「檢查」。
  2. 重新操作一次剛才有問題的步驟。觀察面板裡有沒有出現對應的新紀錄。
  3. 如果完全沒有新紀錄出現,問題可能在前端。操作根本沒有被送出去。
  4. 如果有紀錄,查看它的 Status 欄位。是 2 開頭、4 開頭,還是 5 開頭?
  5. 把觀察到的具體現象(有沒有紀錄、Status 是什麼)提供給工程師。這比單純描述「沒反應」有價值得多。
重點整理

帶走這四句話

  1. Network 頁籤讓您親眼看見前面章節教過的 Request、Response、Method、Status Code。
  2. 它只是觀察工具,不會影響網站運作,可以放心打開來練習。
  3. 看到某筆該有的 Request 完全沒出現,本身就是重要的線索,代表問題可能卡在更早的前端邏輯。
  4. 提供具體的觀察(有沒有紀錄、Status 是什麼),遠比「沒反應」「怪怪的」這類描述更有幫助。

是非題:打開並查看 Network 面板,不會影響網站本身的實際運作。

正確答案是「是」。Network 頁籤純粹是觀察工具,不會對網站的實際運作造成任何影響,可以放心打開來練習。
